# docs/wiki

このプロジェクトで繰り返し現れる定型手順 / 規約と再発指摘を、過去の PR/issue から蒸留して蓄積する場所。issueのPlanやレビューで毎回同じ説明 / 同じ指摘を繰り返す代わりに、ここ1箇所を参照して二重管理と陳腐化を防ぐ。

## 何を書くか / 書かないか

| 種類                                                | 置き場所                                       |
| --------------------------------------------------- | ---------------------------------------------- |
| 定型手順 / 規約。命名規則 / チェックリスト等        | この wiki ディレクトリ                         |
| 再発指摘 / 失敗パターン。レビューで繰り返された指摘 | この wiki ディレクトリ                         |
| 設計判断とその履歴 / 経緯                           | ADR として `docs/decisions/`。ここには書かない |
| 1度きりの個別事情                                   | 書かない。繰り返しになってから                 |

## ページ粒度

1共通項1ページ。ファイル名は kebab-case の `<共通項名>.md`。ページ化の閾値は出現2件。根拠となる PR/issue が2つ以上になったらページを立てる。根拠が1件だけのものは `_candidates.md` に「内容1行 + #番号」の箇条書きで置き、2件目が現れたらページへ昇格して行を消す。最新コードとの突き合わせで成立しなくなった項目は書かない / 消す。

## ページ書式

各ページは 内容 → 定型手順 → 根拠 の順で構成する。読み手が使うのは内容と手順で、根拠は出典の脚注。根拠には元の議論の PR/issue 番号と、現在のコードで成立を確認した位置の両方を付記し、読み手が「なぜこの規約があるか」と「いまどこで成立しているか」をページから直接辿れるようにする。

```markdown
# <共通項名>

## 内容

何をする/しないかの言語化（1〜3文）。

## 定型手順

繰り返す手順・チェックリスト（あれば）。

## 根拠

- #12 何があったか1行
- #34 何があったか1行
- 現行コード: `path/to/file:行`（検証済みの成立根拠）
```

## 育て方

`/scribe` スキルまたは定期巡回の scribe が、closed になった PR/issue から自動で抽出する。初回は全履歴を読んで基礎化し、以降は前回の scribe PR マージ以降に closed になった分から増分更新する。追加 / 修正はPRで提案されるので、レビューしてマージする。マージが承認になる。手動追記も可。
