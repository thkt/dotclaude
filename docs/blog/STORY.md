- 出典: 以下 3 セッション。
  - `5f7e4857` — ADR 整理 + Rust レビューエージェント自作 + dogfood でバグ発覚
  - `e90b3ecf` — 日報作成 (2026-03-09)。「やったこと」は出せても「学び」が空欄
  - `419250b4` — 日報セッション (2026-03-05)。sharpen を作って消す + 日経KAI の KPT 振り返り
- **引用の出所に注意** (セッション 419250b4)。濃い言葉の多くはチャット発話ではなく、取り込んだ 1on1 メモ / KPT シート由来。記事化時は出所を区別すると誠実さを保てる。

## S1. 「決定を棚に置く文書」から「検証パスの自動生成装置」へ (最濃)

**出典: 5f7e4857**

- **失敗**: ADR は「決定を記録して棚に置く」文書になりがちで、書いてもコードと drift して放置される。
- **転機**: ADR の Decision に具体シンボル (`unwrap_or_note`, `fetch_http`) を書く → 「これコードと一致してる?」という grep の動機が自動発火 → drift と bug が次々表面化。実際に scout で、消えたはずの `per ADR-0065` 参照が 7 箇所残る致命的 drift、Gemini 403 のエラー分類バグなどが副産物として出た。
- **学び**: ADR の効用は「決定の宣言」ではなく「verbalize → verify ループの起点」。書いた Decision を「実装に対する仮説」として扱うと即座に検証サイクルに入り、不整合が芋づる式に出る。

**教訓**: ドキュメントは記録のためでなく、検証を強制的に発火させるトリガーとして設計すると価値が跳ね上がる。「書く目的」ではなく「書いた後に何が自動で問われるか」で文書の良し悪しが決まる。

**引用**:

- [本人発話]「ADRを作成したり検証し直すことでコードの問題を発見修正できたじゃないですか。これってどうしてそこまで到達できたのでしょう?」
- [AI]「verbalize は trigger、verify が本体。」
- [AI]「うっわ、めっちゃ重要な drift 発見した! `per ADR-0065` 参照が ... 7 箇所ある。」
- [本人発話]「ええなあ。この教訓はどこかに書いておきたいな。」

## S2. 「全部記録する」誘惑を critic で篩い落とす

**出典: 5f7e4857**

- **失敗**: audit で見つけた設計判断を片っ端から ADR 化しようとすると ADR が肥大化し、本当に重要な判断が埋もれる (当初 25候補・12候補)。
- **転機**: 起票前に critic-design (devil's advocate) に独立判定させた。結果は容赦なく Part1「12 → 3 keep」、Part2「17 → 5 keep」。主因は「enforcement が既に機械的 (clippy / cargo-deny / テスト) に効いていて ADR が行動を変えない」。ADR が価値を持つのは (1) 型や lint で守れない不変条件、(2) 公開 API の互換性コミットメント、の 2 領域だけ、と基準が言語化された。
- **学び**: 記録は多ければ良いのではない。rejection rate 16〜29% は短期的には「自分の判断が否定された数」だが、長期的には「無駄なドキュメントを生成しなかった節約」。

**教訓**: 自分の成果物に敵対的レビューを通すコストを惜しまない。「却下された割合」を品質ゲートの効きと捉え直す。記録する/しないの基準を明文化すれば判断が属人的でなくなる。

**引用**:

- [AI]「個人 OSS で ADR が価値を持つのは (1) 型や lint で守れない不変条件 と (2) 公開 API の互換性コミットメント の 2 領域に限られる。」
- [AI]「critic を skip しない。rejection rate 16-29% は ... 長期的に『無駄 ADR を生成しなかった節約』。」

## S3. やったことは書けても、学びは空欄だった日報

**出典: e90b3ecf**

- **失敗**: AI に日報を作らせたら「今日やったこと」「AI活用作業」は埋まったが、「今日学んだこと」は空欄 (`-`) のまま投稿された。作業の羅列はできても振り返りが抜けた。
- **転機**: 「学んだこと、claudeの実装から出せないかな」と問い直した。同じログを「作業記録」でなく「学びの鉱脈」として読み直させた瞬間、6 つの気づきが抽出できた。
- **学び**: 振り返りは自動では生成されない。同じログでも「何を問うか」で出力が変わる。アウトプット (やったこと) とアウトカム (学び) は別物で、後者は明示的に問い直して初めて表に出る。

**教訓**: 記録の自動化と内省の自動化は別。AI は「何をしたか」は要約できるが「だから何を学んだか」は人間が問いを立てて初めて引き出せる。

**引用**:

- [本人発話]「学んだこと、claudeの実装から出せないかな」
- [本人発話]「1,3,4,5入れるか」 (6 提案のうち 2 つを捨てた = 編集権は人間)

**補足: この日抽出された学び 6 行** (小ネタ集 / 単発コラム素材)

| #   | 学び                              | 文脈                                                                         |
| --- | --------------------------------- | ---------------------------------------------------------------------------- |
| 1   | alias → abbreviation (zsh-abbr)   | 展開後コマンドが見え、何を実行しているか分かる (可視性)                      |
| 2   | ガードレール vs ハーネス          | 制約は用意したがテスト制御機構は未整備。AI coding の落とし穴                 |

| 5   | PURPOSE.md の必要性               | 自動レビュー精度を上げるにはプロジェクトの意図のドキュメント化が必須         |

## S4. 「作る側は楽、読む側が大変」ワークスロップ問題

**出典: e90b3ecf**

- **失敗**: 日報の「AI活用作業」がプロジェクト別に細かく列挙され 30 行超に膨れた。AI に任せると網羅的に書けてしまい情報過多になった。
- **転機**: 「AI活用作業もう少し圧縮しようか分量多すぎるかも」と指摘。細目羅列をテーマ別 5 項目に集約した。
- **学び**: この日抽出した学びの 1 つ「受け取る側の負担」がそのまま実演された。AI 生成は作る側を楽にするが読む側の仕事を増やす。網羅性より読み手が一目で把握できる構造化が質を決める。

**教訓**: AI 時代のドキュメントの良し悪しは「どれだけ書けたか」でなく「受け取る側がどれだけ楽か」で測る。生成コストが下がった分、編集と圧縮にこそ人間の判断を投じる。

**引用**:

- [本人発話]「AI活用作業もう少し圧縮しようか分量多すぎるかも」
- [AI 学び項目]「受け取る側の負担: AI生成で作る側は楽になるが読む側の仕事が増える(ワークスロップ問題)」

## S5. 作って、検証して、消す — sunk cost と YAGNI

**出典: 419250b4**

- **失敗**: ripgrep 出力を AI 向けに最適化する Rust CLI `sharpen` を一日かけて設計・実装・テスト・audit・README まで仕上げた。
- **転機**: 検証の過程で「既存ツール rtk を超えられるか」という軸が立ち、「rg を内包すれば?」→「それすら不要では?」と自分で削ぎ落とし、最終的に**ツール自体が不要**という結論で削除した。実装後に効果検証したことで、実装コストが埋没費用になり判断が鈍るリスクに気づいた。
- **学び**: 作る前に既存手段との差分を検証すべきだった。ユーザー自身が「結局全部 yagni に帰結する」と総合している。

**教訓**: 作る前に既存手段との差分を検証せよ。実装を済ませてから効果を測ると、投じたコストが埋没費用になって「消す」判断が鈍る。提案も同じで、出す前に詰めるほど後の手戻りが減る。

**引用**:

- [本人発話]「結局全部yagniに帰結する気がするし」
- [本人発話]「学びを検証したい。今日やったclaudecodeの実装からの学び出せそう?」

## S6. 目線は揃っていない — 提案を「詰めずに」持っていった反省

**出典: 419250b4**

- **失敗**: 日経KAI のリリースフロー改善提案で、状況と理想状態は伝えたが「目線が揃っている」と思い込み、相手からアイデアが出る前提で提案を持っていった。API側・バックエンド・GitHubフローを詰めきれず「こういう場合は?」とツッコミを受けた。
- **転機**: この反省が KPT の Problem「フロー改善棄却気味。アプローチ変える必要がある」と直結していると気づいた。提案が通らないのは、相手と同じ認識のはずと思い込んで詰めの甘いまま出すから。
- **学び**: 目線が揃っている/いないの思い込みは置いて、提案を詰めてから持っていく。あるいは「一緒に考えませんか」と時間をもらう。

**教訓**: 「相手は自分と同じ認識のはず」を出発点にしない。相手への期待で動きを取り決めず、「こう動いてほしい」から逆算して接し方を設計する。認識合わせは前提でなく毎回取りにいく作業。

**引用**:

- [1on1メモ]「目線が揃っているいないの思い込みは置いて、提案を詰めてから持っていく」
- [1on1メモ]「『自分と同じ認識』は前提ではなく奇跡だと思うべき」
- [1on1メモ]「相手への期待ではなく、こう動いてほしいを想定して、どうしたらそう動いてくれるかの準備をすること」
- [KPTシート・Problem]「フロー改善棄却気味。アプローチ変える必要がある」

## S7. ツールを作って共有しただけでは、使ってもらえない

**出典: 419250b4**

- **失敗**: AI の知見発信としてツール作成・共有を続けたが「思ったほど反応が増えなかった」。仮説は「みんなのために動いているイメージがないから、自分たちに影響があると思ってもらえていない」。KPT でも「Claudeの使い方を説明してこなかった」が課題に。
- **転機**: 「作りました、使ってください」だけでは期待した状態にならないと気づき、解説会・チートシート (`/skills` の使い方含む)・使い方を可視化するダッシュボードへ。「使える状態」まで運ぶことを自分の責務に含めた。

**教訓**: ツールの提供と「使われている状態」は別物。リードの仕事は成果物を置くことでなく、相手が使える状態 (レクチャー・チートシート・可視化) まで運ぶこと。

**引用**:

- [1on1メモ]「ツールを作りました、使ってくださいだけでは期待した状態にならない」
- [KPTシート]「Claudeの使い方を説明してこなかったので解説会やる」

## S8. 「気をつける」では再発する — 仕組みで防ぐ

**出典: 419250b4**

- **失敗**: KPT の Problem に「無断マージして心配をかけさせた」。
- **転機**: シートの Try 欄に二つの方向が同居している。「気をつけておく」(意志案) と「Approvedつかないとマージできなくした」(機構案)。後者に寄せ、PR 概要に見てほしいコードオーナーを指定しその Approve を待つ運用にした。
- **学び**: 人的ミスは「気をつける」では再発する。再発防止は意志でなく仕組み (マージ制約・必須Approve) に落とす。

**教訓**: シートに「気をつける」と「仕組みで防ぐ」が並走している事実自体が判断の分岐点。再発防止は強制力のある機構に落とす。

**引用**:

- [KPTシート・Problem]「無断マージして心配をかけさせた」
- [KPTシート・Try]「仕組みで、Approvedつかないとマージできなくした」
- [KPTシート・Try]「コードオーナーのApproveに限定してもらう依頼をした方が良いかも。現状、気をつけておく」(意志案 ↔ 仕組み案の対比)

## 軸別 深掘り

3 つの記事案を、構成・導入・落としどころまで詰めた。どれも独立して 1 本になる。共通の通奏低音は「**作る/書く より、検証する/詰める ほうが効く**」。

### 軸A. 技術 × マネージメント — 「作る前に検証する」を 1 本に貫く

| 項目       | 内容                                                                                             |
| ---------- | ------------------------------------------------------------------------------------------------ |
| 一言       | 技術の「作って消す」とマネージメントの「詰めずに出す」は、同じ YAGNI の二面だ                    |
| 想定読者   | コードもチームも見るリード層。AI でコード生成が速くなった分だけ「作りすぎ」が増えた人            |
| タイトル案 | 「一日かけて作った CLI を消した日に学んだ、提案の詰め方」/「YAGNI はコードだけの話じゃなかった」 |

構成:

1. 導入 (フック): 一日かけて設計・実装・テスト・audit・README まで仕上げた Rust CLI `sharpen` を、その日のうちに消した話 (S5)。
2. なぜ消せたか / なぜ消しにくかったか: 実装後に効果検証したら、投じたコストが埋没費用になって判断が鈍ると気づいた。「作る前に既存手段との差分を検証すべきだった」。
3. 同じ失敗をマネージメントでやっていた: 日経KAI のフロー改善提案を「目線は揃っているはず」と詰めずに持っていって却下された (S6)。コードの「作ってから測る」と提案の「出してから揉む」は同型。
4. 落としどころ: 検証は後工程ではなく前工程。`sharpen` を消した判断と「提案を詰めてから持っていく」は、どちらも「着手前に差分を verify する」一つの規律。ユーザー自身の総括「結局全部 yagni に帰結する」で締める。

使う引用: [本人発話]「結局全部yagniに帰結する気がするし」/ [1on1メモ]「目線が揃っているいないの思い込みは置いて、提案を詰めてから持っていく」/ [1on1メモ]「『自分と同じ認識』は前提ではなく奇跡だと思うべき」

注意: `sharpen` を消した経緯は原典 (419250b4) の前後文脈を直読で裏取りしてから書くと、削っていく思考過程を具体的に再現できる。

### 軸B. AI 時代の働き方 — やったことは書けても、学びは空欄だった

| 項目       | 内容                                                                                                              |
| ---------- | ----------------------------------------------------------------------------------------------------------------- |
| 一言       | AI は「何をしたか」を要約できても「何を学んだか」は問わないと出てこない。そして放っておくと書きすぎて読めなくなる |
| 想定読者   | AI に日報・議事録・ドキュメントを書かせ始めた人。生成は速いが質に悩む層                                           |
| タイトル案 | 「AI に日報を書かせたら、学びの欄だけ空っぽだった」/「生成が速い時代に、人間が握るべきは編集権」                  |

構成:

1. 導入 (フック): AI 日報が「やったこと」は埋めたのに「学んだこと」を空欄 (`-`) のまま投稿した (S3)。
2. 問いを変えたら学びが出た: 「学んだこと、claudeの実装から出せないかな」と問い直した瞬間、同じログから 6 つの気づきが出た。記録の自動化と内省の自動化は別。出力 (やったこと) と成果 (学び) は別物。
3. 次の失敗は「書きすぎ」: AI に任せたら 30 行超に膨れた。受け取る側の負担 = ワークスロップ問題 (S4)。この日抽出した学びの一つが、その日のうちに実演された皮肉。
4. 落としどころ: AI 時代の記録術は「問いの設計 (何を振り返らせるか) + 受け手への圧縮 (何を削るか)」。生成コストが下がった分、人間は編集権を握る。理論的支柱に S1 (言語化が検証を呼ぶ = 問いが出力を変える) を一段だけ引く。

使う引用: [本人発話]「学んだこと、claudeの実装から出せないかな」/ [本人発話]「1,3,4,5入れるか」(編集権) / [本人発話]「AI活用作業もう少し圧縮しようか分量多すぎるかも」

補強に使える小ネタ: 学び 6 行テーブル (S3 補足) をそのまま「1 セッションから抽出できた学びの実例」として載せられる。

2. 仕掛けの正体: Decision に具体シンボルを書く → grep の動機が自動発火 → drift 発覚。「verbalize は trigger、verify が本体」。
3. 広げる①: 書けば書くほど良いわけではない。critic で「12 → 3」「17 → 5」に絞った話 (S2)。記録の質は rejection rate で測る。
4. 広げる②: 同じ「作る前/書く前に検証」が提案にも効く (S6 を一節だけ引いてマネージメント面へ接続)。
5. 落としどころ: ドキュメントの良し悪しは「書く目的」でなく「書いた後に何が自動で問われるか」。

土台と差分: 既存ドラフトは S1 中心。S2 (critic で篩う) と S6 (提案の検証) を足すと、技術記事からマネージメント寄りに半歩広げられる。既存ドラフトを直読して、どこまで書けているか差分を取るのが次の一手。

### 3 軸の使い分け

| 軸                    | 強み                                           | 弱み                                             |
| --------------------- | ---------------------------------------------- | ------------------------------------------------ |
| A 技術×マネージメント | 2 領域を 1 原理で貫けて新鮮。リード層に刺さる  | sharpen の経緯を原典で裏取りしないと具体性が薄い |
| B AI 時代の働き方     | 引用が全て本人チャット発話で純度が高い。時事性 | 技術的な深さは出ない。読み物寄り                 |
| C ADR 深掘り          | 下書き4500字が既にあり最短で出せる             | 技術色が濃く、マネージメント希望からはやや外れる |
