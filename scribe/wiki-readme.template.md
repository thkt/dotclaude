# docs/wiki

このプロジェクトのコード構造・仕様・定型手順を、issueのPlanから引用するための蓄積場所。
Planで毎回書き直す代わりに、ここ1箇所を参照して二重管理と陳腐化を防ぐ。

## 何を書くか / 書かないか

| 種類                                         | 置き場所                                   |
| -------------------------------------------- | ------------------------------------------ |
| コード構造・仕様（サブシステムの責務・構成） | このディレクトリ（wiki）                   |
| 定型手順・規約（命名規則・チェックリスト等） | このディレクトリ（wiki）                   |
| 設計判断とその履歴・経緯                     | `docs/decisions/`（ADR）。ここには書かない |

## ページ粒度

1サブシステム1ページ。ファイル名は `<サブシステム名>.md`（kebab-case）。
1機能ごとに割らない。サブシステム = 独立して責務を説明できる単位。

## ページ書式

各ページは以下の構成にする。末尾の「対象コードパス」がLint（コードとの乖離検出）の照合対象。
仕様・手順の各項目には、その根拠となるコード位置を `path/to/file` または `path/to/file:行番号` で付記する。
読み手が「この挙動はどこに書いてあるか」をページから直接辿れることが目的。

```markdown
# <サブシステム名>

## 責務

このサブシステムが何をするか（1〜3文）。

## 仕様

外から観測できる挙動の言語化。入力・出力・条件分岐・エラー時の振る舞い。
各項目にコード位置を付記する（例: 「未マージPRがあればスキップする（`run.sh:44`）」）。

## 構成

主要なファイル・モジュールと、それぞれの役割。

## 定型手順・規約

このサブシステムを触る際に繰り返す手順・命名規則・チェックリスト（あれば）。
規約の根拠がコードにある場合はコード位置を付記する。

## 対象コードパス

- `path/to/file1`
- `path/to/dir/`
```

## 育て方

`~/.claude/scribe` が自動でIngestする。初回はコードベース全体を巡回してページを基礎化し、
以降は前回実行以降のコミット差分から増分更新する。
追加・修正はPRで提案されるので、レビューしてマージ（＝承認）する。手動追記も可。
