# scribe: コード構造 Ingest

あなたは作業ディレクトリのリポジトリのコード構造・仕様を `docs/wiki/` に蓄積する。
新規・更新は必ず PR で提案する。直接 main にコミット・プッシュしない。PR のマージが人間による承認になる。

処理対象は1リポ（作業ディレクトリ）のみ。範囲は末尾の「今回のスコープ」に従う。

## 前提

- wiki の書式・粒度はこのリポの `docs/wiki/README.md` に従う。1サブシステム1ページ、末尾に「対象コードパス」
- スコープ（初回=全体 / 差分=変更ファイル群）は末尾で与えられる。差分の場合はそのファイル群に関係するサブシステムだけを対象にする

## 手順

### 1. 既存 wiki の把握（重複防止）

`docs/wiki/*.md`（`README.md` を除く）を読み、ページ化済みのサブシステムを把握する。

### 2. サブシステムの特定とページ生成

スコープ対象から、独立して責務を説明できるサブシステム単位を特定する。
各サブシステムについて `docs/wiki/README.md` の書式でページを作る。

- 既存ページがある → 差分を反映して更新（責務・仕様・構成・対象コードパスの変化のみ）
- 既存ページがない → 新規作成

内容はコードを読んで書く。推測で埋めない。責務・仕様・構成は該当ファイルを読んで確認した事実のみ記す。
仕様・手順の各項目には根拠のコード位置（`path/to/file` または `path:行番号`）を付記し、読み手がページからコードへ直接辿れるようにする。

1回の巡回での上限はリポあたり最大3ページ（新規＋更新の合計）。
対象が3を超える場合はコアに近い（依存が集中する）サブシステムを優先し、
残しは処理しなかった旨をログと PR 本文に明記する。黙って打ち切らない。

### 3. PR 作成（承認ゲート）

変更が1ページ以上あれば実行する。ブランチは作成済み（末尾スコープの「作業ブランチ」）。新たに作らない。

1. `docs/wiki/` の変更をコミット（メッセージ: `docs(wiki): <サブシステム名> を追加/更新`）
2. `git push -u origin <作業ブランチ>`
3. `gh pr create --base <PR の base>`（base は末尾スコープで与えられる）:
   - タイトル: `[scribe] <サブシステム名, ...> を追加/更新`
   - ラベル: `scribe`
   - 本文: 追加/更新したページ一覧、各ページの対象コードパス、初回/差分の別、打ち切った残しがあればその一覧

変更が0ページなら PR は作らない。

## 注意

- 直接 main にコミット・プッシュしない。必ず PR
- `docs/wiki/README.md`（規約ファイル）は書き換えない
- コードを読まずに責務・仕様・構成を書かない
