# AI支援開発

**デフォルトマインドセット**: AIは生産性ツール、人間が意思決定者

## コア哲学

**「AIが生成、人間が検証」** - AI支援テストケース生成の研究に触発

AIツールは強力なアシスタントだが、人間の判断、ドメイン知識、コードレビューを置き換えることはできない。目標は人間の監視を通じて品質を維持しながら、AIを生産性向上に活用すること。

## 主要原則

### 1. AIはツールであり、代替ではない

```markdown
人間の役割:
- ✅ 最終決定を下す
- ✅ AI出力をレビュー・改善
- ✅ コードの「なぜ」を理解
- ✅ ビジネスロジックの正確性を保証

AIの役割:
- ✅ ボイラープレートコードを生成
- ✅ 実装を提案
- ✅ テストの雛形を作成
- ✅ パターンを特定
```

**覚えておく**: AIはビジネスコンテキストを理解しない - それはあなたの仕事。

### 2. 品質 = プロンプトの品質

研究結果に基づき、AI出力の品質は指示の品質に大きく依存:

```typescript
// 悪い例: 曖昧なプロンプト
"ユーザーサービスを作成"

// 良い例: コンテキスト付きの具体的なプロンプト
"UserRepositoryの既存パターンに従ってユーザーサービスを作成。
- データベースには依存性注入を使用
- AAAテストパターンに従う
- スタイルはtests/services/の既存テストを参照"
```

**重要な洞察**: 指示が具体的であるほど、出力は良くなる。

### 3. 過去のパフォーマンス > 抽象的ルール

**最も効果的なアプローチ**: 既存コードを例として参照

```bash
# AIにコード生成を依頼する前に:
1. 類似の既存実装を見つける
2. プロンプトで例として共有
3. AIに同じパターンに従うよう依頼
```

**なぜ機能するか**:

- 具体例 > 理論的ガイドライン
- コードベースとの一貫性を保証
- 大幅な修正の必要性を減らす

**研究根拠**: 過去のパフォーマンスを参照として使用すると、最も実用的で即座に使用可能なAI生成コードが得られることが研究で示されている。

### 4. 人間のレビューは必須

```markdown
AI生成コードワークフロー:

1. AIがコードを生成
2. ✅ 人間が以下をレビュー:
   - ビジネスロジックの正確性
   - エッジケースの処理
   - セキュリティへの影響
   - パフォーマンスの考慮
3. 人間が改善・承認
4. コードが本番環境へ

ステップ2-3を省略しない。
```

**重要なポイント**: AI生成コードは下書きであり、最終製品ではない。

## 実践的な適用

**テスト生成**: 既存テストを検索（grep）→ 例を付けてプロンプト → アサーション、モック、カバレッジをレビュー

**コードレビューチェックリスト**: ビジネスルールは正しい？エッジケースはカバー？エラー処理は適切？標準に従っている？

## 開発原則との統合

| 原則 | AIの役割 | 人間の役割 | 重要な洞察 |
| --- | --- | --- | --- |
| **TDD/ベビーステップ** [@./TDD_RGRC.md] | テストの雛形を生成 | 要件からアサーションを記述 | ベビーステップは依然として適用 - 一度に1テスト |
| **オッカムの剃刀** | ソリューションを生成 | シンプルさのフィルターを適用 | AIは過剰設計しがち - すべての抽象化を疑問視 |
| **プログレッシブエンハンスメント** [@./PROGRESSIVE_ENHANCEMENT.md] | シンプルバージョンを生成 | 実際のニーズに基づいて拡張 | 「将来対応」の提案を実装しない |
| **リーダブルコード** [@./READABLE_CODE.md] | 巧妙なコードを生成する可能性 | 可読性を保証 | 巧妙さより明確さのためにリファクタ |

## 警告サイン

| 警告タイプ | サイン |
| --- | --- |
| **過度の依存** | レビューなしで受け入れ、コードを理解していない、テストをスキップ、絶対視 |
| **活用不足** | すべてのボイラープレートを手動で記述、AIで探索しない、提案を無視、コンテキスト不足 |

## ベストプラクティス

| プラクティス | 方法 | タイミング |
| --- | --- | --- |
| **リッチなコンテキスト** | 提供: タスク詳細、既存パターン（リンク/ペースト）、ビジネスルール、エッジケース、テストフレームワーク、制約 | すべてのAIリクエスト - 具体性 = 品質 |
| **反復的な改善** | 生成 → レビュー → プロンプト改善 → 再生成 → 繰り返し | 初期出力に改善が必要な場合 |
| **出力から学ぶ** | 良いコード: なぜか分析、プロンプトを記録、パターン再利用<br>悪いコード: プロンプトのギャップを特定、将来のために記録、技術を改善 | すべてのAIインタラクション後 |

## ガードレール

| カテゴリ | ルール |
| --- | --- |
| **禁止** | レビューなしのコミット、専門家レビューなしのセキュリティ重要部分、アーキテクチャの盲目的受け入れ、テストのスキップ |
| **必須** | 常に人間のレビュー、AIコードのテスト、使用前に理解、AI支援を記録 |

## 覚えておく

> 「最高のAI支援開発は、AIの速度と人間の知恵を組み合わせる。」

重要なポイント:

- **AIは加速**、人間が品質を保証
- **過去の例**は抽象的な指示に勝る
- **レビューは必須**、オプションではない
- **コードはあなたのもの**、AIのものではない

AIはアシスタントであり、代替ではない。賢く使おう。
