# テスト生成 - 体系的なテスト設計

**コア原則**: アドホックな推測ではなく、実証済みの技法を使用して体系的にテストを設計。

## 哲学

**量より質**: 適切に設計されたテストは、少ないテストケースでより多くのバグを捕捉。

### 主要原則

1. **体系的なカバレッジ** - 完全性を保証するために形式的技法を使用
2. **効率性** - 最小限のテストですべてのシナリオをカバー
3. **保守性** - テストは理解・更新しやすくあるべき
4. **フレームワーク非依存** - 原則はテストフレームワークに関係なく適用

## テスト設計技法

| 技法           | 概念                                                            | パターン                            | 例                                                                                      |
| -------------- | --------------------------------------------------------------- | ----------------------------------- | --------------------------------------------------------------------------------------- |
| **同値分割**   | 同様に動作する入力をグループに分割                              | 各パーティションから1つの値をテスト | 年齢検証: <18（無効）、18-120（有効）、>120（無効）→ 17、30、121をテスト                |
| **境界値分析** | バグが潜むパーティションのエッジをテスト                        | [min-1、min、max、max+1]をテスト    | 年齢境界18、120 → 17、18、120、121をテスト                                              |
| **決定表**     | 複雑なロジック（3つ以上の条件）のすべての条件組み合わせをマップ | 真理値表を作成、テストを導出        | アクセス制御: isLoggedIn × isPremium × isActive → テストすべき5つの意味のある組み合わせ |

## カバレッジ目標

### 推奨ターゲット

業界のベストプラクティスとコスト対効果分析に基づく:

- **C0（ステートメントカバレッジ）**: 最低80%
- **C1（ブランチカバレッジ）**: 最低70%

### なぜこれらの数値？

- **80/70ルール**: 収穫逓減なしでほとんどのクリティカルパスをカバー
- **コスト vs 効果**: 90%を超えると、わずかな利益のために労力が指数関数的に増加
- **クリティカルパスに焦点**: 100%カバレッジはバグフリーコードを保証しない

### カバレッジの測定

```bash
# ほとんどのフレームワークがカバレッジレポートをサポート
npm test -- --coverage          # Jest, Vitest
yarn test --coverage            # Yarnで
pytest --cov=src tests/         # Python
```

### カバレッジ分析

```markdown
優先カバレッジ領域:

1. **クリティカルなビジネスロジック** - 決済、認証、データ整合性
2. **パブリックAPI** - すべての公開インターフェース
3. **エラーハンドラー** - 例外パス
4. **複雑な条件** - 決定ポイント

低優先度:

- ロジックなしのゲッター/セッター
- フレームワーク生成コード
- シンプルなデータ構造
- 設定ファイル
```

## テスト設計ワークフロー

### ステップ1: テストユニットを特定

テスト可能なユニットに分解（通常はパブリックメソッド）:

```typescript
// 良い例: 一度に1つのメソッドをテスト
describe("UserService.validateAge", () => {
  // validateAgeのテスト
});

// 悪い例: クラス全体を一度にテスト
describe("UserService", () => {
  // 複数メソッドのテストが混在
});
```

### ステップ2: テストシナリオを設計

上記の**テスト設計技法**テーブルを各ユニットに適用する。

### ステップ3: テストケースを記述

```typescript
// 明確なテスト構造 - テストごとに1つのアサーション焦点
describe("validateAge", () => {
  // 同値パーティション
  describe("無効な年齢", () => {
    test("負の年齢を拒否", () => {
      expect(validateAge(-1)).toBe(false);
    });

    test("最大値を超える年齢を拒否", () => {
      expect(validateAge(150)).toBe(false);
    });
  });

  // 境界値
  describe("境界条件", () => {
    test("最小有効年齢を受け入れ", () => {
      expect(validateAge(18)).toBe(true);
    });

    test("最小値の1つ下を拒否", () => {
      expect(validateAge(17)).toBe(false);
    });
  });

  // 有効パーティション
  describe("有効な年齢", () => {
    test("典型的な成人年齢を受け入れ", () => {
      expect(validateAge(30)).toBe(true);
    });
  });
});
```

## 他の原則との統合

| 原則                 | テストへの適用                                                   | キーパターン                                                                          |
| -------------------- | ---------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| **オッカムの剃刀**   | シンプルなケースにはテストビルダー、ヘルパーを避ける             | 直接テスト > 複雑なテストインフラ                                                     |
| **リーダブルコード** | 明確な命名、テストごとに1つの焦点、AAA構造、最小限のセットアップ | テスト名は何を検証するかを説明                                                        |
| **ミラーの法則**     | テストをカテゴリにグループ化（7±2制限）                          | `describe('validation', () => { describe('email', ...), describe('password', ...) })` |

## フレームワーク非依存パターン

| フレームワーク  | 構造                                                             | アサーション                        |
| --------------- | ---------------------------------------------------------------- | ----------------------------------- |
| **Jest/Vitest** | `describe('unit', () => { test('behavior', () => {...}) })`      | `expect(actual).toBe(expected)`     |
| **Mocha/Chai**  | `describe('unit', () => { it('should behavior', () => {...}) })` | `expect(actual).to.equal(expected)` |
| **Pytest**      | `class TestUnit: def test_behavior(self): ...`                   | `assert actual == expected`         |
| **RSpec**       | `describe 'unit' do it 'behavior' do ... end end`                | `expect(actual).to eq(expected)`    |

## よくある落とし穴

| 落とし穴               | 問題                                                  | 解決策                                                                 |
| ---------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------- |
| **ランダムテスト**     | 根拠なく任意の値（25、50、75）をテスト - 境界を見逃す | 体系的設計を使用: 境界（min-1、min、max、max+1）+ 1つの同値値をテスト  |
| **実装のテスト**       | 内部メソッド呼び出しをスパイ - 脆く、実装に結合       | 動作をテスト: 結果を検証、達成方法ではなく                             |
| **不完全なカバレッジ** | ハッピーパスのみをテスト - エッジケースを見逃す       | すべてのパーティション + 境界をカバー: [min-1、min、valid、max、max+1] |

## クイックリファレンス

### テスト設計チェックリスト

各パブリックメソッド/関数について:

- [ ] 同値パーティションを特定
- [ ] 各パーティションから1つの値をテスト
- [ ] 境界値を特定
- [ ] 各境界で[min-1、min、max、max+1]をテスト
- [ ] 複雑なロジック（3つ以上の条件）には決定表を作成
- [ ] C0カバレッジ≥80%を確認
- [ ] C1カバレッジ≥70%を確認
- [ ] テストは読みやすく保守可能

## 覚えておく

> 「テストの目標は100%カバレッジを達成することではなく、適切に設計されたテストでバグを効率的に捕捉すること。」

**重要なポイント**:

- **体系的 > ランダム**: 推測ではなく形式的技法を使用
- **質 > 量**: 少数の適切に設計されたテストが多数のランダムテストに勝る
- **境界が重要**: ほとんどのバグはエッジに潜む
- **シンプルに保つ**: テストは本番コードより理解しやすくあるべき

**判断フレームワーク**:

テストを書く前に尋ねる:

1. 同値パーティションは何？
2. 境界値は何？
3. ロジックは決定表が必要なほど複雑？
4. テストはすべての意味のあるシナリオをカバーしている？
5. カバレッジを維持しながらテストを簡素化できる？
