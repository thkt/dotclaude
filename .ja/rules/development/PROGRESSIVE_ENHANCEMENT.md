# プログレッシブエンハンスメント - アウトカムファースト開発

**デフォルトアプローチ**: シンプルに構築 → 段階的に拡張

## コア哲学

- **根本原因**: 「どう修正するか？」ではなく「なぜ？」
- **予防 > パッチ**: 最良の解決策は問題を予防
- **シンプル > 複雑**: エレガンス = 正しい問題を解決

## 優先度

1. **HTML** - 構造
2. **CSS** - ビジュアル/レイアウト
3. **JavaScript** - 必要な場合のみ

## CSS優先ソリューション

- **レイアウト**: Grid/Flexbox
- **配置**: Transform（リフローなし）
- **表示/非表示**: visibility/opacity
- **レスポンシブ**: Media/Containerクエリ
- **状態**: :target, :checked, :has()

## 例

```css
/* ✅ CSS Gridオーバーレイ */
.container {
  display: grid;
}
.panel {
  grid-column: 1 / -1;
  grid-row: 1 / -1;
}
```

## 適用対象

レイアウト、配置、表示/非表示、レスポンシブ、アニメーション、視覚状態

## 覚えておく

「最高のコードはコードなし」 - CSSで解決できるなら、JSをスキップ

## アウトカムファースト開発

**コア原則**: アーキテクチャではなく、アウトカムを出荷

### 実装フェーズ

| フェーズ       | アクション             | トリガー             |
| -------------- | ---------------------- | -------------------- |
| 動くようにする | 直接の問題を解決       | 常に最初             |
| 堅牢にする     | エラーハンドリング追加 | エラー発生時         |
| 速くする       | 最適化                 | 遅さが測定されたとき |
| 柔軟にする     | オプション追加         | ユーザー要求時       |

### 判断フレームワーク

| 質問                         | 「いいえ」なら |
| ---------------------------- | -------------- |
| 今存在する実際の問題を解決？ | 追加しない     |
| 本番環境で実際に失敗？       | 追加しない     |
| ユーザーから苦情？           | 追加しない     |
| 測定された証拠？             | 追加しない     |

### プログレッシブエンハンスメントのマインドセット

| 原則             | アクション                 |
| ---------------- | -------------------------- |
| ハッピーパス優先 | シンプルに始める           |
| 現実駆動         | 必要なときにのみ複雑さ追加 |
| コードは負債     | すべての行にコストあり     |
| YAGNIデフォルト  | 投機的に構築しない         |

## 重要なポイント

「最高のコードは、存在する必要のないコード」
