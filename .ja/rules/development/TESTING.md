# Testing

カバレッジの観点 / 優先領域 / テスト命名 / delta-based gate を扱う。

## Coverage は補助指標

カバレッジは「コードが実行されたか」を測る指標であり、「正しく検証されたか」までは測らない。数値そのものを目標にせず、何を確かめるべきかという観点と、優先すべき領域に基づいて品質を守る。

| Level | 名称     | 基準                                             |
| ----- | -------- | ------------------------------------------------ |
| C0    | 命令網羅 | 各行を少なくとも 1 回実行                        |
| C1    | 分岐網羅 | if / switch の両側を実行                         |
| C2    | 条件網羅 | 複合条件の各オペランドを true / false 両方で実行 |

## 優先領域

全領域を一律検証せず、影響の大きい領域ほど深く検証する。ここでの深度とは、その領域で到達すべき検証レベルを指す。故障が金銭損失 / 認可漏れ / データ消失に直結する領域は C2 とする。C1 の領域であっても、複合条件（&& / || / 三項演算子）を含む箇所は C2 へ引き上げる。

| 領域               | 例                             | 深度 |
| ------------------ | ------------------------------ | ---- |
| 業務ルール中核     | 計算 / 判定 / 変換ロジック     | C2   |
| 権限制御           | ロール判定 / 認可              | C2   |
| 課金・請求         | 金額計算 / レート判定          | C2   |
| データ整合性       | トランザクション / unique 制約 | C1   |
| ステータス遷移     | 状態マシン                     | C1   |
| ユーザー導線       | 予約 / 決済                    | C1   |
| 表示コンポーネント | props レンダリング             | C0   |
| 薄いラッパー       | 委譲のみ                       | C0   |

## テスト作成の順序

観点から始めて、カバレッジは漏れ検出にだけ使う。

1. テスト対象の仕様を読み、満たすべき条件を列挙する
2. シグネチャとドメインに該当する観点を観点チェックリストから選ぶ
3. 条件 (Step 1) と観点 (Step 2) を満たすテストを書く
4. カバレッジを実行し、未通過の分岐/条件には対応する観点でテストを追加する

### 観点チェックリスト

| 観点         | 確認内容                                                                              |
| ------------ | ------------------------------------------------------------------------------------- |
| 同値分割     | 同じ振る舞いになる入力をグループ化し、各グループの代表値 1 つをテスト                 |
| 境界値       | 各境界の両側 (min - 1 と min、max と max + 1)。ゼロと空も特異値として加える           |
| 分岐 (C1)    | 各 if / switch の全経路                                                               |
| 条件 (C2)    | 各複合条件のオペランドごとに true / false                                             |
| 組み合わせ   | 独立した条件が 2 つ以上絡むなら、デシジョンテーブルを書いて各行をカバー               |
| 状態遷移     | 許可される遷移は通り、禁止される遷移は拒否されることを確認                            |
| 異常系       | 不正入力 / null を与え、例外パスを通す                                                |
| 事故パターン | 二重送信 / タイムゾーン / 権限バイパス / 中途半端な状態など、プロダクトで起きうる事故 |
| 並行性       | 競合 / 順序依存。対象が async / multi-threaded のときだけ適用                         |

## テスト命名

テスト名は検証する仕様を述べる (例: "amount が負のとき deposit を拒否する")。

## バグ修正の再現テスト

バグ修正は、原因を直す前にそのバグを再現するテストを 1 件追加し、Red を確認してから修正に入る。修正後は全スイートを実行し、再現テストが green に変わりリグレッションが残っていないことを確認する。この再現テストが恒久的なガードとして残り、同じバグの再発を捕える。

## テストダブルの優先順位

依存は実物に近いほど統合のずれを捕える。下記の優先順で選び、上位が使えない理由 (遅い / 外部副作用 / 非決定的) があるときだけ下位に落とす。

| 種別 | いつ使うか                                              |
| ---- | ------------------------------------------------------- |
| Real | 実物が十分速く副作用が閉じているなら最優先              |
| Fake | 実物が遅い / 外部依存。軽量な動作実装で挙動ごと代替する |
| Stub | 戻り値だけ固定できれば足りるとき                        |
| Mock | 呼び出し自体 (回数 / 引数) の検証が目的のときだけ       |

## Coverage Gate

PR で C0 / C1 を下げない。評価は差分ベースで行い、絶対値の下限は設けず、数値を目標化しない。絶対値が必要な場合は、非機能要件として定義する（例: セキュリティツールで C0 が 90% 以上）。例外は、自動生成コード / データ定義 / テストファイル / 移行中のレガシーコード。
