# Testing

カバレッジ閾値は `THRESHOLDS.md` 参照 (delta-based gate)。本ファイルは観点と質を扱う。

## Coverage は補助指標

カバレッジは「コードが実行されたか」を測る指標。「正しく検証されたか」は測らない。数字を目標化せず、観点と優先領域から守る。

| Level | 観点                                          |
| ----- | --------------------------------------------- |
| C0    | 命令網羅: 各行を少なくとも 1 回実行           |
| C1    | 分岐網羅: if / switch の両側を実行            |
| C2    | 条件網羅: 複合条件の各オペランド true / false |

## 優先領域

均一に観点を追わず、影響大の領域から深く守る。`Required depth` で Level を判定する。

| 領域               | 例                            | Required depth                                     |
| ------------------ | ----------------------------- | -------------------------------------------------- |
| 業務ルール中核     | 計算・判定・変換ロジック      | Up to C2                                           |
| 権限制御           | ロール判定・認可              | Up to C2                                           |
| 課金・請求         | 金額計算・レート判定          | Up to C2                                           |
| データ整合性       | トランザクション・unique 制約 | Up to C1; condition が && / \|\| / ternary なら C2 |
| ステータス遷移     | 状態マシン                    | Up to C1; condition が && / \|\| / ternary なら C2 |
| ユーザー導線       | 予約・決済                    | Up to C1; condition が && / \|\| / ternary なら C2 |
| 表示コンポーネント | props レンダリング            | C0                                                 |
| 薄いラッパー       | 委譲のみ                      | C0                                                 |

中核扱いの判定: 故障時に直接ユーザー影響 (金銭損失・認可漏れ・データ消失) があれば中核。

## アンチパターン

### AI 生成テストの偏り

AI にテスト生成を依頼すると次の偏りが出る。生成後にチェックリストを通し、当てはまる項目で再プロンプトする。

| 偏り                       | 再プロンプト文                                                                                           |
| -------------------------- | -------------------------------------------------------------------------------------------------------- |
| 正常系のみ                 | 不正入力・null・例外パスのテストを追加せよ                                                               |
| 無難なケースのみ           | 各入力パラメータの edge case を列挙し、それぞれにテストを追加せよ                                        |
| C0 達成、C1 弱い           | 各 if / switch の false 分岐のテストを追加せよ                                                           |
| C1 通過、C2 漏れ           | 各複合条件について、各オペランドの true / false を独立してテストせよ                                     |
| 条件を組み合わせていない   | 複合条件のデシジョンテーブルを作成し、各行のテストを追加せよ                                             |
| 不正遷移を確認していない   | 禁止される状態遷移が拒否されることをアサートするテストを追加せよ                                         |
| プロダクト固有の事故が無い | 事故パターン (二重送信, タイムゾーン, 権限バイパス, 中途半端な状態) を列挙し、それぞれにテストを追加せよ |
| アサーション弱い           | toBeTruthy を期待値の正確な比較に置き換えよ                                                              |
| テスト意図が曖昧           | 各テスト名を、検証する仕様を述べる形に変更せよ                                                           |

### 数字目的化の罠

カバレッジ数値を追うと次が起きる。

- 意味あるアサーションのないテストが増える
- C0 だけ満たす薄いテストでパディング
- 実装詳細に結合し、リファクタに脆くなる
- メンテコストだけが上がる
- 数字達成で偽の安心感が出て、本当のリスクへの注意が薄れる

`THRESHOLDS.md` が delta-based gate を採用するのはこの罠を避けるため。

## テスト作成の順序

カバレッジから始めない。観点から始める。カバレッジは漏れ検出にだけ使う。

| Step | アクション                                                                 |
| ---- | -------------------------------------------------------------------------- |
| 1    | 関数 / 機能の仕様を読み、満たすべき条件を再述する                          |
| 2    | 関数のシグネチャとドメインに該当する観点を、下のチェックリストから列挙する |
| 3    | Step 1 (仕様) と Step 2 (観点) を AI に文脈として渡してテストを生成する    |
| 4    | カバレッジを実行し、未通過の分岐 / 条件には対応する観点でテストを追加する  |

### 観点チェックリスト

| 観点         | 確認内容                                                                            |
| ------------ | ----------------------------------------------------------------------------------- |
| 同値分割     | 入力を振る舞いのクラスでグルーピング。クラス毎に代表値 1 つ                         |
| 境界値       | min, min - 1, max, max + 1, ゼロ, 空                                                |
| 分岐         | 各 if / switch の各経路                                                             |
| 条件         | 各複合条件の各オペランド: true と false                                             |
| 組み合わせ   | 2 条件以上は先にデシジョンテーブルを書き、各行をカバー                              |
| 状態遷移     | 各遷移。許可される遷移と禁止される遷移の両方                                        |
| 異常系       | 不正入力, null, 例外パス                                                            |
| 事故パターン | プロダクト固有の事故パターン (二重送信, タイムゾーン, 権限バイパス, 中途半端な状態) |
| 並行性       | 競合, 順序依存 (async または multi-threaded のみ)                                   |

## 質チェック

merge 前に実行する。質は量に優先する。

| 項目         | Reject                 | Accept                                                    |
| ------------ | ---------------------- | --------------------------------------------------------- |
| アサーション | toBeTruthy のみ        | 期待値の正確な比較                                        |
| 境界値       | 中央値のみ             | 各境界の両側 (観点チェックリスト参照)                     |
| 異常系       | 正常系のみ             | 不正入力と例外パスをカバー                                |
| 分岐         | 片側のみ               | 両側                                                      |
| 条件         | 結果のみ               | 各オペランド: true と false                               |
| テスト名     | "test1", "should work" | 仕様を述べる (例: "amount が負のとき deposit を拒否する") |
