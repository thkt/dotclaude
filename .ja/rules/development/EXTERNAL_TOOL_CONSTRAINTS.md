# 外部ツール制約チェック

ルール: テストやコード実装の前に、外部ツールの制約を確認する。

## よく踏む地雷 Top 10

| #   | ツール/コンテキスト  | 地雷                                       | 確認方法                                  |
| --- | -------------------- | ------------------------------------------ | ----------------------------------------- |
| 1   | Git ブランチ/タグ    | `[` `]` `~` `^` `:` `\` `..` スペース禁止  | `git check-ref-format --branch "name"`    |
| 2   | Cargo crate名        | `_` は `-` に変換、英数字と `-` のみ       | `cargo search crate-name` または Docs確認 |
| 3   | npm パッケージ名     | 214文字以下、小文字強制、`@scope/` 形式    | `npm info package-name`                   |
| 4   | Docker イメージ/タグ | 小文字のみ、`/` は namespace、`:` は tag   | 公式ドキュメント確認                      |
| 5   | Shell 配列           | POSIX に配列なし、bash/zsh 構文差異        | `echo $0` でシェル確認                    |
| 6   | 正規表現エンジン     | PCRE vs ERE vs BRE vs JavaScript 差異      | ツールの `--help` で使用エンジン確認      |
| 7   | macOS ファイル名     | `:` 禁止、デフォルトで大文字小文字区別なし | `touch "test:file"` でエラー確認          |
| 8   | semver バージョン    | `MAJOR.MINOR.PATCH` 必須、先頭 `v` は別物  | `npx semver "version"` で検証             |
| 9   | 環境変数名           | 英数字と `_` のみ、数字始まり禁止          | `export` で既存確認、POSIX準拠            |
| 10  | JSON キー名          | ダブルクォート必須、trailing comma 禁止    | `jq . file.json` でパースエラー確認       |

## チェックタイミング

| フェーズ     | アクション                                    |
| ------------ | --------------------------------------------- |
| テスト設計前 | テスト対象の外部ツール制約を WebSearch で確認 |
| 実装前       | 命名規則・フォーマット制約を確認              |
| エラー発生時 | 「制約違反では？」を最初に疑う                |

## 確認の原則

| ルール               | 説明                                       |
| -------------------- | ------------------------------------------ |
| 公式ドキュメント優先 | Stack Overflow より公式ドキュメント        |
| 実機確認             | 可能なら `--dry-run` や検証コマンドで確認  |
| バージョン依存を意識 | ツールのバージョンで挙動が変わる可能性あり |

## 地雷を踏んだときの対処

1. エラーメッセージから制約違反を特定
2. 公式ドキュメントで正確な制約を確認
3. テストケースから無効なケースを除外または修正
4. 同種の制約違反がないか横展開チェック
