# Tidyings - Kent Beck風

小さなコード改善を行うとき、Kent Beckのように考え行動する - 機能を壊すリスクなしにマイクロ改善を蓄積。

## 実装ポリシー

### タイミング

メインタスク後、コミット前、編集したファイルのみ

### Kent Beckのように考える

- 「これは動作を変更する？」 - Yesならそれはtidyingsではない
- 「この変更に不安を感じる？」 - Yesならより小さくする
- 「これをより小さくできる？」 - 小さな変更はより安全
- 「次の小さな改善は？」 - 常に次のマイクロステップを探す

## 具体的な基準

1. **空白**: 末尾スペース、複数スペース→単一、EOF整理、インデント修正
2. **インポート**: 未使用を削除、アルファベット順、同パッケージをグループ、タイプでグループ
3. **変数**: 未使用を削除、単一使用をインライン化（可読性が保たれる場合）、let→const
4. **コメント**: 解決済みTODOを削除、冗長な説明、デッドコード
5. **型（TS）**: 推論可能なアノテーションを削除、any→具体的、重複を統合
6. **フォーマット**: 一貫したセミコロン/クォート/カンマ（プロジェクト規約に従う）
7. **命名**: タイポ修正、ケースの不一致を修正

## 絶対にしない

**絶対に**: ロジック変更、構造変更、新機能、パフォーマンス最適化、パブリックAPI変更、テスト修正

## レポートフォーマット

`🧹 Tidyings: 空白 ✓ インポート ✓ 未使用コード ✓ その他 ✓`
