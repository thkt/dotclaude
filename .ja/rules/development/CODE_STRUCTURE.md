# コード構造

## コードベース一貫性原則

優先順：1 > 2 > 3（衝突時は上位が勝つ）

| 優先 | 原則           | 内容                                                                             |
| ---- | -------------- | -------------------------------------------------------------------------------- |
| 1    | 既存パターンに従う | 構造・命名・エラー処理・状態管理のパターンを揃える。変更はユーザー指示のときだけ |
| 2    | ないならフラット | AIが構造を発明しない。分割判断はユーザーに委ねる。指示があれば戦略の手札から選択 |
| 3    | コロケーション | 分割するときは一緒に変わるものを近くに                                           |

## デフォルト値

フレームワーク規約がある場合はそちらが優先。

| 項目           | デフォルト               |
| -------------- | ------------------------ |
| ディレクトリ深さ | 3階層まで              |
| 横断モジュール | `shared/`                |
| テスト配置     | ソースと同階層           |

## 構造戦略の手札

既存構造がないとき、What/Howで選択。階層ごとに独立適用する（例：プロジェクト
全体はパイプライン、ステージ内はフィーチャー）。

| 戦略         | ハマるとき                          |
| ------------ | ----------------------------------- |
| フラット     | 小規模、単機能                      |
| パイプライン | データが一方向に流れる（CLI, ETL）  |
| レイヤー     | リクエスト-レスポンス型（API）      |
| フィーチャー | 独立した機能群（UI, SaaS）          |
| ドメイン     | 境界づけられたコンテキストが複数    |

```
小規模                  → フラット
概念1つ + 線形          → パイプライン
概念1つ + ツリー        → レイヤー
概念複数 + ツリー       → フィーチャー
概念複数 + 複雑         → ドメイン
```

## ファイル内整理

関数定義順は呼び出し順に揃える。実装時に適用し、TIDYINGSでは並べ替えしない。
