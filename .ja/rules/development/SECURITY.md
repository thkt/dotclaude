# セキュリティ

guardrails（静的検出）を補完する。guardrailsが検出できないパターン——バリデーションの欠落、認可チェックの不在、ミドルウェアの未設置——をカバーする。

## 入力

| ルール | 詳細 |
| ------ | ---- |
| MUST | すべてのAPIエンドポイント入力をZodスキーマで検証する（クライアント・サーバー両面） |
| MUST | 検索クエリ・自由入力フィールドに `.max()` で長さ上限を設ける |
| MUST NOT | クライアントから受け取った型情報（`as`, `satisfies` 含む）をそのままDBに渡す |

## アクセス制御

| ルール | 詳細 |
| ------ | ---- |
| MUST | リソースアクセス時にオーナーシップを検証する（`resource.userId === currentUser.id`） |
| MUST NOT | URLパラメータ・ボディの数値IDだけでアクセス制御を行う（IDOR） |
| MUST | 認証が必要なルートグループにミドルウェアをマウントする（個別エンドポイントでの確認は漏れが出る） |
| MUST NOT | 未認証ユーザーにリソースの存在を示す情報を返す（404 を優先し 403 を避ける） |
| MUST | 認証エンドポイント（login, register, password-reset）にレート制限を設ける |
| MUST | 外部API呼び出しを含むエンドポイントにリクエスト制限を設ける |

## シークレット

| ルール | 詳細 |
| ------ | ---- |
| MUST | 必須の環境変数が未設定の場合はフォールバックせずエラーをスローする |
| MUST | セッションID・トークン・ワンタイムコードには `crypto.randomBytes()` / `crypto.randomUUID()` を使う（`Math.random()` は暗号論的に安全でない） |

## 出力

| ルール | 詳細 |
| ------ | ---- |
| MUST NOT | スタックトレース・内部パス・DB詳細をレスポンスに含める（汎用メッセージを返し、詳細はサーバーログのみ） |
| MUST NOT | 認証失敗時に原因を分けて返す（例：「パスワードが違います」「メールが存在しません」） |
