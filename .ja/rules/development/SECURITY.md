# セキュリティ

guardrails（静的検出）を補完する。バリデーションの欠落、認可チェックの不在、ミドルウェアの未設置など、guardrails が検出できないパターンをカバーする。

## 入力

- MUST すべてのAPIエンドポイント入力をスキーマで検証する（クライアント・サーバー両方）
- MUST 検索クエリ・自由入力フィールドに長さ上限を設ける
- MUST NOT クライアントから受け取った型情報を DB や他の信頼境界にそのまま渡さない

## アクセス制御

- MUST リソースアクセス時にオーナーシップを検証する（リクエスターが対象を所有しているか）
- MUST NOT URLパラメータ・ボディの数値IDだけでアクセス制御を行う（IDOR リスク）
- MUST 認証が必要なルートグループにミドルウェアをマウントする
- MUST NOT 未認証ユーザーにリソースの存在を示す情報を返す（404 を優先し 403 を避ける）
- MUST 認証エンドポイント（login, register, password-reset）にレート制限を設ける
- MUST 外部API呼び出しを含むエンドポイントにリクエスト制限を設ける

## シークレット

- MUST 必須の環境変数が未設定の場合はエラーをスローする。ハードコード値にフォールバックしない
- MUST セッションID・トークン・ワンタイムコードには暗号論的に安全な乱数ソースを使う

## 出力

- MUST NOT スタックトレース・内部パス・DB詳細をレスポンスに含める。汎用メッセージを返し、詳細はサーバーログのみ
- MUST NOT 認証失敗時に原因を分けて返す（例：「パスワードが違います」「メールが存在しません」）
