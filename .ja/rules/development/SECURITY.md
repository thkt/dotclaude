# Security

guardrails (静的検出) を補完する。検証の欠落、認可チェックの欠落、ミドルウェアの欠落をカバーする。

## Input

| Directive | Rule                                                                                                          |
| --------- | ------------------------------------------------------------------------------------------------------------- |
| MUST      | すべての API エンドポイント入力をクライアント側とサーバー側でスキーマで検証する                               |
| MUST      | 検索クエリと自由記述フィールドに長さ制限を適用する                                                            |
| MUST      | `Content-Length` を検証し `NaN` を拒否してペイロードサイズのバイパスを防ぐ                                    |
| MUST      | `returnUrl` などのリダイレクト先をホスト許可リストで検証する。`javascript:` と `data:` スキームを拒否する     |
| MUST NOT  | クライアント由来の型情報を DB やその他の信頼境界へ直接渡さない                                                |
| MUST NOT  | リクエストオブジェクトを `...body` スプレッドで構築しない。明示的にフィールドを代入してプロトタイプ汚染を防ぐ |

## Access Control

| Directive | Rule                                                                                          |
| --------- | --------------------------------------------------------------------------------------------- |
| MUST      | リソースアクセス時に所有権を検証する (要求者がターゲットを所有する)                           |
| MUST      | 認証ミドルウェアはエンドポイント単位ではなくルートグループにマウントする                      |
| MUST      | 認証エンドポイント (login, register, password-reset) にレート制限を適用する                   |
| MUST      | 外部 API 呼び出しを誘発するエンドポイントにリクエスト制限を適用する                           |
| MUST      | 状態変更リクエスト (POST/PUT/PATCH/DELETE) の CSRF トークンを Double Submit Cookie で検証する |
| MUST NOT  | アクセス制御に URL パラメータや body の数値 ID のみを使わない (IDOR リスク)                   |
| MUST NOT  | 未認証ユーザーにリソース存在を露呈しない (403 より 404 を選ぶ)                                |

## Secrets

| Directive | Rule                                                                                             |
| --------- | ------------------------------------------------------------------------------------------------ |
| MUST      | 必須 env var が欠落したらエラーを投げる。ハードコード値にフォールバックしてはならない            |
| MUST      | セッション ID、トークン、ワンタイムコードには暗号学的に安全な乱数源を使う                        |
| MUST      | トークンと署名の比較には定数時間比較を使う (全バイトを XOR し、最後に判定する)。`===` を使わない |
| MUST      | webhook 署名を HMAC-SHA256 と定数時間比較で検証する                                              |

## Outbound Requests

| Directive | Rule                                                                                                                          |
| --------- | ----------------------------------------------------------------------------------------------------------------------------- |
| MUST      | webhook の送信先 URL をホスト許可リストで検証する。プライベート IP 範囲 (127.0.0.0/8, 10.0.0.0/8 など) を拒否して SSRF を防ぐ |

## Output

| Directive | Rule                                                                                                                                      |
| --------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| MUST      | CSP、`X-Frame-Options: DENY`、HSTS (`max-age=63072000`)、`X-Content-Type-Options: nosniff`、`Permissions-Policy` をミドルウェアで設定する |
| MUST NOT  | レスポンスにスタックトレース、内部パス、DB 詳細を含めない。汎用メッセージを返し、詳細はサーバー側にのみ記録する                           |
| MUST NOT  | 認証失敗の理由を区別しない (例: 「パスワードが違う」と「メールが見つからない」を区別しない)                                               |
| MUST NOT  | `Access-Control-Allow-Origin: *` を設定しない。許可するオリジンを明示的に指定する                                                         |
