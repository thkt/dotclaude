# 漏れのある抽象化の法則 - Joel Spolsky風

**コア原則**: 「すべての非自明な抽象化は、ある程度、漏れる」

## コア哲学

抽象化は単純化を目的とするが、必然的に基礎となる複雑さを露出する。**この現実を受け入れ、それに応じて設計する。**

目標は完璧な抽象化を作ることではなく:

- **漏れが発生する場所を認識**
- **発生したときの計画を立てる**
- **抽象化をできるだけシンプルに保つ**
- **いつ突破すべきかを知る**

## 完璧な抽象化の問題

```typescript
// 悪い例: 幻想: 「SQLを知る必要はない」
const users = await User.findAll({
  where: { active: true },
  include: ['posts', 'comments']
})

// 良い例: 現実: パフォーマンスがSQL知識を強制
const users = await db.raw(`
  SELECT u.*, COUNT(p.id) as posts
  FROM users u
  LEFT JOIN posts p ON u.id = p.user_id
  WHERE u.active = true
  GROUP BY u.id
`)
```

抽象化が漏れた。結局SQL知識が必要だった。

## 一般的な漏れのある抽象化

| 抽象化 | 漏れ方 | 解決策 |
| --- | --- | --- |
| **ORM** | N+1クエリ、SQL制御なし | 生クエリのエスケープハッチを提供 |
| **ネットワーク呼び出し** | ローカルを装うが予測不能に失敗 | タイムアウト、リトライ、エラーハンドリングを追加 |
| **クロスプラットフォーム** | プラットフォーム固有のパス/API | プラットフォーム対応ライブラリを使用 |

## 漏れを念頭に置いた設計

| 戦略 | 方法 | タイミング |
| --- | --- | --- |
| **プログレッシブ抽象化** | 80%にシンプルAPI、20%に生アクセス | `findUsers(criteria)` + `findUsersRaw(sql)` |
| **エスケープハッチ** | 基礎クライアントを公開 | `get rawClient()` で直接アクセス |
| **境界を文書化** | 制限付きJSDoc | 最大レスポンス: 5MB、タイムアウト: 30秒 |

## 抽象化を突破するタイミング

### パフォーマンス要件

```typescript
// 遅すぎる: 5秒
await orm.findAll({ include: ['author', 'tags'] })

// 突破: 100ms
await db.raw('SELECT * FROM posts...')
```

## 漏れのある抽象化の管理ガイドライン

### 抽象化をレイヤー化

```markdown
高レベル（ビジネスロジック）
    ↓
中レベル（サービスレイヤー）  ← ほとんどのコードはここ
    ↓
低レベル（直接アクセス）   ← エスケープハッチ
```

### 抽象化の健全性を監視

抽象化が漏れすぎているサイン:

- 常にエスケープハッチを使用
- 全員が実装詳細を知る必要がある
- 通常ケースよりエッジケースが多い

### スタックを知る

- **一段下を理解**: Reactを使うなら、DOMを理解
- **失敗モードを学ぶ**: ORMがどう失敗するか知る
- **エッジケースを研究**: フレームワークを壊すものは？

## 他の原則との統合

### プログレッシブエンハンスメント

- シンプルな抽象化から始める
- 漏れが現れたときにのみ複雑さを追加

### オッカムの剃刀

- シンプルで漏れる抽象化 > 複雑な「完璧な」抽象化

### YAGNI

- 想像上のニーズのために抽象化レイヤーを追加しない
- 漏れが実際に発生するまで待つ

## 実践的な適用

### コンポーネントライブラリ

```tsx
// 良い例: エスケープハッチを提供
function Button({
  children,
  onClick,
  className,      // スタイルのエスケープハッチ
  ...restProps    // DOM propsのエスケープハッチ
}) {
  return (
    <button
      className={cn('btn-default', className)}
      onClick={onClick}
      {...restProps}
    >
      {children}
    </button>
  )
}
```

## 覚えておく

> 「漏れのある抽象化の法則は、誰かが私たちを超効率的にするはずの新しいコード生成ツールを思いつくたびに、『まず手動でやり方を学べ、それからツールを使って時間を節約しろ』と言う多くの人がいることを意味する。」 - Joel Spolsky

**重要なポイント**:

- すべての抽象化は漏れる - それを計画に入れる
- 抽象化の一段下を知る
- エスケープハッチを提供
- 境界を文書化
- プログレッシブエンハンスメントは抽象化にも適用
