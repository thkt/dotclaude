# PRE_TASK_CHECK ルール

## 基本ルール

### 95%理解ルール

**信頼度<95%では進めない**。まず `AskUserQuestion` で確認を行う。

### チェック実行タイミング

- ファイル変更（作成/編集/削除）
- コマンド実行
- 複数ステップのワークフロー
- 理解度<95%

### スキップ条件

- 単純な質問、確認
- 読み取り専用クエリ、フォローアップ確認

### バグ調査コンテキスト

**発動条件**: ユーザーが問題や予期しない動作を報告している場合

文脈ベースの判断基準:

- 期待通りに動作しない
- エラーや障害が発生している
- リグレッション（以前は動いていた）
- 予期しない・混乱する動作

バグコンテキスト検出時、**必ず確認**:

1. **再現手順**: どうすれば再現できるか？（前提条件 → 手順 → エラー）
2. **正常ケース**: 正常に動作するケースは？（差分検出用）
3. **ログ情報**: エラーログやコンソール出力はあるか？

## コマンド選択

| 理解度 | アクション         | 理由                   |
| ------ | ------------------ | ---------------------- |
| <70%   | `/research`        | 計画するには不確実すぎ |
| 70-94% | `/think`           | 実装前に計画が必要     |
| ≥95%   | `/code` または直接 | 進めるのに十分明確     |

## 確認

すべての確認と明確化には `AskUserQuestion` ツールを使用。

## フロー

1. 分析 → 信頼度をマーク
2. <95%の場合 → AskUserQuestionで確認 → 停止
3. ≥95%の場合 → チェックを表示 → AskUserQuestionで確認
4. Dry-run / 計画 → 条件に応じて表示
5. 実行

### Dry-run 条件

| 条件                                         | 表示                       |
| -------------------------------------------- | -------------------------- |
| 3ファイル以上、コア設定、認証/セキュリティ   | 完全表示（Dry-run + 計画） |
| 単純、ドキュメントのみ、単一ファイル10行未満 | 計画のみ                   |
