# PRE_TASK_CHECK ルール

## マーカー定義

| マーカー | 意味     | 必要なアクション |
| -------- | -------- | ---------------- |
| [✓]      | 確認済み | なし - 続行可    |
| [→]      | 推測     | 確認してから続行 |
| [?]      | 不明     | 調査してから続行 |

ルール: 全項目が [✓] でなければ続行不可。[→] や [?] がある場合は先に解決する。

## チェック項目（7項目）

| #   | 項目         | チェック基準                                      |
| --- | ------------ | ------------------------------------------------- |
| 1   | 目的         | なぜ必要か + ユーザーの真の意図（文字通りでなく） |
| 2   | 範囲         | 対象ファイル/関数を特定している                   |
| 3   | 制約         | 技術的要件 + 制限事項 + 依存関係                  |
| 4   | 完了条件     | 完了基準 + 検証方法 + エッジケース                |
| 5   | コンテキスト | 実際のコードを読んだ（推測ではない）              |
| 6   | 影響範囲     | 影響を受ける領域 + 潜在的リスク                   |
| 7   | 前提条件     | 技術スタック/規約を確認している                   |

## タスク分割

いずれかの閾値を超えた場合は分割:

| 条件       | 閾値 |
| ---------- | ---- |
| ファイル数 | ≥5   |
| 機能数     | ≥3   |
| レイヤー数 | ≥3   |
| 行数       | ≥200 |

## フロー

1. 7項目をチェック → [✓]/[→]/[?] をマーク
2. [✓] 以外あり? → 先に解決（質問/読む）
3. 全て [✓]? → チェック表示 → ユーザーに確認
4. 閾値超え? → 分割
5. 実行

## スキップ条件

- 簡単な質問/確認
- 読み取り専用のクエリ
- フォローアップの明確化

## 「変更不要」検証ゲート

「変更不要」と報告する前に:

1. 証拠: 読んだファイル:行番号を明示
2. 説明: なぜ現状が目標を満たすか
3. 確認: AskUserQuestion でユーザーに確認
