# PREFLIGHT

実装前にスコープを確認する。質問 / 読取専用 / 同一ファイルのフォローアップ編集 / 同一セッションの計画継続では省略する。

## Rationalization Counters

| 言い訳                                 | カウンター                                                                   |
| -------------------------------------- | ---------------------------------------------------------------------------- |
| 「これは単なるフォローアップ」         | スコープ変更はフォローアップを装う。最低限スコープを確認する                 |
| 「コードベースは既に理解している」     | 理解は検証ではない。対象ファイルを最後に読んだのはいつか                     |
| 「ユーザーはプロセスより速度を求める」 | スコープ確認は 30 秒。誤スコープ実装は数時間                                 |
| 「これは計画でカバー済み」             | 計画は意図を述べる。スコープ確認は現状を検証する                             |
| 「根本修正はユーザーにとって高コスト」 | コスト判断はユーザーのもの。コスト付きで選択肢を提示し、ユーザーに判断させる |

## Task Decomposition

いずれかの閾値を超えたら分割する。

| 条件       | 閾値  |
| ---------- | ----- |
| ファイル数 | ≥5   |
| 機能数     | ≥3   |
| レイヤー数 | ≥3   |
| 行数       | ≥200 |

## Interpretation Clarity

| トリガー                           | アクション                                                 |
| ---------------------------------- | ---------------------------------------------------------- |
| 複数の妥当な解釈が存在             | すべて列挙する。黙って一つを選ばない。確認を待つ           |
| タスクの意図が不明                 | 停止する。何が不明かを明示する。続行しない                 |
| Issue URL または外部仕様の参照あり | 計画 (変更点、ファイル、TODO) を提示し、実装前に承認を待つ |
