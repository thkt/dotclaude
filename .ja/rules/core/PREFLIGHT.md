# PREFLIGHT

実装前にスコープを確認する。質問/読取専用/同一ファイルのフォローアップ編集/同一セッションの計画継続では省略する。

## アウトカム参照

タスク開始時、まず `.claude/OUTCOME.md` を読み、アウトカムに照らしてスコープを判断する。同一セッション内で既に読んでいる場合は再読を省略する。

| 条件                             | アクション                                               |
| -------------------------------- | -------------------------------------------------------- |
| OUTCOME.md が不在                | /outcome で stub を生成                                  |
| 非ゴール領域への変更を要求された | 非ゴール再定義か別タスクとして切り出すかをユーザーに確認 |
| Constraints に抵触する変更       | 制約変更の妥当性をユーザーに確認して停止                 |
| Outcome state に貢献しない変更   | YAGNI に該当しないか再確認                               |

## 解釈の明確化

| トリガー                           | アクション                                         |
| ---------------------------------- | -------------------------------------------------- |
| 複数の妥当な解釈が存在             | すべて列挙して確認を待つ                           |
| タスクの意図が不明                 | 何が不明かを明示して停止する                       |
| Issue URL または外部仕様の参照あり | 計画 (変更点、ファイル、TODO) を提示して承認を待つ |

## タスク分割

いずれかの閾値を超えたら分割する。

| 条件       | 閾値 |
| ---------- | ---- |
| ファイル数 | ≥5   |
| 機能数     | ≥3   |
| レイヤー数 | ≥3   |
| 行数       | ≥200 |

## 言い訳カウンター

| 言い訳                             | カウンター                                           |
| ---------------------------------- | ---------------------------------------------------- |
| 単なるフォローアップ               | 最低限のスコープを確認して無駄なスコープ変更を排除   |
| コードベースは既に理解している     | 対象ファイルを最後に読んだのはいつか確認し、検証する |
| ユーザーはプロセスより速度を求める | 正しいプロセスでの実装は結果的に速度を出せる         |
| 計画でカバー済み                   | スコープ確認は現状を検証するものであり計画とは異なる |
| 根本修正は高コスト                 | コスト判断はユーザーに委ねる                         |
| ゼロから実装する                   | 書く前にプロジェクト内で実装を調査する               |
