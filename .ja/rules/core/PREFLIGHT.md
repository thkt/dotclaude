# PREFLIGHT

実装前にスコープを確認する。質問 / 読み取り専用 / 同一ファイルへのフォローアップ編集 / 同一セッション内の計画続行はスキップ。

## 合理化への反論

| 言い訳                         | 反論                                                         |
| ------------------------------ | ------------------------------------------------------------ |
| 「単なるフォローアップ」       | スコープ変更がフォローアップに偽装する。最低限スコープを確認 |
| 「コードベースは理解している」 | 理解は検証ではない。対象ファイルを最後に読んだのはいつ？     |
| 「ユーザーはスピード優先」     | スコープ確認は30秒。スコープ違いの実装修正は数時間           |
| 「計画でカバーされている」     | 計画は意図を記述する。スコープ確認は現在の状態を検証する     |
| 「根本対応はユーザーに重すぎ」 | コスト判断はユーザーがする。選択肢とコストを提示して仰ぐ     |

## タスク分割

いずれかの閾値を超えた場合は分割する。

| 条件       | 閾値 |
| ---------- | ---- |
| ファイル数 | ≥5   |
| 機能数     | ≥3   |
| レイヤー数 | ≥3   |
| 行数       | ≥200 |

## 解釈の明確化

| トリガー                 | アクション                                    |
| ------------------------ | --------------------------------------------- |
| 有効な解釈が複数存在する | すべて列挙する。黙って1つ選ばない。確認を待つ |
| タスクの意図が不明確     | 中断する。何が不明かを明示する。先に進まない  |
