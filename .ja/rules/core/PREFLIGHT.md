# PREFLIGHT

実装前にスコープを確認する。質問 / 読取専用 / 同一ファイルのフォローアップ編集 / 同一セッションの計画継続では省略する。

## アウトカム参照

タスク開始時、まず `.claude/OUTCOME.md` を読み、アウトカムに照らしてスコープを判断する。同一セッション内で既に読んでいる場合は再読を省略する。スコープを確定する直前に「この変更の次に何が起きるか」を自問し、二次影響 (呼び出し元、運用、他タスク) を 1 行で言語化する。

| 条件                               | アクション                                                     |
| ---------------------------------- | -------------------------------------------------------------- |
| OUTCOME.md が不在                  | /outcome で stub を生成                                        |
| Non-goals 領域への変更を要求された | Non-goals の再定義か、別タスクとして切り出すかをユーザーに確認 |
| Constraints に抵触する変更         | 制約変更の妥当性をユーザーに確認して停止                       |
| Outcome state に貢献しない変更     | YAGNI に該当しないか再確認                                     |

## 解釈の明確化

| トリガー                           | アクション                                         |
| ---------------------------------- | -------------------------------------------------- |
| 複数の妥当な解釈が存在             | すべて列挙して確認を待つ                           |
| タスクの意図が不明                 | 何が不明かを明示して停止する                       |
| Issue URL または外部仕様の参照あり | 計画 (変更点、ファイル、TODO) を提示して承認を待つ |

### 質問の出し方

明確化の質問には自分の仮説 (予想する答え) を添える。ユーザーは一から説明せず仮説を訂正するだけで済み、こちらの理解度も同時に伝わる。次に出す質問の答えを予測できる状態になったら理解は十分とみなし、質問を止めて計画提示に進む。

## タスク分割

いずれかの閾値を超えたら分割する。タスク名が and / と で 2 つの動作を繋いでいる場合も、責務が 2 つ以上ある兆候として分割を検討する。

| 条件       | 閾値     |
| ---------- | -------- |
| ファイル数 | 5 以上   |
| 機能数     | 3 以上   |
| レイヤー数 | 3 以上   |
| 行数       | 200 以上 |

## 言い訳カウンター

| 言い訳                             | カウンター                                                  |
| ---------------------------------- | ----------------------------------------------------------- |
| 単なるフォローアップ               | 最低限のスコープを確認して無駄なスコープ変更を排除          |
| コードベースは既に理解している     | 対象ファイルを最後に読んだのはいつか確認し、検証する        |
| ユーザーはプロセスより速度を求める | 正しいプロセスでの実装は結果的に速度を出せる                |
| 計画でカバー済み                   | スコープ確認は現状を検証するものであり計画とは異なる        |
| 根本修正は高コスト                 | コスト判断はユーザーに委ねる                                |
| ゼロから実装する                   | 書く前にプロジェクト内で実装を調査する                      |
| この既存コードは不要そう           | なぜ存在するかを説明できるまで消さない (Chesterton's Fence) |
