# PRE_TASK_CHECK 詳細仕様

理解度チェックと実行計画のための技術仕様。基本ルールはPRE_TASK_CHECK_COMPACT.mdを参照。

## 実行ルール

**主要条件**: PRE_TASK_CHECKは以下のタスクで実行される:

- ファイル操作（作成/編集/削除）
- コマンド実行（bash、npmなど）
- 複数ステップのワークフロー
- 以下のいずれかの基準を満たすリクエスト:
  - **曖昧**: 明確な参照先のない代名詞（「それ」「あれ」）または曖昧な数量（「いくつか」「少し」）を含む
  - **複雑**: 3つ以上の異なるアクションが必要、または5つ以上のファイルに影響、または2つ以上のサブシステムが関与
- 理解度が95%未満の場合

**スキップ条件**（AI判断可）:

- 単純な事実質問（「Xとは何ですか？」）
- 確認レスポンス（「y」「yes」「ok」）
- 既存コンテンツに関する直接的な情報クエリ
- アクションを必要としないフォローアップ確認
- 説明やドキュメントのリクエスト

**常に必須**:

- ファイルシステムの変更
- コード実装リクエスト
- システムコマンド実行
- /commandsを必要とするタスク
- ユーザーが明示的に計画を要求した場合

## 分析方法

ユーザーリクエストを分析して以下を決定:

- 利用可能な情報に基づく理解度パーセンテージ
- 明確な要素と不明確な要素
- 以下の優先基準に基づくコマンド選択:
  1. **タスクスコープ**: 単一ファイル = /fix、複数ファイル = /code、調査 = /research
  2. **変更タイプ**: バグ修正 = /fix、新機能 = /think → /code
  3. **信頼度レベル**: <70% = まず/research、70-90% = /think、>90% = 直接/code
- 実行の実現可能性

**出力検証可能性要件**（AI動作原則 #4 による）:

理解度チェックの各要素について評価:

- **信頼度レベル**: 高（検証済み）、中（推論）、低（仮定）
- **証拠根拠**: 直接観察、論理的推論、または仮定
- **知識のギャップ**: 未知または不確実な点を明示的に認識

**信頼度レベルのマーカー**:

- ✓ = 高信頼度（ファイル/コンテキストから直接検証）
- → = 中信頼度（妥当な推論）
- ? = 低信頼度（確認が必要な仮定）

## 95%理解ルール

**エビデンスベースの実践からの基本原則:**

理解度が95%未満での実装は進めない。この閾値はエビデンスに基づく（下記の根拠を参照）。

### 適用タイミング

**以下の場合は常に確認質問を行う:**

- 要素全体の平均信頼度が95%未満
- いずれかの重要な要素が[?]（低信頼度）でマーク
- 複数の要素が[→]（中信頼度）でマーク
- ユーザー要件に曖昧さがある
- 実装アプローチが以下の不明確な条件のいずれかに該当:
  - 30秒の分析内で変更する具体的なファイルを特定できない
  - 複数の有効なアプローチが存在し、明確な選択基準がない
  - 必要な技術/フレームワークが明示的に記載されていない
  - 成功基準が測定可能な用語で定義できない

**ルール**: 加重平均信頼度 < 95%の場合、フォローアップ質問を表示し、進める前にユーザーの回答を待つ。

### フォローアップ質問形式

理解度が95%未満の場合、何を明確にする必要があるかを明示:

```markdown
🤔 95%信頼度に達するための質問:
1. [不明確な要素に関する具体的な質問]
2. [仮定した要素に関する具体的な質問]
3. [実装アプローチに関する具体的な質問]

進める前にこれらの明確化が必要です。
```

### 分析方法との統合

1. ユーザーリクエストを**分析** → 各要素の信頼度を決定
2. 加重平均信頼度を**計算**
3. **<95%の場合**: 確認質問を表示して待機
4. **≥95%の場合**: 理解度チェック表示に進む
5. **常に**: ✓/→/? マーカーで信頼度を明示

**根拠**: 95%未満の信頼度での実装は頻繁にやり直しが必要になる。事前に確認することで時間を節約し信頼を構築する。

**覚えておく**: 確認を求めることは、誤った仮定をするよりも常に良い。

**プログレスバーの構築**:

- 100%を表すのに合計10ブロックを使用
- 各ブロック（█ または ░）は10%を表す
- パーセンテージに基づいて左から右に埋める
- █ = 埋まっている（信頼度レベル達成）
- ░ = 空（信頼度レベル未達成）

**例**:

```text
100% → ██████████ 100%  (10個埋まっている)
 90% → █████████░  90%  (9個埋まり、1個空)
 70% → ███████░░░  70%  (7個埋まり、3個空)
 50% → █████░░░░░  50%  (5個埋まり、5個空)
 30% → ███░░░░░░░  30%  (3個埋まり、7個空)
  0% → ░░░░░░░░░░   0%  (10個空)
```

**実装パターン**:

```text
filled_count = Math.floor(percentage / 10)
empty_count = 10 - filled_count
bar = "█".repeat(filled_count) + "░".repeat(empty_count)
```

**完了定義セクション**: 実装/修正タスク（/code、/fix コマンド）でのみ表示。このセクションはTodoWriteとは別 - 完了定義は目標（何を達成するか）を表し、TodoWriteは実行ステップ（どう達成するか）を追跡する。

## 影響シミュレーション（Dry-run）

**表示タイミング**: Y確認後、実行計画の前、ファイル操作または複雑な変更の場合。

予想される変更とその影響の簡潔なシミュレーションを提供。「Dry-run」アプローチに基づく - 実際の実行なしに変更をシミュレートして効果をプレビュー。

**表示フォーマット**:

```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔍 影響シミュレーション（Dry-run）

予想される変更:
• 変更するファイル: [主要ファイル2-5個をリスト]
• 影響を受けるコンポーネント: [影響を受けるモジュール/コンポーネントをリスト]
• リスクレベル: 🟢 低 / 🟡 中 / 🔴 高
• 注意点: [重要な考慮事項]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**ガイドライン**:

- **長さ制限**:
  - 箇条書き: 3-5項目（最大5を強制）
  - 箇条書きごとの文字数: 最大100文字
  - 影響シミュレーション全体: 最大300文字
- **影響に焦点**: 何が変わるか、何が壊れる可能性があるか
- **リスク評価**: 潜在的な問題の正直な評価
- **実行可能**: 特別な注意が必要な領域を強調

**例**:

```markdown
🔍 影響シミュレーション（Dry-run）

予想される変更:
• 変更するファイル: src/auth/login.ts, src/components/LoginForm.tsx
• 影響を受けるコンポーネント: AuthService, UserSession, LoginPage
• リスクレベル: 🟡 中（認証フロー変更）
• 注意点: 既存のユーザーセッションは再ログインが必要な場合あり、新フローのテストを更新
```

**スキップ条件**（以下がすべてFALSEの場合のみ影響シミュレーションを適用）:

- **単純なファイル読み取り**: 変更なしのReadツール
- **ドキュメント更新**: *.mdファイルのみが影響、コード変更なし
- **明らかに低リスクな変更**: 単一ファイル、10行未満の変更、依存関係への影響なし

以下のいずれかが該当する場合、常に影響シミュレーションを表示:

- コア設定の変更（CLAUDE.md、PRE_TASK_CHECK.md、settings.json）
- 3つ以上のファイルを同時に変更
- 認証、セキュリティ、またはデータ整合性ロジックに影響

## エッジケース

- マッチなし（スコア<30）: 「コマンド: N/A」
- 複雑なタスク: ワークフローを提案
- 新しいコマンド: コマンドディレクトリにあれば自動探索

## 統合ワークフロー

1. **理解度チェック**: 理解度を評価＆アプローチを提案
2. **ユーザー確認**: `AskUserQuestion` を使用 ← **停止ポイント**
3. **影響シミュレーション**（確認済みかつ複雑/リスクのある変更の場合）: dry-runプレビューを表示
4. **実行計画**（確認済みかつファイル/コマンド操作の場合）: 具体的なアクションを表示
5. **計画確認**: `AskUserQuestion` を使用 ← **停止ポイント**
6. **実行**: 承認されたアクションを実行

## 出力検証可能性との統合

AI動作原則 #4 を ✓/→/? マーカーを使用して実装し、事実と仮定を区別し不確実性を認識する。
