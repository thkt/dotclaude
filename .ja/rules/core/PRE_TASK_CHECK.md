# PRE_TASK_CHECK

## マーカー

| マーカー | 意味     | アクション       |
| -------- | -------- | ---------------- |
| [✓]      | 確認済み | 続行             |
| [→]      | 推測     | 確認してから続行 |
| [?]      | 不明     | 調査してから続行 |

全項目が [✓] でなければ続行不可。

## チェックリスト（7項目）

| #   | 項目         | チェック基準                                      |
| --- | ------------ | ------------------------------------------------- |
| 1   | 目的         | なぜ必要か + ユーザーの真の意図（文字通りでなく） |
| 2   | 範囲         | 対象ファイル/関数を特定している                   |
| 3   | 制約         | 技術的要件 + 制限事項 + 依存関係                  |
| 4   | 完了条件     | 完了基準 + 検証方法 + エッジケース                |
| 5   | コンテキスト | `.analysis/architecture.md` 確認後、コードを読む  |
| 6   | 影響範囲     | 影響を受ける領域 + 潜在的リスク                   |
| 7   | 前提条件     | 技術スタック/規約を確認している                   |

## 表示フォーマット

```text
[✓] 目的: {なぜこの変更が必要か}
[✓] 範囲: {対象ファイル/関数}
[✓] 制約: {技術的制約}
[✓] 完了条件: {何をもって完了とするか}
[✓] コンテキスト: {既存コードを読んだ}
[✓] 影響範囲: {影響を受ける領域}
[✓] 前提条件: {技術スタック、規約}

ステータス: 準備完了 / 要確認 / ブロック中
```

## フロー

1. 7項目をチェック → [✓]/[→]/[?] をマーク
2. [✓] 以外あり? → 先に解決（質問/読む）
3. 全て [✓]? → チェック表示 → ユーザーに確認
4. 閾値超え? → 分割
5. 実行

## タスク分割

いずれかの閾値を超えた場合は分割:

| 条件       | 閾値 |
| ---------- | ---- |
| ファイル数 | ≥5   |
| 機能数     | ≥3   |
| レイヤー数 | ≥3   |
| 行数       | ≥200 |

## スキップ条件

- 簡単な質問/確認や読み取り専用のクエリ
- フォローアップの明確化や同一ファイルへの連続編集
- 承認済み計画の続行（同一セッション内）
- ユーザーが「とりあえずやって」「チェックスキップ」と明示

## 完了定義

| タイプ           | 基準                                                |
| ---------------- | --------------------------------------------------- |
| 機能追加         | 機能が動作、リグレッションなし、テスト追加済        |
| 修正             | 問題が再現しない、根本原因が解決済                  |
| リファクタリング | 動作に変更なし（テスト通過）、品質が向上（測定可能）|
| 調査             | 再現確認、根本原因特定、正常ケースの理解            |

既存状態で十分な可能性がある場合は「[条件] であれば変更不要」を追記。

## 「変更不要」ルール

「変更不要」と報告する前に: 具体的な file:line を引用、現状が目標を満たす理由を説明、AskUserQuestion で確認。
