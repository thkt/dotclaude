# PRE_TASK_CHECK

実装前にスコープを確認する。質問、読み取り専用、同一ファイルへのフォローアップ編集、同一セッション内の計画続行の場合はスキップ。

## 合理化への反論

| 言い訳                         | 反論                                                         |
| ------------------------------ | ------------------------------------------------------------ |
| 「単なるフォローアップ」       | スコープ変更がフォローアップに偽装する。最低限スコープを確認 |
| 「コードベースは理解している」 | 理解 ≠ 検証。対象ファイルを最後に読んだのはいつ？            |
| 「ユーザーはスピード優先」     | スコープ確認は30秒。スコープ違いの実装修正は数時間           |
| 「計画でカバーされている」     | 計画は意図を記述する。スコープ確認は現在の状態を検証する     |

## タスク分割

いずれかの閾値を超えた場合は分割:

| 条件       | 閾値 |
| ---------- | ---- |
| ファイル数 | ≥5   |
| 機能数     | ≥3   |
| レイヤー数 | ≥3   |
| 行数       | ≥200 |

## 完了定義

| タイプ           | 基準                                                 |
| ---------------- | ---------------------------------------------------- |
| 機能追加         | 機能が動作、リグレッションなし、テスト追加済         |
| 修正             | 問題が再現しない、根本原因が解決済                   |
| リファクタリング | 動作に変更なし（テスト通過）、品質が向上（測定可能） |
| 調査             | 再現確認、根本原因特定、正常ケースの理解             |

## 変更不要ルール

「変更不要」と報告する前に: 具体的な file:line を引用し現状が目標を満たす理由を説明、ユーザーに確認。
