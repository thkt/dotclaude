# PRE_TASK_CHECK

## マーカー定義

| マーカー | 意味     | 必要なアクション |
| -------- | -------- | ---------------- |
| [✓]      | 確認済み | なし - 続行可    |
| [→]      | 推測     | 確認してから続行 |
| [?]      | 不明     | 調査してから続行 |

ルール: 全項目が [✓] でなければ続行不可。[→] や [?] がある場合は先に解決する。

## チェック項目（7項目）

| #   | 項目         | チェック基準                                      |
| --- | ------------ | ------------------------------------------------- |
| 1   | 目的         | なぜ必要か + ユーザーの真の意図（文字通りでなく） |
| 2   | 範囲         | 対象ファイル/関数を特定している                   |
| 3   | 制約         | 技術的要件 + 制限事項 + 依存関係                  |
| 4   | 完了条件     | 完了基準 + 検証方法 + エッジケース                |
| 5   | コンテキスト | `.analysis/architecture.md` 確認後、コードを読む  |
| 6   | 影響範囲     | 影響を受ける領域 + 潜在的リスク                   |
| 7   | 前提条件     | 技術スタック/規約を確認している                   |

## フロー

1. 7項目をチェック → [✓]/[→]/[?] をマーク
2. [✓] 以外あり? → 先に解決（質問/読む）
3. 全て [✓]? → チェック表示 → ユーザーに確認
4. 閾値超え? → 分割
5. 実行

## タスク分割

いずれかの閾値を超えた場合は分割:

| 条件       | 閾値 |
| ---------- | ---- |
| ファイル数 | ≥5   |
| 機能数     | ≥3   |
| レイヤー数 | ≥3   |
| 行数       | ≥200 |

## スキップ条件

- 簡単な質問/確認や読み取り専用のクエリ
- フォローアップの明確化や同一ファイルへの連続編集
- 承認済み計画の続行（同一セッション内）
- ユーザーが「とりあえずやって」「チェックスキップ」と明示

## Done Definition テンプレート

注記: 既存状態で十分な場合は「- [ ] 現状で十分な場合は変更不要」を各テンプレートに追加。

### 機能追加

```markdown
- [ ] 指定された機能が動作する
- [ ] リグレッションなし | テスト追加
```

### 修正

```markdown
- [ ] 問題が再現しなくなる
- [ ] 根本原因が解決されている
```

### リファクタリング

```markdown
- [ ] 動作が変わっていない（テスト通過）
- [ ] 品質が向上している（測定可能）
```

### 調査（バグ/問題）

```markdown
- [ ] 再現手順を確認した
- [ ] 根本原因を特定した（症状だけでなく）
- [ ] 正常ケースを理解した（差分検出用）
```

## 「変更不要」検証

「変更不要」と報告する前に:

1. 証拠: 読んだファイル:行番号を明示
2. 説明: なぜ現状が目標を満たすか
3. 確認: AskUserQuestion でユーザーに確認

```markdown
📋 変更不要の分析:

- 読んだ: src/auth/login.ts:45-67
- 現状: エラーケースXは既に処理されている
- 結論: 既存実装が要件を満たす
  → クローズ前にユーザーに確認
```
