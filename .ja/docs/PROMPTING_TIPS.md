# Prompting Tips

Claude Code のための効果的なプロンプティング テクニック。

## やり直し・改善

| プロンプト                                                             | 効果                         | タイミング     |
| ---------------------------------------------------------------------- | ---------------------------- | -------------- |
| 「今わかっていることを踏まえて、これを破棄しエレガントな解で実装して」 | 蓄積された文脈でより良い実装 | 平凡な実装の後 |

## レビュー・検証

| プロンプト                                                                 | 効果                                  | タイミング |
| -------------------------------------------------------------------------- | ------------------------------------- | ---------- |
| 「この変更を厳しくレビューしてくれ。テストに合格するまで PR は作らないで」 | Claude を厳格なレビュアーに切り替える | PR 作成前  |

## コンテキスト・計算

| 戦略                                     | タイミング・理由                          |
| ---------------------------------------- | ----------------------------------------- |
| プロジェクトあたり MCP ツール ≤10       | 200k → 70k のコンテキスト縮小を防ぐ      |
| 必要なプラグインのみ有効化               | それぞれがトークン オーバーヘッドを増やす |
| 要求に「サブエージェントを使って」を追記 | 大きなタスクの並列処理                    |
| 並列作業前に `/fork`                     | 隔離コンテキスト、汚染を防ぐ              |

## デバッグ

| プロンプト                         | 効果                     | タイミング |
| ---------------------------------- | ------------------------ | ---------- |
| 「失敗している CI テストを直して」 | 細かく管理せずに走らせる | CI 失敗時  |

## ワークフロー

| テクニック                                                  | 効果                       |
| ----------------------------------------------------------- | -------------------------- |
| `/think` → 別の Claude セッションでレビュー                | 計画品質の向上             |
| 何かおかしいと思った瞬間に `/think` に切り替える            | 悪い経路で時間を浪費しない |
| 訂正のたびに「同じ間違いをしないよう CLAUDE.md を更新して」 | 繰り返しの間違いを防ぐ     |
