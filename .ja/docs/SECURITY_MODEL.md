# Security Model

## Permission Evaluation Flow は UX、セキュリティではない

Claude Code の権限評価フロー (PreToolUse Hook → Deny Rules → Allow Rules → Ask Rules → Permission Mode) は tool_use ブロックをフィルタする。セキュリティ境界ではない。

LLM はテキスト生成器であり、tool_use は出力フォーマットの 1 つ。Bash が許可されたら、OS のケーパビリティはそのまま通る。

参考: https://zenn.dev/commander/articles/72a907ce68a8c1

## 防御レイヤー

| レイヤー              | 実装               | 何を止めるか                 | Bash 経由でバイパス可能か |
| --------------------- | ------------------ | ---------------------------- | ------------------------- |
| L1: Deny Rules        | settings.json deny | 個別 tool_use ブロック       | 可                        |
| L2: PreToolUse Hook   | shields check      | Bash 内の危険パターン        | 部分的                    |
| L3: PermissionRequest | shields acl        | サブエージェント設定変更     | N/A                       |
| L4: macOS Seatbelt    | Claude Code 内蔵   | 部分的なファイルシステム制限 | OS が強制                 |
| L5: Process Sandbox   | 未実装             | ネットワークと FS 全体       | 不可                      |

重要: L1 から L3 は人間の介入ポイントの調整 (UX)。L4 から L5 が実セキュリティ境界。

## 既知のギャップ

| ギャップ                                           | リスク                                    | 緩和                                                     |
| -------------------------------------------------- | ----------------------------------------- | -------------------------------------------------------- |
| 許可された Bash コマンドからのネットワーク呼び出し | curl/wget 経由でデータ流出可能            | shields check でファイル アップロード パターンをブロック |
| 許可ツール経由の外部送信                           | scout, gh api 経由でデータ流出可能        | 正当な利用と区別困難。運用で対応                         |
| npm/pnpm install postinstall                       | 任意コード実行                            | 信頼できるパッケージのみ使用                             |
| L5 プロセス サンドボックス未実装                   | OS レベルでエージェント通信を制限できない | コンテナ化を検討 (将来)                                  |

## チーム ガイドライン

| ガイドライン                        | 説明                                                                               |
| ----------------------------------- | ---------------------------------------------------------------------------------- |
| deny は安全と等価でない             | deny ルールの追加だけでは完全なセキュリティ対応にならない                          |
| Bash 許可 = OS ケーパビリティの委譲 | Bash を許可すると deny ルールはバイパス可能                                        |
| hook は確率的防御                   | shields check はパターン マッチ。未知のパスは網羅できない                          |
| bypassPermissions は隔離環境のみ    | 本番環境や開発環境で使わない                                                       |
| シークレットは環境外に保つ          | .env や認証情報は deny + hook で二重に守られているが、プロセス レベルの制限はない  |
| サブエージェントの権限              | shields acl がセキュリティ ファイルへの変更を拒否する。Bash ルートは依然 L2 に依存 |

## コンテナを使う場合

L5 が必要なケース。

- CI/CD でエージェントを自動実行する
- 信頼できないリポジトリでエージェントを実行する
- エージェントの外部通信を禁止する必要がある
- コンプライアンス要件がプロセス分離を要求する

人間の監督下のローカル開発では L1 から L4 で十分。
