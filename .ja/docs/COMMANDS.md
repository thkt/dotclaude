## Dry-run影響シミュレーション

**自動安全機能** - PRE_TASK_CHECKワークフローに統合。

ファイル操作や複雑な変更を確認する際、Claude Codeは実行前に影響シミュレーションを自動表示します：

- **変更対象ファイル**: 主要ファイル2-5件を一覧表示
- **影響を受けるコンポーネント**: 影響するモジュールを表示
- **リスクレベル**: Low / Medium / High
- **重要な注意事項**: 注意が必要な領域をハイライト

この「Dry-run」アプローチは実行なしで変更をプレビューし、以下を支援します：

- 変更範囲の理解
- 潜在的リスクの特定
- 進行前の情報に基づく判断

**ワークフロー統合**:

```txt
1. 理解度チェック
2. ユーザー確認 (Y) ← 停止ポイント
3. 影響シミュレーション ← NEW（リスクのある変更で自動表示）
4. 実行計画
5. 計画確認 (Y) ← 停止ポイント
6. 実行
```

## 標準ワークフロー

### 機能開発

複雑さに応じて選択：

```txt
[複雑 - アーキテクチャ決定が必要]
(/research →) Plan Mode → /think → /code → /test → /audit → /validate

[標準 - 要件が明確]
/think → /code → /test → /audit → /validate

[シンプル - 小規模機能]
/code → /test
```

**Plan Mode**: `Shift+Tab`で入る。コードベースを探索し、アプローチを設計し、進行前にユーザーの承認を得る。

**注**: `/research`はPlan Mode前に、永続的なドキュメント付きの深い調査が必要な場合にオプションで使用。

### 進捗モニタリング

```txt
/sow（いつでも進捗確認）
```

### バグ調査と修正

```txt
/research → /fix
```

### 調査のみ（実装なし）

```txt
/research（結果は.claude/workspace/research/に保存）
```

### 自動化ワークフロー（SlashCommand Tool）

```txt
/auto-test        # 自動テスト → fixサイクル
/full-cycle       # 完全な自動開発フロー
```

## コマンド選択ガイド

### `/fix`を使用する場合

- 問題が小規模で明確に定義されている
- 開発環境で作業している
- 迅速な反復が必要

### Plan Modeを使用する場合

- アーキテクチャ決定が必要な複雑な機能
- 複数の有効なアプローチが存在
- 計画前にコードベースの探索が必要
- 実装前にアプローチのユーザー承認が欲しい

**入り方**: `Shift+Tab`を押すか、「enter plan mode」と入力

### `/research`を使用する場合

- 永続的なドキュメント付きの深い調査が必要
- 実装なしでソリューションオプションを探索
- 将来の参照用に調査結果を保存したい
- Plan Modeと組み合わせ可能: `/research` → Plan Mode

### `/think`を使用する場合

- 新機能の開始
- バリデーション付きの構造化された計画が必要
- 検証可能なSOWドキュメントを作成
- 自動進捗追跡が欲しい

### `/sow`を使用する場合

- 実装進捗の確認が必要
- 受け入れ基準のステータスを確認したい
- アクティブな開発作業のモニタリング

### `/validate`を使用する場合

- 実装の検証準備ができた
- SOWに対する適合性チェックが必要
- 不足している要件を特定したい

### `/rabbit`を使用する場合

- 外部AI視点が欲しい（内部エージェントから独立）
- 高速なCLIベースのレビューが必要（10-30秒）
- commit/PR前のクイックサニティチェックを探している
- `/audit`を第二の意見で補完

### `/adr`を使用する場合

- 重要なアーキテクチャ決定を行う
- 技術的選択をドキュメント化する必要がある
- 決定の理由を記録したい
- チームが決定の可視性を必要としている

### `/rulify`を使用する場合

- ADR決定がAIの動作に影響を与えるべき
- プロジェクト固有のパターンを強制したい
- AIがアーキテクチャ決定に自動的に従う必要がある

### `/workflow:create`を使用する場合

- 繰り返しのブラウザ操作を自動化する必要がある
- E2Eテストシナリオを作成
- 重要なユーザーフローのモニタリング設定
- データ収集ワークフローの構築
- 複雑な手動テスト手順をドキュメント化したい
- 再現可能なブラウザ自動化が必要

## コマンド、エージェント、スキルのアーキテクチャ

Claude Codeは3層構造を使用：

| レイヤー | 役割 | 例 |
| --- | --- | --- |
| **コマンド** | ユーザー呼び出しワークフロー（`/command`） | `/code`, `/audit`, `/adr` |
| **エージェント** | 専門的な分析/レビュー | `performance-reviewer`, `audit-orchestrator` |
| **スキル** | ナレッジベース、自動トリガーガイド | `applying-code-principles`, `generating-tdd-tests` |

## 設定

### 言語設定

- コマンドファイル: 英語
- ユーザーへの出力: 日本語（CLAUDE.mdに従う）

### 関連ファイル

- `~/.claude/CLAUDE.md` - グローバル設定とルール
- `~/.claude/rules/` - 開発原則
- `~/.claude/settings.json` - ツール権限
