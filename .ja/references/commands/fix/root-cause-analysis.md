# 根本原因分析（フェーズ1）

症状だけでなく、バグの真の原因を見つける。

## 目的

修正前にExploreエージェントと5 Whys方法論を適用して根本原因を特定。

## 動的コンテキスト

### 最近の変更

```bash
!`git diff HEAD~1 --stat | head -10`
```

バグを導入した可能性のある最近のコミットをレビュー。

### テストステータス

```bash
!`npm run || yarn run || pnpm run || echo "No package manager"`
```

現在のテストスイートのステータスを確認。

### Spec参照（オプション）

`/think`からspec.mdが存在する場合:

- `.claude/workspace/planning/**/spec.md`を確認
- FR-xxxとテストシナリオを参照として使用
- バグが既存の要件を壊すか確認

## フェーズ1: 根本原因分析

クイックコンテキスト用にExploreエージェントを使用（30秒）:

```typescript
Task({
  subagent_type: "Explore",
  thoroughness: "quick",
  description: "バグコンテキスト調査",
  prompt: `
    バグ: "${bugDescription}"
    検索: 関連ファイル、依存関係、最近のコミット
    5 Whysを適用: 症状ではなく根本原因を特定
    返却: [✓/→/?]マーカー付きの所見
  `
})
```

### キー質問（5 Whys）

1. **なぜこれが起きたか？**
   - 直接の原因は何か？

2. **なぜその条件が存在したか？**
   - 何がこれを発生させたか？

3. **なぜこれがキャッチされなかったか？**
   - プロセスの何がこれを見逃したか？

4. **なぜこのパターンがここにあるか？**
   - これは孤立したものか体系的か？

5. **なぜこの方法で行っているか？**
   - 設計自体に欠陥があるか？

### 分析出力

特定して分類:

- **症状**: ユーザーが見るもの
- **直接の原因**: 直接失敗したもの
- **根本原因**: なぜ失敗したか
- **スコープ**: 影響を受けるファイルとコンポーネント

### パターン認識

これが何かを判断:

- [Isolated] **孤立した問題**: 特定条件での一回限りのバグ
- [Pattern] **パターン問題**: 複数の場所で同じ問題
- [Critical] **体系的問題**: リファクタリングが必要な設計上の欠陥

### 信頼度マーカー

分析全体で使用: [@~/.claude/rules/development/COMPLETION_CRITERIA.md#confidence-metrics]

### 分析例

```markdown
根本原因分析

**症状**:
- [✓] 割引 > 価格の場合にユーザーに負の合計が表示される

**直接の原因**:
- [✓] calculateTotalがチェックなしでprice - discountを返す

**根本原因**:
- [✓] 割引が価格を超える場合のバリデーションがない
- [→] 要件がこのエッジケースを指定していなかった
- [?] 他の割引計算も同じ問題がある可能性

**スコープ**:
- [✓] ファイル: src/utils/pricing.ts:42
- [→] 類似パターン: src/checkout/discount.ts
- [?] 影響の可能性: src/reports/sales.ts

**パターン**: [Pattern] 潜在的パターン - 他の割引ロジックの確認が必要
**信頼度**: 0.85
```

## 分析をスキップする場合

- **些細な修正**: タイポ、明らかな欠落チェック
- **高信頼度 (>0.95)**: 原因が即座に明確
- **時間的プレッシャー**: 原因が分かっている本番ホットフィックス

これらの場合、フェーズ1.5（リグレッションテスト）に直接進む。

## エスカレートする場合

根本原因分析で以下が判明した場合:

- **信頼度 < 0.7**: より深い調査のために`/research`に切り替え
- **体系的問題**: 適切な設計のために`/think` → `/code`に切り替え
- **複数コンポーネント**: 複雑さがクイックフィックスの範囲を超える

## 統合ポイント

- **次**: フェーズ1.5（リグレッションテスト）
- **エスカレート先**: `/research`（低信頼度）または`/think`（体系的）
- **参照**: Spec.md（利用可能な場合）

## 出力フォーマット

```markdown
根本原因分析完了

バグ: [説明]
根本原因: [特定された原因] [信頼度: 0.XX]
影響: [ファイル]
パターン: [孤立/パターン/体系的]

次: リグレッションテストを書く
```
