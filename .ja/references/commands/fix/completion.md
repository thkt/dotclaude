# 完了とレポート

`/fix`コマンドの完了定義と出力フォーマット。

→ **共通基準**: [@../../../../rules/development/COMPLETION_CRITERIA.md](~/.claude/rules/development/COMPLETION_CRITERIA.md)

## /fix固有の基準

**信頼度目標**: 全体 ≥0.9

バグ修正の完了条件:

- [x] **根本原因を特定**（症状だけでなく、5 Whysを適用）
- [x] **最小複雑度の解決策**（オッカムの剃刀）
- [x] **品質ゲートに合格**（共通基準を参照）
- [x] **リグレッションが検出されない**

### オプション（推奨）

- [Recommended] 追加テストを生成（フェーズ3.5）
- [Recommended] 動作が変わった場合はドキュメントを更新

### エスカレーション基準

| 信頼度 | フェーズ | アクション |
| --- | --- | --- |
| <0.5 | 根本原因 | → 即座に`/research` |
| 0.5-0.7 | いずれか | → ユーザーに質問: 続行またはエスカレート？ |
| 0.7-0.8 | 実装 | 防御的アプローチで続行 |
| >0.8 | いずれか | 通常通り続行 |

### エスカレーションコマンド

```bash
# 調査のため
/research "Why does [bug] happen?"

# 計画のため
/think "Redesign [component] with proper [feature]"

# フル実装のため
/code "Implement [feature] with TDD"
```

## 適用された原則

修正を導いた原則をドキュメント化:

### 一般的な原則

- **オッカムの剃刀**: 最もシンプルな解決策を選択
  - 例: "複雑なバリデーションの代わりにMath.maxを使用"

- **TIDYINGS**: 関連するコードのみを触った
  - 例: "calculateTotalのみ修正、関連コードは変更なし"

- **プログレッシブエンハンスメント**: UIにはCSSファースト
  - 例: "JavaScript配置の代わりにGridレイアウトを使用"

- **DRY**: パターンの重複を特定
  - 例: "3つの類似した割引計算を共有関数にリファクタリング"

### 原則テンプレート

```markdown
## 適用された原則

- **オッカムの剃刀** ✅
  バリデーションフレームワークより`Math.max(0, result)`を選択

- **TIDYINGS** ✅
  影響を受けた関数のみ変更、周囲のコードはそのまま

- **プログレッシブエンハンスメント** ✅
  CSS Gridでレイアウト修正、JavaScript不要
```

## 次のステップ

修正完了後:

### 成功パス

1. **ドキュメント**: 修正を説明するコメントを追加
2. **コミット**: 意味のあるコミットメッセージを作成
3. **レビュー**: 変更をセルフレビュー
4. **PR**: プルリクエストを作成（必要な場合）

### フォローアップパス

1. **調査**: 類似パターンが見つかった場合
2. **リファクタリング**: 技術的負債が特定された場合
3. **ドキュメント**: 動作が変わった場合

### 追加アクション不要

バグ修正、テスト通過、完了！
