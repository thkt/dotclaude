# /branch - Gitブランチ名生成

現在のGit変更を分析し、規約パターンに従った適切なブランチ名を提案します。

**実装**: このコマンドは最適なパフォーマンスとコンテキスト効率のため、専門の `branch-generator` サブエージェントに委譲します。

## 動作概要

このコマンドを実行すると:

1. Taskツールで `branch-generator` サブエージェントを起動
2. サブエージェントがgit diffとstatusを分析（コードベースコンテキスト不要）
3. 規約に従ったブランチ名を生成
4. 複数の命名案を返却

## 使い方

### 基本的な使い方

```bash
/branch
```

現在の変更を分析し、ブランチ名を提案します。

### コンテキスト付き

```bash
/branch "OAuthによるユーザー認証を追加"
```

説明を提案に組み込みます。

### チケット付き

```bash
/branch "PROJ-456"
```

ブランチ名にチケット番号を含めます。

## ブランチ命名規約

### タイプ接頭辞

| 接頭辞 | 用途 |
| --- | --- |
| `feature/` | 新機能 |
| `fix/` | バグ修正（緊急含む） |
| `refactor/` | コード改善 |
| `docs/` | ドキュメント |
| `test/` | テスト追加/修正 |
| `chore/` | メンテナンス作業 |
| `perf/` | パフォーマンス改善 |
| `style/` | フォーマット/スタイル |

### フォーマット

```text
<type>/<scope>-<description>
<type>/<ticket>-<description>
<type>/<description>
```

### 良い例

```bash
✅ feature/auth-add-oauth-support
✅ fix/api-resolve-timeout-issue
✅ docs/readme-update-install-steps
✅ feature/PROJ-123-user-search
```

### 悪い例

```bash
❌ new-feature (タイプ接頭辞なし)
❌ feature/ADD_USER (大文字)
❌ fix/bug (曖昧すぎる)
❌ update_code (セパレータが違う)
```

## 出力形式

コマンドは以下を提供します:

- **現在の状態**: 現在のブランチ、変更されたファイル
- **分析**: 変更タイプ、主要スコープ、主な変更
- **推奨名**: 分析に基づいた最適な名前
- **代替案**: スコープ付き、説明的、簡潔版
- **使用方法**: ブランチの作成または名前変更の方法

## ワークフローとの統合

以下とシームレスに連携:

- `/commit` - まずブランチを作成、その後コミット
- `/pr` - ブランチ作成後にPR説明を生成
- `/think` - ブランチ作成前の計画

## 技術詳細

### サブエージェントのメリット

- **90%のコンテキスト削減**: git操作のみ、コードベース読み込みなし
- **2-3倍の高速実行**: git分析に最適化された軽量エージェント
- **専門ロジック**: ブランチ名生成に特化
- **並列実行**: 他の操作と同時実行可能

### 使用するGit操作

サブエージェントはgitコマンドのみを実行:

- `git branch --show-current` - 現在のブランチを確認
- `git status` - ファイル状態を確認
- `git diff` - 変更を分析
- ファイルシステムアクセスやコード解析なし

## 関連コマンド

- `/commit` - コミットメッセージを生成
- `/pr` - PR説明を作成
- `/research` - ブランチ作成前の調査

## ベストプラクティス

1. **早めにブランチ作成**: 変更を加える前に
2. **明確な命名**: 変更について具体的に
3. **規約に従う**: プロジェクトパターンを維持
4. **チケットを含める**: 該当する場合はIssueにリンク
5. **簡潔に**: 50文字以下

## コンテキスト効率

このコマンドは最小限のコンテキスト使用に最適化:

- ✅ コードベースファイルの読み込みなし
- ✅ gitメタデータのみを分析
- ✅ 高速実行（5秒未満）
- ✅ 他のタスクと並列実行可能
