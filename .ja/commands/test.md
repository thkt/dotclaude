# /test - テスト実行＆品質検証

## 目的

プロジェクトテストを実行し、包括的なテストと検証でコード品質を確保。

## 初期探索

### パッケージマネージャの確認

```bash
!`ls package*.json 2>/dev/null | head -1`
```

## テスト実行プロセス

### 1. テスト実行

TodoWriteでテスト進捗を追跡。

まず、package.jsonで利用可能なスクリプトを確認:

```bash
cat package.json
```

その後、適切なテストコマンドを実行:

```bash
npm test
```

代替パッケージマネージャ:

```bash
yarn test
```

```bash
pnpm test
```

```bash
bun test
```

### 2. カバレッジ分析

カバレッジ付きでテスト実行:

```bash
npm test -- --coverage
```

カバレッジディレクトリを確認:

```bash
ls coverage/
```

### 3. テストギャップ分析

**目的**: カバレッジデータに基づいて不足テストを自動特定し改善を提案。

test-generatorエージェントでカバレッジギャップを分析:

```typescript
Task({
  subagent_type: "test-generator",
  description: "テストカバレッジギャップを分析",
  prompt: `
テスト結果: ${testResults}
カバレッジ: ${coverageData}

ギャップを分析:
1. カバーされていないコード: ファイル<80%、未テスト関数、ブランチ [✓]
2. 不足シナリオ: エッジケース、エラーパス、境界 [→]
3. 品質問題: 浅いテスト、アサーション不足 [→]
4. 優先領域のテストコードを生成 [→]

返す: テストコードスニペット（説明ではなく）、カバレッジ改善推定。
マーク: [✓] 検証済みギャップ、[→] 提案テスト。
  `
})
```

#### ギャップ分析出力

```markdown
## テストカバレッジギャップ

### 高優先度（< 50%カバレッジ）
- **ファイル**: src/utils/validation.ts
  - **行**: 45-67（23行未カバー）
  - **問題**: エラーケースのテストなし
  - **提案テスト**:
    ```typescript
    describe('validation', () => {
      it('should handle invalid input', () => {
        expect(() => validate(null)).toThrow('Invalid input')
      })
    })
    ```

### 中優先度（50-80%カバレッジ）
- **ファイル**: src/services/api.ts
  - **行**: 120-135
  - **問題**: ネットワークエラー処理が未テスト

### カバーされていないエッジケース
1. 境界条件（空配列、null値）
2. 並行操作
3. タイムアウトシナリオ

### 推定インパクト
- 提案テスト追加: 65% → 85%カバレッジ
- 工数: 約2時間
- クリティカルパスカバー: 95%+
```

## 結果分析

### テスト結果サマリー

以下の明確なサマリーを提供:

- 実行されたテスト総数
- 合格/失敗の内訳
- 実行時間
- カバレッジ率（測定された場合）

### 失敗分析

失敗したテストについて:

1. テストファイルと行番号を特定
2. 失敗理由を分析
3. 具体的な修正を提案
4. 関連コードにリンク

### カバレッジレポート

カバレッジが利用可能な場合:

- 行カバレッジ率
- ブランチカバレッジ率
- 未カバーのクリティカルパス
- 改善の提案

## TodoWrite統合

自動タスク追跡:

```markdown
1. テストインフラを探索
2. テストスイートを実行
3. 失敗を分析（あれば）
4. カバレッジレポートを生成
5. テストギャップを分析（NEW - test-generator）
6. 品質チェックを実行
7. 結果をサマリー
```

**test-generatorで強化**: ステップ5に自動ギャップ分析とテスト提案を含む。

## ベストプラクティス

1. **即座に修正**: テスト負債を蓄積しない
2. **カバレッジを監視**: 時間経過での傾向を追跡
3. **失敗を優先**: 新しいテストを追加する前に壊れたテストを修正
4. **問題を記録**: 将来の参照用に失敗パターンを保持

## 次のステップ

結果に基づいて:

- **テスト失敗** → `/fix` で特定の失敗に対処
- **低カバレッジ** → 未カバーのクリティカルパスにテスト追加
- **すべてグリーン** → コミット/PRの準備完了
- **品質問題** → まずリント/型エラーを修正
