# バグレポートテンプレート

## 構造

```markdown
## 概要

[What: 何が壊れているか — 1文で] [Why: なぜ問題か — ユーザー影響]

## 再現手順

1. [手順 1]
2. [手順 2]
3. [手順 3]

## 期待 vs 実際

- **期待**: [こうなるべき — 具体的な値や振る舞い]
- **実際**: [こうなっている — エラーメッセージや実際の値]

## スコープ

- **スコープ内**: [この Issue で修正する範囲]
- **スコープ外**: [関連するが今回は対応しないこと]

## 制約（任意）

- [修正方針の制約: 破壊的変更禁止、パフォーマンス要件など]
- [禁止事項: ワークアラウンドではなく根本原因を修正、など]

## 環境（任意）

- ブラウザ/OS: [例: Chrome 120 / macOS 14]
- バージョン: [例: v1.2.3]

## 備考（任意）

[エラーログ、スクリーンショット、関連 Issue など]
```

## ガイドライン

| フィールド | 説明                                                                |
| ---------- | ------------------------------------------------------------------- |
| 概要       | What + Why を分けて書く。影響度が伝わらないと優先度を誤る           |
| 再現手順   | 最小手順。AI はこれを元にテストケースを書く                         |
| 期待/実際  | 具体的な値で対比。「正しく動く」は NG、「200 を返す」は OK          |
| スコープ   | スコープ外で修正範囲を限定。バグ修正がリファクタに膨らむのを防ぐ    |
| 制約       | 「ワークアラウンド禁止」「後方互換維持」など。AI の安易な修正を防止 |
| 環境       | 環境依存バグの場合のみ記載                                          |
