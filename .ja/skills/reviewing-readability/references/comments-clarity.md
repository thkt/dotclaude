# コメントとコードの明確さ

## コア原則

**Why, not What** - コードは自己説明、コメントは意図を説明

| コメント種別 | 例                                            | 判定 |
| ------------ | --------------------------------------------- | ---- |
| What         | `// カウンターを1増加`                        | Bad  |
| What         | `// ユーザーが管理者かチェック`               | Bad  |
| Why          | `// サーバー過負荷防止のための指数バックオフ` | Good |
| Why          | `// 法的要件: コンプライアンスで7年保持`      | Good |

## コメントが必要な場合

| コンテキスト       | 例                                          |
| ------------------ | ------------------------------------------- |
| 複雑なアルゴリズム | `// Boyer-Moore: O(n/m) vs 単純 O(n*m)`     |
| ビジネスルール     | `// GDPR: EUユーザーは明示的オプトイン必須` |
| ワークアラウンド   | `// TODO(2025 Q2): API v2デプロイ時に削除`  |
| パフォーマンス     | `// レンダリング毎の高コストDBクエリ回避`   |

## アンチパターン

| パターン             | 問題                         | 解決策           |
| -------------------- | ---------------------------- | ---------------- |
| コメントアウトコード | デッドコードの混乱           | 削除 (gitを使用) |
| 自明なコメント       | `// nameを'John'に設定`      | 削除             |
| ジャーナルコメント   | `// 2024-01-05: 変更 - John` | git履歴を使用    |
| 古いコメント         | Xと記載、コードはY           | 更新または削除   |

## 自己文書化コード

```typescript
// Bad: コメントが必要
// ユーザーがプレミアム機能にアクセスできるかチェック
if (u.sub && u.sub.exp > Date.now() && !u.ban) {
}

// Good: 自己説明
if (canAccessPremiumFeatures(user)) {
}
```

## 判断テスト

| 質問                                    | Noの場合           |
| --------------------------------------- | ------------------ |
| これを削除してコードを自己文書化できる? | コードをリファクタ |
| WhyではなくWhatを説明している?          | 削除または書き直し |
| このコメントはまだ正確?                 | 更新または削除     |
