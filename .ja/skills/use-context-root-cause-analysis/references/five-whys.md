# 5 Whys - 実例

良い問題提起は具体的かつ観察可能であること。「ダッシュボードのロードに 5 秒かかる」は分析可能、「アプリが遅い」は不可。

## 例。ダッシュボードのロードに 5 秒かかる

| ステップ | 質問              | 回答                           |
| -------- | ----------------- | ------------------------------ |
| 1        | なぜ 5 秒?        | API `/api/dashboard` が 4.5 秒 |
| 2        | なぜ API が遅い?  | 15 テーブルにクエリしている    |
| 3        | なぜ 15 テーブル? | 全モジュールを 1 ビューで表示  |
| 4        | なぜ一度に?       | mount 時に全てレンダリング     |
| 5        | なぜ mount で?    | lazy loading なし              |

解決策。セクションごとの lazy loading、必要時取得。

## 例。フォームが 2 重送信される

| ステップ | 質問                  | 回答                                    |
| -------- | --------------------- | --------------------------------------- |
| 1        | なぜ 2 回?            | リクエスト完了前に 2 回クリックされた   |
| 2        | なぜクリック可能?     | 送信中にボタンが disabled でない        |
| 3        | なぜ disabled でない? | `isSubmitting` がハンドラ開始後にセット |
| 4        | なぜ後に?             | React が state 更新をバッチ化           |
| 5        | なぜバッチ化?         | 命令的な用途で state を使っている       |

解決策。即時フラグには `useRef`、または form action から `disabled={pending}`。
