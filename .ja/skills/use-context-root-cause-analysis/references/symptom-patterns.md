# 症状パターン

## Symptom → Root Cause のマッピング

| 症状                                 | 根本原因                           | 修正                                             |
| ------------------------------------ | ---------------------------------- | ------------------------------------------------ |
| DOM のための `setTimeout`            | React ライフサイクルを使っていない | `useRef` + `useEffect` を使う                    |
| 二重実行を防ぐためのフラグ           | StrictMode、または cleanup なし    | cleanup 付きの `useEffect`                       |
| state を同期する複数の effect        | derived state を独立に扱っている   | derived データに `useMemo`                       |
| prop を state に同期する `useEffect` | 不要な内部 state                   | prop を直接使う                                  |
| 子の state にアクセスするための ref  | 不適切なデータフロー               | state を持ち上げ、props で渡す                   |
| 兄弟間のための event bus             | state が持ち上げられていない       | 共通の親に持ち上げる                             |
| 全部 memo 化                         | 早すぎる最適化                     | まず profile、必要なところだけ memo              |
| 全箇所で throttle/debounce           | 間違ったアプローチ                 | CSS (`position: sticky`)、`IntersectionObserver` |
| show/hide のための JavaScript        | 過剰設計                           | `<details>` または CSS                           |

## 判断フレームワーク

| 質問                        | No の場合            |
| --------------------------- | -------------------- |
| 完全に防げるか?             | より良い設計を検討   |
| 単純な技術で解けるか?       | まず HTML/CSS で試す |
| 原因を扱うか結果を扱うか?   | 5 Whys を適用        |
| この修正はスケールするか?   | アプローチを再考     |
| フレームワーク流のやり方か? | パターンを学ぶ       |

## 根本原因のカテゴリ

| カテゴリ       | 例                               |
| -------------- | -------------------------------- |
| 設計           | 誤った抽象、欠けたパターン       |
| アーキテクチャ | 結合、欠けた境界                 |
| プロセス       | 検証の欠如、リトライロジックなし |
| 知識           | フレームワーク/ライブラリの誤解  |
| 要件           | 不明確な要件 → 誤った解決策     |
