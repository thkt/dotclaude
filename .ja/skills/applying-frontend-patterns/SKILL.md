# フロントエンドパターン

保守可能なフロントエンドアーキテクチャのためのコンポーネント設計パターン。

## コアパターン

| パターン                 | コンセプト                       | 使用タイミング               |
| ------------------------ | -------------------------------- | ---------------------------- |
| Container/Presentational | ロジックとUIの分離               | データ取得 + 表示            |
| カスタムフック           | 再利用可能なステートフルロジック | 共有する振る舞い             |
| コンポジション           | シンプルから複雑を構築           | 柔軟なコンポーネント         |
| 状態管理                 | アプリデータの整理               | ローカル → 共有 → グローバル |

## Container/Presentational

| Container（ロジック） | Presentational（UI）       |
| --------------------- | -------------------------- |
| データを取得          | propsでデータを受け取る    |
| 状態を管理            | ステートレス（理想）       |
| イベントを処理        | コールバックpropsを呼ぶ    |
| スタイリングなし      | すべてのスタイリングがここ |

## Hooksガイドライン（React）

| Hook        | 用途             | 落とし穴       |
| ----------- | ---------------- | -------------- |
| useEffect   | 副作用           | 依存関係の欠落 |
| useMemo     | 高コストな計算   | 早すぎる最適化 |
| useCallback | 安定した関数参照 | 過度なメモ化   |

## 状態管理戦略

| スコープ   | ツール        | 例                     |
| ---------- | ------------- | ---------------------- |
| ローカル   | useState      | フォーム入力、トグル   |
| 共有       | Context       | テーマ、認証状態       |
| グローバル | Zustand/Redux | アプリ全体のキャッシュ |

## パターンを使わないとき

- シンプルな一回限りのコンポーネント
- プロトタイプ（YAGNI）
- 再利用が期待されない

**ルール**: 予測ではなく、痛みを感じたときにパターンを追加。
