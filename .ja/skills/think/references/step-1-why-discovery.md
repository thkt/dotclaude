# Step 0: Why Discovery

コードを探索したりアプローチを生成したりする前に、この作業が達成すべき成果を確立する。Why は明示的に求めない限り浮かび上がらない。

## 質問

| 質問                        | 目的                              |
| --------------------------- | --------------------------------- |
| 誰がこれを必要としているか? | ステークホルダー / ユーザーの特定 |
| どんな痛みが存在するか?     | 問題の根拠 (仮定ではなく)         |
| 成功とは何か?               | 計測可能な結果 (成果物ではなく)   |
| なぜ今か?                   | 優先度の正当化                    |
| やらない場合どうなるか?     | 不作為のコスト                    |

## 出力: Why Statement

```markdown
## Why

- For: [誰のためか]
- Problem: [根拠ある痛み]
- Outcome: [成功の姿、機能ではなく結果]
- Urgency: [なぜ今、あとではないのか]
- Inaction cost: [これを飛ばすと何が起きるか]
```

## ゲート

Why Statement が明確になるまで Step 1 へ進まない。曖昧または仮定の項目があるときは、プレースホルダではなく対話で詰める。ゴールはユーザーが既に分かっているがまだ言語化していないものを引き出すこと。

| ルール                     | 詳細                                                     |
| -------------------------- | -------------------------------------------------------- |
| 1 メッセージにつき 1 質問  | 推奨回答とその根拠を添える                               |
| コードベース優先           | コードベースで解決できるなら問う前に探索する             |
| 仮説を表に出す             | あなたの Why の読み方を述べ、確認または修正を求める      |
| 対比的なフレーミングを提示 | 別の解釈を提示してユーザーが意図を言語化する手助けをする |
| 曖昧な outcome に挑む      | 「UX を改善」 → 誰のため? どう測るか?                    |
