# 文章レビュー

生成した SOW / Spec の本文を精査し、形式や構造に出る冗長・重複を検出して直す。構造の妥当性は reviewer-spec が gate するためここでは扱わない。語彙レベルの空句もここでは扱わず、起動フローの別ステップで点検する。

| パターン       | シグナル                                                                     | 修正                                                                |
| -------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------------- |
| doc の書き写し | リンク済み設計 doc や調査 / 検証 / CI ログの内容を本文に貼る・再説明する     | リンク + 要点 1 行 (数値は 1 行)。差分だけ書く                      |
| 理由の繰り返し | 同じ設計理由を 1 文書内で 2 回説明する                                       | 決定が着地する箇所で 1 回だけ述べる                                 |
| 案の列挙       | 棄却した案 A / 案 B / 案 C や比較表で本文を埋める                            | 採用案だけ残し、比較は ADR / 設計スレッドへ                         |
| 冗長な表現形式 | 同じ構造や情報を図 / 表 / list の複数形式で重複表現する                      | 最も読みやすい 1 形式に絞り、残りを削る                             |
| 太字の多用     | 太字が数行おきに散在する                                                     | 構造は見出しに任せ、太字は警告のみ                                  |
| 過剰指定の AC  | 受け入れ条件に実装の書き方 (関数名、ファイル構成、ライブラリ設定) を列挙する | 観測可能な criterion は残し、実装の詳細は Spec / 実装フェーズへ送る |
| 惰性セクション | 書くことがないのに optional セクション (Boundaries, Risks 等) を埋める       | 空の optional セクションは省略する                                  |
