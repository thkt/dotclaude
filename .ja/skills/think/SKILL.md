# /think

深い設計探索。アプローチ比較、前提検証、SOWとSpec生成。

## Rationalization Counters

| 言い訳                                     | 反論                                                                             |
| ------------------------------------------ | -------------------------------------------------------------------------------- |
| "Whyは自明、省略していい"                  | 誰にとって自明？未検証のWhyは下流で暗黙の判断を生む                              |
| "DAチャレンジはやりすぎ"                   | すべての設計に隠れた前提がある。DAエージェントはセルフレビューの見落としを捉える |
| "ユーザーの要求がそのままコードに対応する" | 要求は症状を記述する。設計前に根本の問題を特定すること                           |
| "この機能はシンプルだからSpec不要"         | シンプルに見えるタスクほど曖昧さが潜む。形式化することで実装前に表面化できる     |

## Why Discovery（Step 0）

### Step 0: Why Discovery

コードを探索する前、アプローチを生成する前に、この作業が達成すべきアウトカムを確立する。Whyは明示的に求めない限り出てこない。

| 問い                     | 目的                             |
| ------------------------ | -------------------------------- |
| 誰のためか？             | ステークホルダー/ユーザーの特定  |
| 今ある痛みは何か？       | 問題の証拠（仮定ではなく）       |
| 成功とは何か？           | 計測可能な結果（成果物ではなく） |
| なぜ今か？               | 優先度の根拠                     |
| やらなかったらどうなる？ | 不作為のコスト                   |

出力: Why Statement

```markdown
## Why

- For: [誰のため]
- Problem: [証拠に基づく痛み]
- Outcome: [成功の姿 — 機能ではなく結果]
- Urgency: [なぜ今か]
- Inaction cost: [やらなかった場合に起こること]
```

ゲート: Why Statementが明確になるまでStep 1に進まない。

いずれかのフィールドが曖昧、または仮定に基づく場合、仮置きせず壁打ちで固める。

| ルール | 詳細 |
| ------ | ---- |
| 1問ずつ | 推奨回答を根拠付きで添える |
| コードベース優先 | コードベースで解決できる問いは調べてから聞かない |
| 仮説を提示 | 自分のWhy解釈を示し、確認・訂正を求める |
| 対立するフレーミングを提示 | 複数の解釈を並べてユーザーが言語化しやすくする |
| 曖昧なアウトカムを突く | 「UX改善」→ 誰にとって？どう測る？ |
| ゲート | Why Statementが確認されるまで繰り返す |

ユーザーが知っているが言語化できていないものを引き出すのが目的。

## 設計探索（Steps 2-5）

### Step 2: コードベース探索

関連コードを読む。`.claude/workspace/research/`
に最近の調査結果があれば読み込み、先行調査のコンテキストを継承。パターン、制約、アーキテクチャ、先行事例を理解。

### Step 3: アプローチ生成

≥2つの異なるアプローチを異なる視点から生成:

- Pragmatist: 動くもっともシンプルな解決策は？
- Architect: 拡張可能で構造的に優れた方法は？
- DX Advocate: 開発者/ユーザー体験にとって最善は？

アプローチ内に独立した技術決定（例: フレームワーク、状態管理、API設計）がある場合、
各決定を個別の選択肢として提示すること — 1メッセージに1問、推奨と
プロジェクトへの影響を付記。密結合な決定のみバンドル可。

PREFLIGHTの分解閾値（ファイル数 ≥5、機能 ≥3、レイヤー ≥3）を超える場合、独立したUnitに分解すること。各Unitは独自のSOW/Specを持ち
`/code` で個別に実装可能。

### Step 4: 設計チャレンジ

`devils-advocate-design`エージェントを（バックグラウンドで）起動し、Step 3のアプローチを検証する。エージェントはRead/Grep/Globで自身のコンテキストを収集。

DA結果を判定テーブル + アクション項目で提示。Step 5に進む前に指摘に基づきアプローチを修正。

### Step 5: 設計構成

生き残ったアプローチから最適設計を構成。2つの視点を順に検討:

#### 5-1. ドメイン視点（What）

技術非依存のビジネスロジックモデリング。深さはコンテキストに応じて変動:

| コンテキスト                                       | 深さ | フォーカス                                                 |
| -------------------------------------------------- | ---- | ---------------------------------------------------------- |
| ビジネスアプリ（エンティティ ≥3 or 業務ルール ≥3） | 詳細 | エンティティ、関係、不変条件、業務ルール、ドメインイベント |
| CLIツール / 設定 / シンプルUI                      | 簡潔 | 主要データ構造とバリデーションルールのみ                   |

```markdown
### ドメイン視点

- エンティティ: [主要データ型とその関係]
- 業務ルール: [ドメイン固有のルールと制約]
- 不変条件: [常に成立すべき条件]
```

#### 5-2. 技術視点（How）

ドメイン理解を実装設計に変換:

```markdown
### 技術視点

- コンポーネントアーキテクチャ: [階層、境界、責務]
- 状態戦略: [サーバー状態 vs クライアント状態、管理アプローチ]
- NFR適用: [パフォーマンス、セキュリティ、アクセシビリティパターン]
- 運用面の考慮: [エラーバウンダリ、ロギング、ローディング状態]
```

#### 統合出力

```markdown
## 設計

### 主要な決定

| 決定 | 選択 | 根拠              |
| ---- | ---- | ----------------- |
| ...  | ...  | [視点] にトレース |

### 実装スケッチ

- 変更するファイル: [file:line のリスト]
- 作成するファイル: [目的付きリスト]
- 推定スコープ: [行数, ファイル数]

### トレードオフ

| 採用したこと | 見送ったこと | 理由   |
| ------------ | ------------ | ------ |
| [やること]   | [諦めたこと] | [根拠] |
```

## ドキュメント生成（Steps 7-8）

### Step 7: SOW

テンプレート `~/.claude/skills/think/templates/sow.md` をRead。設計コンテキスト（Steps
0-6）から生成。ID形式: AC-N。出力:
`.claude/workspace/planning/YYYY-MM-DD-[feature]/sow.md`

品質ゲート（各セクション記述前に適用）:

| セクション | ゲート                                                                                                                                      |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Why        | 5フィールド全記入。Outcome = 成果物ではなく計測可能な結果                                                                                   |
| AC         | 各ACがWhy Outcomeにトレース。孤立AC禁止。Why Problemを超えたスコープ不可                                                                    |
| Scope      | YAGNIチェックリスト項目に理由を付けてチェック（単なる除外は不可）。Out of Scope: 各除外にWhy not（Whyフィールド or 制約へのトレース）を記述 |
| Impl       | フェーズごとにFiles < 5。Stepは具体的な変更内容を記述                                                                                       |
| Test       | 全ACに≥1テスト。Verificationは何を検証するか具体的に記述                                                                                    |
| Risks      | ≥1リスク特定 + 緩和策                                                                                                                       |

### Step 8: Spec

テンプレート `~/.claude/skills/think/templates/spec.md` をRead。SOWから生成。ID形式: FR-001,
T-001, NFR-001。トレーサビリティ: `FR-001 Implements: AC-001` →
`T-001 Validates: FR-001` UI関連: Component
API（Props、variants、states、usage）を含める。出力:
`.claude/workspace/planning/[same-dir]/spec.md`

品質ゲート（各セクション記述前に適用）:

| セクション | ゲート                                                               |
| ---------- | -------------------------------------------------------------------- |
| FR         | EARS構文必須。1文につきSHALL 1つ。曖昧な値禁止（appropriate等）      |
| FR         | 設計判断がある場合は補足で根拠を記載（variant流用、YAGNI論拠等）     |
| Domain     | 概念レベルのみ。型名・フィールド名は書かない。InvariantsはFRに紐付け |
| Test       | 全FRに≥1シナリオ。Given-When-Thenの全列に具体値                      |
| NFR        | Measurement列に計測方法を明記（コードレビュー、手動計測等）          |
| Trace      | AC → FR → Test → NFR チェーン途切れなし                              |

## Specレビュー（Step 9）

Spec生成後、sow-spec-reviewerを実行。

スコア ≥90 → パス。スコア <90 → 指摘を修正して再実行（最大3ループ）。3ループ後も残る指摘はユーザーに提示して続行。

## ADR提案（Step 6.5）

ユーザーが設計を承認した後、技術的決定（フレームワーク/ライブラリ選定、アーキテクチャパターン、非推奨化、トレードオフ選択）についてADRが必要か確認。単純な機能追加ならスキップ。

## タスク分解（Step 10）

### 原則

| 原則                  | ルール                                 |
| --------------------- | -------------------------------------- |
| サブ締め切り必須      | フェーズ単位のマイルストーンと完了条件 |
| 並行グルーピング      | 並行可能なら1フェーズ1タスクにしない   |
| 初手を明示            | 最初に着手すべきタスクと理由を示す     |
| スコープカット = 末端 | コア依存タスクは削減不可               |
| 緊急性で煽らない      | 構造的に分析、反射的に対応しない       |

### TaskCreate

クロスセッション: `export CLAUDE_CODE_TASK_LIST_ID="[feature]-tasks"`
（最大10タスク）

| ソース              | subject           | description                 | addBlockedBy |
| ------------------- | ----------------- | --------------------------- | ------------ |
| Implementation Plan | `Phase N: [説明]` | ステップ + validates AC-XXX | [依存ID]     |
| Test Plan (HIGH)    | `Test: [説明]`    | （複雑な場合）              | [依存ID]     |

### スコープ検証

タスク作成前に、Implementation
Planの各Phaseのユニークファイル数をカウントする。ファイル数 ≥5のPhaseは独立Unitに分割し、全Phaseがファイル数 <
5になるまで繰り返す。

### マイルストーン

```text
Phase 1 [Day X]: タスク一覧（完了条件）
Phase 2 [Day Y]: ...
```

### 初手

→ タスクN:[理由 — なぜこれがもっとも多くの後続タスクをアンブロックするか]

### スコープカット候補

末端タスクのみ列挙。コア依存は削減不可。

## Q&Aカテゴリ（Step 1）

スコープ、優先度（MoSCoW）、制約、リスク。

目的、ユーザー、成功基準はStep 0（Why Discovery）でカバー。

## 出力

このパスを正確に使用すること — Writeツールが親ディレクトリを自動作成する。

`.claude/workspace/planning/YYYY-MM-DD-[feature]/sow.md` と `spec.md`

## 検証

全ステップ完了必須: Why確立、コードベース探索、≥2アプローチ比較、DAチャレンジ適用、設計構成、ユーザーレビュー、sow.mdとspec.md生成、Specレビュー通過、タスク分解（マイルストーン + 初手 + スコープカット候補）。
