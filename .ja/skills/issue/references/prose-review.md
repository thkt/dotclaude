# 文章レビュー

文脈を共有しリンク先を開けるチームメイトに向けて書く。Issue が運ぶのは差分、背景はリンクが運ぶ。消しても読み手が判断を誤らない行は書かない。語彙レベルの空句はここでは扱わず、phrases ファイルで別途点検する。

## 構造

| チェック   | 質問                                                                         |
| ---------- | ---------------------------------------------------------------------------- |
| 問題の提示 | 問題や要望が冒頭の 1〜3 行で述べられているか                                 |
| 具体性     | bug は再現手順、feature はユースケースが具体的で、期待結果が推測任せでないか |
| 差分集中   | コードを読めば分かる what を避け、why と完了条件に絞っているか               |
| 分量       | 本文が約 40 行に収まっているか。超えるなら削るか別 issue / リンクへ逃がす    |

## 冗長パターン

| パターン       | 修正                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------ |
| doc の書き写し | リンク済み doc やログの再説明はリンク + 要点 1 行に畳む。差分だけ書く                      |
| 決定の重複     | 同じ設計理由は決定が着地する箇所で 1 回だけ述べる。案の比較は本文に入れない (think の役割) |
| 過剰指定の AC  | criterion は残し、書き方の詳細 (story 名や設定値の列挙) を削る                             |
| 1 文 1 主張    | 60 字を超える 1 文や、決定 / 根拠 / 参照を詰め込んだ段落は主張ごとに文を切る               |
