# 文章レビュー

文脈を共有しリンク先を開けるチームメイトに向けて書く。ゼロ文脈の読者向けではない。Issue が運ぶのは差分、背景はリンクが運ぶ。消しても読み手が判断を誤らない行は書かない。

## 構造

Issue 本文が満たすべき構造を確認する。

| チェック         | 質問                                                                                                       |
| ---------------- | ---------------------------------------------------------------------------------------------------------- |
| 問題の提示       | 問題や要望が冒頭の 1〜3 行で述べられているか                                                               |
| 再現 / 具体性    | Bug は再現手順が具体的か。Feature はユースケースが具体的か                                                 |
| 期待結果         | 期待される振る舞いが明示されており、読者の推測に任されていないか                                           |
| 読者のアクション | 求めるアクションが具体的か ("spec をレビュー", "原因を調査", "X までに判断")                               |
| スコープ         | 1 つの問題に集中しており、関連する懸念をまとめて投げ込んでいないか                                         |
| 分量             | 本文が約 40 行に収まっているか。超えるなら削るか別 Issue / リンクへ逃がす                                  |
| 差分集中         | コードを読めば分かる what を避け、why と完了条件に絞っているか                                             |
| Outcome 整合     | Outcome state に貢献するか。Non-goals に踏み込むなら body で明示的にフラグする                             |
| 表の活用         | 同じ属性の組が 3 件以上並ぶ列挙を bullet でなく表で書いているか。Plan 節は抽出契約があるため bullet のまま |

## 冗長・重複パターン

形式や構造に出る冗長・重複を検出して直す。言語に依存しない。語彙レベルの空句はここでは扱わず、起動フローの別ステップで点検する。

| パターン       | シグナル                                                                                                         | 修正                                                                                                |
| -------------- | ---------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| doc の書き写し | リンク済み設計 doc や調査 / 検証 / CI ログの内容を Issue 内に貼る・再説明する                                    | リンク + 要点 1 行 (数値は 1 行)。差分だけ書く                                                      |
| 理由の繰り返し | 同じ設計理由を 1 Issue 内で 2 回説明する                                                                         | 決定が着地する箇所で 1 回だけ述べる                                                                 |
| 案の列挙       | 案 A / 案 B / 案 C や比較表で本文を埋める                                                                        | 決まったこと (or 決めたい 1 点) だけ残し、比較は設計スレッドへ                                      |
| 冗長な表現形式 | 同じ構造や情報を図 / 表 / list の複数形式で重複表現する (依存関係を図と順序表と Tracking list で 3 重に書くなど) | 最も読みやすい 1 形式に絞り、残りを削る                                                             |
| 太字の多用     | 太字が数行おきに散在する                                                                                         | 構造は見出しに任せ、太字は警告のみ                                                                  |
| 過剰指定の AC  | 受け入れ条件に書き方の詳細 (story 名, addon 設定) を列挙する                                                     | criterion は残し書き方の詳細を削る。UI コンポーネント issue では Storybook / a11y を DoD として残す |
| 惰性セクション | 書くことがないのに optional セクションを埋める                                                                   | 空の optional セクションは省略する                                                                  |
| 高密度段落     | 1 つの段落や bullet に決定 / 根拠 / 参照を 3 文以上詰め込み、括弧書きの補足が入れ子になる                        | 1 文 1 主張に分割する。列挙は bullet / 表へ、括弧書きの根拠は独立した文に昇格する                   |
| 長い 1 文      | 読点や接続助詞で主張を連結し、1 文が 60 字を超える                                                               | 主張ごとに文を切る。条件や理由は独立した文にする                                                    |
