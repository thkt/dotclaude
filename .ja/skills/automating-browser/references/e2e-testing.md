# ブラウザ自動化によるE2Eテスト

claude-in-chrome MCPを使用したエンドツーエンドテストパターン。

## E2Eテストアプローチ

### テスト構造

```markdown
1. セットアップ - 開始点に移動
2. 実行 - ユーザーアクションを実行
3. 検証 - 期待される結果を確認
4. クリーンアップ - 必要に応じて状態をリセット
```

### テスト例

```markdown
テスト: ユーザーがログインしてダッシュボードを表示できる

セットアップ:
  - "/login"に移動
  - wait duration: 2

実行:
  - ユーザー名をform_input
  - パスワードをform_input
  - ログインボタンをクリック
  - wait duration: 3

検証:
  - read_page
  - ダッシュボード要素を確認
  - 証拠としてスクリーンショット
```

## 一般的なテストシナリオ

### 認証フロー

```markdown
1. ログインページに移動
2. 認証情報を入力（モック/テストアカウント）
3. フォームを送信
4. ダッシュボードへのリダイレクトを検証
5. ユーザー情報の表示を検証
6. ログアウトをテスト
7. ログインへのリダイレクトを検証
```

### フォーム送信フロー

```markdown
1. フォームページに移動
2. すべての必須フィールドを入力
3. フォームを送信
4. 成功メッセージを検証
5. データの永続化を検証（該当する場合）
```

### ナビゲーションフロー

```markdown
1. ホームページから開始
2. ナビゲーションリンクをクリック
3. 正しいページが読み込まれたことを検証
4. パンくずリスト/URLを確認
5. 戻るナビゲーションをテスト
```

### 検索フロー

```markdown
1. 検索ページに移動
2. 検索クエリを入力
3. 検索を送信
4. 結果が表示されることを検証
5. 結果をクリック
6. 詳細ページを検証
```

## 検証テクニック

### 視覚的検証

```markdown
1. computer action: "screenshot"
2. 期待されるレイアウトと比較
3. 欠落した要素を確認
```

### コンテンツ検証

```markdown
1. get_page_text
2. 期待される文字列を確認
3. エラーメッセージの不在を検証
```

### 構造検証

```markdown
1. read_page filter: "interactive"
2. 期待される要素数をカウント
3. 要素階層を検証
```

### コンソール検証

```markdown
1. read_console_messages onlyErrors: true
2. JavaScriptエラーがないことを検証
3. 期待されるログを確認
```

### ネットワーク検証

```markdown
1. read_network_requests urlPattern: "/api"
2. API呼び出しが成功したことを検証
3. レスポンスステータスコードを確認
```

## テストデータ管理

### テストアカウント

- 専用のテストアカウントを使用
- 本番の認証情報は絶対に使用しない
- テストアカウントへのアクセスを文書化

### テストデータ

- 予測可能なテストデータを使用
- 必要に応じてテスト間でリセット
- データ依存関係を文書化

## エラーシナリオ

### テスト: 無効なログイン

```markdown
1. ログインに移動
2. 無効な認証情報を入力
3. 送信
4. エラーメッセージが表示されることを検証
5. まだログインページにいることを検証
```

### テスト: フォームバリデーション

```markdown
1. フォームに移動
2. 空のフォームを送信
3. バリデーションエラーを検証
4. 必須フィールドを入力
5. エラーがクリアされたことを検証
```

### テスト: 404ページ

```markdown
1. 存在しないURLに移動
2. 404ページが表示されることを検証
3. "ホームへ戻る"リンクをテスト
```

## テスト証拠の記録

### スクリーンショット文書化

```markdown
各テストに対して:
1. スクリーンショット: 初期状態
2. スクリーンショット: 重要なアクション後
3. スクリーンショット: 最終状態
4. スクリーンショット: エラーがあれば
```

### GIF録画

```markdown
複雑なフローに対して:
1. GIF録画を開始
2. スクリーンショット付きでテストステップを実行
3. 録画を停止
4. 説明的なファイル名でエクスポート
```

## テスト整理

### 機能別

```text
/tests
  /auth
    login.md
    logout.md
    password-reset.md
  /forms
    contact.md
    checkout.md
  /navigation
    header.md
    footer.md
```

### 優先度別

```text
/tests
  /critical     # リリースに必須
  /important    # 合格すべき
  /nice-to-have # オプション
```

## ベストプラクティス

### テストの独立性

- 各テストはクリーンな状態から開始
- テスト間に依存関係なし
- 任意の順序で実行可能

### 信頼性

- 安定したセレクタを使用（座標ではなくref ID）
- 適切な待機を追加
- 不安定な操作にリトライ

### メンテナンス性

- テストの目的を文書化
- 説明的な名前を使用
- テストを集中させる（1テスト1事項）

### パフォーマンス

- 待機を最小化
- 可能な場合はセットアップを再利用
- 独立したテストを並列化

## /testコマンドとの統合

`/test`コマンドはブラウザテストを実行可能:

```text
/test --e2e           # E2Eテストを実行
/test --browser       # ブラウザテストのみ
/test src/e2e/*.test  # 特定のテストファイル
```
