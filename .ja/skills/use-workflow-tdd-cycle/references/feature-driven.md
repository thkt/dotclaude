# 機能駆動TDD

`/code` が新機能開発でTDDを使用する方法。`${CLAUDE_SKILL_DIR}/SKILL.md`（RGRCサイクル、Baby Steps）を前提とする。

## コンテキスト

| 観点       | 値                                 |
| ---------- | ---------------------------------- |
| ソース     | 機能要件を含むspec.md              |
| アプローチ | すべてのテストをスキップ状態で生成 |
| 順序       | シンプル → 複雑（Baby Steps）      |
| 制御       | ユーザーが1テストずつ有効化        |

## ワークフロー

| Step | アクション                                       |
| ---- | ------------------------------------------------ |
| 1    | spec.md読込 → FR-xxx抽出 → Given-When-Then特定   |
| 2    | スキップテスト生成（`it.skip()`）シンプル→複雑順 |
| 3    | 対話的有効化: [Y]es/[S]kip/[Q]uit → RGRCサイクル |
| 4    | すべてのテストが有効かつパスするまで繰り返し     |

テストごとの実行は `SKILL.md` の正式な RGRC サイクルに従う。機能駆動の差分は、Red で次に選んだテストの `.skip` を外すところから始める点。

## フレームワークスキップマーカー

| フレームワーク | スキップ構文                                        |
| -------------- | --------------------------------------------------- |
| Jest/Vitest    | `it.skip('test', () => { // TODO: [SKIP] FR-001 })` |
| Mocha          | `it.skip()` または `xit()`                          |
| 不明           | `// TODO: [SKIP]` マーカー                          |

## 主要特性

| 特性           | 説明                   |
| -------------- | ---------------------- |
| プロアクティブ | 実装前にテスト         |
| 仕様駆動       | 要件 → テスト → コード |
| ユーザー制御   | 明示的な有効化         |
| Baby Steps     | 一度に1テスト          |

## 判断ポイント

| 質問                   | 回答                                 |
| ---------------------- | ------------------------------------ |
| いつテストをスキップ？ | 複雑すぎる、依存関係が未準備         |
| いつコミット？         | 全テストグリーン、コードリファクタ済 |
| コミット頻度           | テスト有効化ごと（約5-10分）         |

## よくある落とし穴

| 落とし穴                   | 修正                      |
| -------------------------- | ------------------------- |
| 複数テストを同時有効化     | 一度に1テストのみ         |
| テスト前に実装             | 最初にテスト有効化（Red） |
| Refactorフェーズをスキップ | コミット前に必ず整理      |
