# バグ駆動TDD

`/fix` がバグ修正でTDDを使用する方法。`${CLAUDE_SKILL_DIR}/SKILL.md`（RGRCサイクル、Baby Steps）を前提とする。

## コンテキスト

| 観点       | 値                                       |
| ---------- | ---------------------------------------- |
| ソース     | バグ説明と再現手順                       |
| アプローチ | 失敗テスト作成 → 修正 → エッジケース追加 |
| テスト状態 | 有効（スキップなし）                     |
| フォーカス | リグレッション防止                       |

## ワークフロー

| Step | アクション                                          |
| ---- | --------------------------------------------------- |
| 1    | バグ再現 → 失敗テスト作成 → 失敗確認                |
| 2    | バグ修正 → 最小実装 → テストパス                    |
| 3    | リグレッション防止 → エッジケーステスト追加（任意） |
| 4    | 全テストパス確認                                    |

実行は `SKILL.md` の正式な RGRC サイクルに従う。バグ駆動の差分は、Red でバグの正確なシナリオを再現する点と、エッジケース拡張は Green 後に任意で行う点。

## 主要特性

| 特性               | 説明                 |
| ------------------ | -------------------- |
| リアクティブ       | バグ → テスト → 修正 |
| バグ駆動           | 再現が最初           |
| リグレッション重視 | 再発防止             |
| 高速イテレーション | 最小サイクル時間     |

## 判断ポイント

| 質問                             | 基準                                       |
| -------------------------------- | ------------------------------------------ |
| リグレッションテストをスキップ？ | ドキュメントのみ、CSS、設定、軽微な修正    |
| 必ずテストを書く場合？           | ロジック、セキュリティ、データ、バグ修正   |
| エッジケーステストを生成？       | 複雑なバグ、重要ロジック、類似バグの可能性 |
| 最小限とは？                     | 必要なものだけ、リファクタリングなし       |

## よくある落とし穴

| 落とし穴               | 修正                                   |
| ---------------------- | -------------------------------------- |
| 修正後にテスト         | 失敗テストを最初に書く                 |
| バグを再現しないテスト | 正確な失敗シナリオをテスト             |
| 過剰設計の修正         | 最もシンプルな解決策（オッカムの剃刀） |
| テスト失敗を確認しない | 必ずGreen前にRedを確認                 |
