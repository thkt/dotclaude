# Bug-Driven TDD

`/fix` がバグ修正で TDD をどう使うか。${CLAUDE_SKILL_DIR}/SKILL.md の RGRC サイクルと Baby Steps を基盤とする。

## 文脈

| 観点       | 値                                             |
| ---------- | ---------------------------------------------- |
| ソース     | バグの説明と再現手順                           |
| アプローチ | 失敗するテストを書く → 修正 → エッジケース追加 |
| テスト状態 | active。skip しない                            |
| 焦点       | リグレッション防止                             |

## ワークフロー

| ステップ | アクション                                     |
| -------- | ---------------------------------------------- |
| 1        | バグを再現 → 失敗するテストを書く → 失敗を確認 |
| 2        | バグ修正 → 最小実装 → テストがパスする         |
| 3        | 再発防止 → エッジケーステストを追加 (任意)     |
| 4        | 全テストがパスすることを確認                   |

実行は `SKILL.md` の正典 RGRC サイクルに従う。Bug-driven は Red でバグの正確なシナリオを再現する点と、エッジケース展開が任意かつ Green の後である点が異なる。

## 主な特徴

| 特徴               | 説明                 |
| ------------------ | -------------------- |
| Reactive           | バグ → テスト → 修正 |
| Bug-driven         | まず再現             |
| リグレッション中心 | 再発防止             |
| 高速反復           | 最小サイクル時間     |

## 判断ポイント

| 質問                            | 基準                                               |
| ------------------------------- | -------------------------------------------------- |
| リグレッションテストをスキップ? | ドキュメントのみ、CSS、config、些細な修正          |
| 必ずテストを書く対象?           | ロジック、セキュリティ、データ、バグ修正           |
| エッジケーステストを生成?       | 複雑なバグ、クリティカルロジック、類似バグの可能性 |
| どこまで minimal が minimal か? | 必要なものだけ、リファクタなし                     |

## よくある落とし穴

| 落とし穴                       | 修正                         |
| ------------------------------ | ---------------------------- |
| 修正後にテストを書く           | まず失敗するテストを書く     |
| テストがバグを再現しない       | 正確な失敗シナリオをテスト   |
| 修正の over-engineering        | 最も単純な解 (Occam's Razor) |
| テストが落ちることを確認しない | Green の前に必ず Red を確認  |
