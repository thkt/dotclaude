# TDDサイクル

public API を通じて振る舞いをテストする。モックはシステム境界のみ。

## バリアント選択

| トリガー                                | バリアント       | 参照                                               |
| --------------------------------------- | ---------------- | -------------------------------------------------- |
| spec.md / 新機能（`/code`）              | 機能駆動         | `${CLAUDE_SKILL_DIR}/references/feature-driven.md` |
| バグ報告 / リグレッション（`/fix`）      | バグ駆動         | `${CLAUDE_SKILL_DIR}/references/bug-driven.md`     |
| 既存コードのカバレッジギャップ          | カバレッジ駆動   | アクティブテスト、スキップなし。RGRCを流用         |

## 何をテストするか

| 優先度     | 内容                                                       |
| ---------- | ---------------------------------------------------------- |
| 必須       | ビジネスロジック、サービス、クリティカルパス、エッジケース |
| 状況次第   | 複雑なユーティリティ、カスタムフック、変換                 |
| スキップ   | シンプルなアクセサ、UIレイアウト、外部ライブラリ動作       |

### TDDを使わない場面

| 文脈                       | 理由                                 |
| -------------------------- | ------------------------------------ |
| プロトタイプ（使い捨て）   | 破棄される可能性が高く、コスト過大   |
| 外部API統合                | 統合ではなくAPIをモックする          |
| シンプルな単発スクリプト   | テストより短く書ける                 |
| UI実験                     | ビジュアルファースト、後からロジック抽出 |

## 機能駆動 vs バグ駆動

| 観点       | 機能駆動           | バグ駆動               |
| ---------- | ------------------ | ---------------------- |
| トリガー   | 仕様               | バグ報告               |
| テスト状態 | 最初はスキップ状態 | 有効                   |
| テスト数   | 全テスト事前生成   | 1メイン + エッジケース |
| 有効化     | ユーザー制御       | 即時                   |
| フォーカス | 機能完成           | リグレッション防止     |

## テスト哲学（Classical/Detroit）

| 原則               | ルール                                                      |
| ------------------ | ----------------------------------------------------------- |
| 実装より振る舞い   | public API の出力をテスト、内部呼び出しはテストしない       |
| 状態検証           | 結果値でアサート、「X が呼ばれたか」ではなく                |
| 実オブジェクト優先 | 実際の依存関係を使用。外部 I/O のみモック                   |
| ブラックボックス   | public インターフェース経由でユニットをブラックボックス扱い |
| 社交的テスト       | 協調者を参加させる。境界でのみ隔離                          |

## RGRCサイクル

| フェーズ | 目標           | ルール                                                                                 | よくある間違い     |
| -------- | -------------- | -------------------------------------------------------------------------------------- | ------------------ |
| Red      | 失敗テスト     | 失敗が意図した振る舞いのギャップと一致することを確認（構文・インポートエラーではなく） | テストが即座に通過 |
| Green    | テストをパス   | "罪を犯してよい" - dirty OK                                                            | 過剰実装           |
| Refactor | クリーンコード | テストをグリーンに保つ                                                                 | 動作を変更         |
| Commit   | 状態を保存     | 全チェックをパス                                                                       | チェックをスキップ |

## Baby Steps（2分サイクル）

30秒: 失敗テスト作成 → 1分: パス → 10秒: テスト実行 → 30秒: 小さなリファクタ → 20秒: グリーンならコミット。バグは常に直近2分の変更にある。

## テスト失敗時の判断基準

テスト失敗時、テストと実装のどちらを修正すべきか判断する。

| 判定       | 条件                      | アクション                   |
| ---------- | ------------------------- | ---------------------------- |
| 実装バグ   | テストがspec/FR-xxxに合致 | 実装を修正。テストは触らない |
| テストバグ | テストがspecと乖離        | テストを修正                 |
| 判断不能   | specが曖昧または不在      | ユーザーにエスカレーション   |

バグ駆動フロー（`/fix`）の場合は、再現手順がspecの役割を果たす。

## テスト設計

| 技法     | 用途                     | 例                     |
| -------- | ------------------------ | ---------------------- |
| 同値分割 | 同じ振る舞いをグループ化 | 年齢: <18, 18-120      |
| 境界値   | 境界をテスト             | 17, 18, 120, 121       |
| 決定表   | 複数条件ロジック         | isLoggedIn × isPremium |

## アサーション品質

すべてのテストは具体的なアウトカムを検証しなければならない。弱いアサーション単独は禁止。

| カテゴリ             | マッチャー                                                              | 許容される条件                                     |
| -------------------- | ----------------------------------------------------------------------- | -------------------------------------------------- |
| 弱い（存在）         | toBeTruthy, toBeDefined, toBeFalsy, toBeNull, toBeUndefined             | 同じテスト内に意味のあるアサーションがある場合のみ |
| 意味あり（値）       | toBe, toEqual, toStrictEqual, toMatch, toContain, toThrow, toHaveLength | 常に優先                                           |
| 意味あり（呼び出し） | toHaveBeenCalledWith, toHaveBeenCalledTimes, toHaveReturnedWith         | 副作用を検証する場合                               |

Bad: `expect(result).toBeTruthy()`
Good: `expect(result).toEqual({ id: 1, name: "Alice" })`

1テスト1概念。同じ関数・同じ引数パターンでアサートする2テストは統合するか `test.each` でパラメータ化。

## モック

システム境界でモック。外部 API、データベース、ファイルシステム、ネットワーク、非決定的な依存（時間、ランダム）、2分サイクルをブロックする遅い依存。

| ルール                 | 閾値                                   |
| ---------------------- | -------------------------------------- |
| テストあたりのモック数 | アサーション数を超えてはならない       |
| モックスコープ         | 外部依存関係のみ                       |
| モック対象             | テスト対象モジュール自体をモックしない |

| アンチパターン                     | 問題                                          | 代わりに                                           |
| ---------------------------------- | --------------------------------------------- | -------------------------------------------------- |
| モックの呼び出しをアサート         | モックの動作をテスト、コンポーネントではない  | 観察可能な出力や副作用でアサート                   |
| テスト専用のプロダクションメソッド | テストアクセスのためにプロダクションAPIを汚染 | テストユーティリティに抽出または public API を使用 |
| 理解前にモック                     | 実際の依存関係の動作を隠す                    | まず依存関係を理解してからモック                   |
| 部分的なモック構造                 | 欠落フィールドが偽のパスを生む                | 完全な実際のAPI構造をミラー                        |
| モック過剰                         | アサーションより多いモックは配線のテスト      | モック削減、または意味のあるアサーション追加       |

### UT分離

UTがインポートするのは: 対象モジュール + 型 + テストインフラのみ。テストデータは型/リテラルから構築。

## テスト構成

### AAAパターン

```typescript
test("name", () => {
  // Arrange - セットアップ
  // Act - 実行
  // Assert - 検証
});
```

### 命名

| レベル | パターン                                |
| ------ | --------------------------------------- |
| Suite  | `describe("[対象]", ...)`               |
| Group  | `describe("[メソッド]", ...)`           |
| Test   | `it("when [条件], should [期待]", ...)` |
