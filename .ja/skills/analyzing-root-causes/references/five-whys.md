# 5 Whys - 分析プロセス

## 問題定義

| 品質 | 例                                      |
| ---- | --------------------------------------- |
| 悪い | 「アプリが遅い」                        |
| 良い | 「ダッシュボードの読み込みに5秒かかる」 |

## 分析例

### ダッシュボードの読み込みに5秒かかる

| Step | 質問             | 回答                         | レベル         |
| ---- | ---------------- | ---------------------------- | -------------- |
| 1    | なぜ5秒？        | API `/api/dashboard` が4.5秒 | 観察可能な事実 |
| 2    | なぜAPI遅い？    | 15テーブルをクエリ           | 実装詳細       |
| 3    | なぜ15テーブル？ | 全モジュールを1ビューで表示  | 設計決定       |
| 4    | なぜ全部一度に？ | マウント時に全てレンダリング | アーキテクチャ |
| 5    | なぜマウント時？ | **遅延読み込みがない**       | 根本原因       |

**解決策**: セクション単位の遅延読み込み、オンデマンドでフェッチ

### フォームが2回送信される

| Step | 質問               | 回答                                 | レベル         |
| ---- | ------------------ | ------------------------------------ | -------------- |
| 1    | なぜ2回？          | リクエスト完了前に2回クリック        | 観察可能な事実 |
| 2    | なぜクリック可能？ | 送信中にボタン無効化されていない     | 実装詳細       |
| 3    | なぜ無効化なし？   | `isSubmitting`がハンドラ開始後に設定 | 設計決定       |
| 4    | なぜ後？           | Reactがステート更新をバッチ処理      | アーキテクチャ |
| 5    | なぜバッチ問題？   | **命令的関心事にステート使用**       | 根本原因       |

**解決策**: `useRef`で即時フラグ、または`disabled={pending}`

## ヒント

| ヒント           | 説明                            |
| ---------------- | ------------------------------- |
| 事実に基づく     | 推測ではなく証拠                |
| 早期に止めない   | 最初の「なぜ」は稀に根本原因    |
| 深く掘りすぎない | 実行可能なものに達したら止める  |
| 逆方向で検証     | 「[5]だから、したがって[4]...」 |
| 修正を検証       | 「これで問題は防げるか？」      |

## よくある根本原因

| カテゴリ       | 例                                 |
| -------------- | ---------------------------------- |
| 設計           | 間違った抽象化、欠落したパターン   |
| アーキテクチャ | 結合、欠落した境界                 |
| プロセス       | 欠落した検証、リトライロジックなし |
| 知識           | フレームワーク/ライブラリの誤解    |
| 要件           | 不明確な要件 → 間違った解決策      |
