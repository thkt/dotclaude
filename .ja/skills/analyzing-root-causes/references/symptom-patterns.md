# 症状パターン

## 症状 → 根本原因マッピング

| 症状                                | 根本原因                           | 修正                                             |
| ----------------------------------- | ---------------------------------- | ------------------------------------------------ |
| DOMを待つための`setTimeout`         | Reactライフサイクル未使用          | `useRef` + `useEffect`を使用                     |
| 二重実行を防ぐフラグ                | StrictModeまたはクリーンアップなし | クリーンアップ付き`useEffect`                    |
| ステート同期のための複数effect      | 派生ステートを独立として扱う       | 派生データに`useMemo`                            |
| propをステートに同期する`useEffect` | 不要な内部ステート                 | propを直接使用                                   |
| 子のステートにアクセスするref       | 不適切なデータフロー               | ステートをリフト、props経由で渡す                |
| 兄弟間のイベントバス                | ステートがリフトされていない       | 共通の親にリフト                                 |
| すべてをメモ化                      | 時期尚早な最適化                   | まずプロファイル、必要時のみmemo                 |
| 至る所でthrottle/debounce           | アプローチが間違っている           | CSS (`position: sticky`)、`IntersectionObserver` |
| 表示/非表示のためのJavaScript       | 過度な設計                         | `<details>`またはCSS                             |

## 判断フレームワーク

| 質問                                         | Noの場合           |
| -------------------------------------------- | ------------------ |
| 完全に防げないか？                           | より良い設計を検討 |
| よりシンプルな技術で解決できないか？         | まずHTML/CSSを試す |
| 原因を治療しているか、結果を治療しているか？ | 5 Whysを適用       |
| この修正はスケールするか？                   | アプローチを再考   |
| フレームワークのやり方か？                   | パターンを学ぶ     |
