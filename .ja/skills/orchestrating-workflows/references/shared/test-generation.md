# テスト生成パターン

コマンド間でのtest-generatorエージェント使用パターン。

## パターン概要

| パターン       | ユースケース       | テスト状態       |
| -------------- | ------------------ | ---------------- |
| Spec駆動       | 機能開発 (`/code`) | スキップモード   |
| バグ駆動       | バグ修正 (`/fix`)  | アクティブモード |
| カバレッジ駆動 | カバレッジ改善     | アクティブモード |

## パターン1: Spec駆動 (機能開発)

**ユースケース**: `/code`コマンド用にspec.mdからテスト生成。

```typescript
Task({
  subagent_type: "test-generator",
  description: "Generate skipped tests from specification",
  prompt: `
Feature: "${featureDescription}"
Spec: ${specContent}

スキップモードでテスト生成:
1. FR-xxx要件 → スキップテストケース
2. Given-When-Thenシナリオ → スキップ実行テスト
3. テスト順序: 単純 → 複雑 (Baby Steps順)
4. フレームワーク適切なスキップマーカー使用
  `,
});
```

**特徴**:

- 全テストはスキップ状態で開始
- ユーザーが1つずつ有効化
- Baby Steps順（単純 → 複雑）

## パターン2: バグ駆動 (バグ修正)

**ユースケース**: `/fix`コマンド用にリグレッションテスト生成。

```typescript
Task({
  subagent_type: "test-generator",
  description: "Generate regression test for bug fix",
  prompt: `
Bug: "${bugDescription}"
Root Cause: "${rootCause}"
Fix Applied: "${fixSummary}"

生成:
1. 元のバグを再現するテスト（今は通過すべき）
2. 修正に関連するエッジケーステスト
3. クロスコンポーネント修正なら統合テスト
  `,
});
```

**特徴**:

- テストは即座にアクティブ（スキップしない）
- リグレッション防止に焦点
- 修正周辺のエッジケース

## パターン3: カバレッジ駆動

**ユースケース**: カバレッジ改善のためのテスト追加。

```typescript
Task({
  subagent_type: "test-generator",
  description: "Generate tests for uncovered code paths",
  prompt: `
File: ${filePath}
Uncovered lines: ${uncoveredLines}
Existing test style: ${testStyle}

未カバーコードパスのテストを生成。
目標: C0 ≥90%, C1 ≥80%
  `,
});
```

## フレームワークスキップマーカー

| フレームワーク | スキップ構文                                            |
| -------------- | ------------------------------------------------------- |
| Jest/Vitest    | `it.skip('test', () => { // TODO: [SKIP] FR-001 })`     |
| Mocha          | `it.skip('test', function() { })` or `xit('test', ...)` |
| 不明           | `// TODO: [SKIP]` マーカーでコメントアウト              |

## テスト設計技法

| 技法               | 用途                     | 例                              |
| ------------------ | ------------------------ | ------------------------------- |
| 同値分割           | 同じ動作の入力をグループ | 年齢: <18, 18-120, >120         |
| 境界値分析         | エッジをテスト           | 17, 18, 120, 121                |
| デシジョンテーブル | 複雑な複数条件           | isLoggedIn × isPremium → access |

## AAAパターン

```typescript
test("説明的な名前", () => {
  // Arrange - セットアップ
  // Act - 実行
  // Assert - 検証
});
```

## 命名規則

```typescript
describe("[対象クラス/関数名]", () => {
  describe("[メソッド名/シナリオ]", () => {
    it("[条件]の場合、[期待結果]であるべき", () => {
      // AAA
    });
  });
});
```

## ベストプラクティス

| プラクティス | Good                       | Bad              |
| ------------ | -------------------------- | ---------------- |
| コンテキスト | 具体的な要件               | "テストを生成"   |
| マーカー     | FR-xxx付きの明確なスキップ | マーカーなし     |
| 順序         | 単純 → 複雑 (Baby Steps)   | ランダム順       |
| 焦点         | テストごとに1つの動作      | 複数アサーション |

## カバレッジ目標

| レベル              | 目標 | 焦点       |
| ------------------- | ---- | ---------- |
| C0 (ステートメント) | 90%  | 全行実行   |
| C1 (ブランチ)       | 80%  | 全分岐実行 |

## テスト優先度マトリクス

### [優先度1] 必ずテスト

- ビジネスロジック: 計算、バリデーション、状態遷移
- サービス/リポジトリ層: データ操作
- クリティカルパス: 課金、認証、データ永続化
- エッジケース: 境界値、null/undefined

### [優先度2] 状況次第

- 複雑なユーティリティ関数
- カスタムフック（状態管理部分）
- 複雑な変換処理

### [スキップ] テスト不要

- シンプルなプロパティアクセサ
- UIレイアウト/スタイリング
- 外部ライブラリの動作
- シンプルなCRUD（フレームワーク提供）
