# TDDサイクル (RGRC) リファレンス

テスト駆動開発のためのRed-Green-Refactor-Commitサイクル。

## クイックリファレンス

| フェーズ | 目標                       | 信頼度 | 最大時間 |
| -------- | -------------------------- | ------ | -------- |
| Red      | 期待する理由でテストが失敗 | 0.9    | 2分      |
| Green    | 通過するための最小限コード | 0.7    | 5分      |
| Refactor | 動作変更なしでクリーンに   | 0.95   | 3分      |
| Commit   | 全チェック通過             | 1.0    | 1分      |

## Baby Steps (2分サイクル)

**「各ステップで可能な限り小さな変更を行う」** - t_wada

| ステップ | 時間 | アクション                 |
| -------- | ---- | -------------------------- |
| 1        | 30s  | 最小の失敗テストを書く     |
| 2        | 1min | 最小限で通過させる         |
| 3        | 10s  | テスト実行                 |
| 4        | 30s  | 小さなリファクタ（必要時） |
| 5        | 20s  | グリーンならコミット       |

**理由**: バグは常に直近2分の変更にある。常にグリーンまで数秒。

## フェーズ詳細

### Redフェーズ

1. 失敗テストを書く/有効化
2. テスト実行 → 失敗を確認
3. **重要**: 失敗理由が意図と一致すること

```bash
npm test -- --testNamePattern="[test]"
```

**よくある間違い**: テストが即座に通過 → テストが実際の動作を確認していない。

### Greenフェーズ

1. 通過するための**最小限**のコードを書く
2. 「罪を犯してよい」 - quick/dirty OK
3. テストが要求するもののみ

```bash
npm test -- --watch --testNamePattern="[test]"
```

**よくある間違い**: テスト要件を超えた過剰実装。

### Refactorフェーズ

1. コード構造のみをクリーンに
2. テストをグリーンに保つ
3. SOLID、DRY、オッカムの剃刀を適用

**よくある間違い**: 動作を変更（構造のみ変更すべき）。

### Commitフェーズ

1. 全品質チェックを実行
2. 意味のあるメッセージでコミット
3. 安定状態を保存

```bash
npm run lint && npm test && npm run type-check
```

## 統合ポイント

| コマンド | TDD使用法                                         |
| -------- | ------------------------------------------------- |
| `/code`  | spec.mdで機能駆動 (フェーズ0: スキップモード)     |
| `/fix`   | リグレッションテストでバグ駆動 (アクティブモード) |

## RGRCチェックリスト

コピーして進捗を追跡:

```markdown
TDDサイクル:

- [ ] Red - 失敗するテスト作成 (正しい失敗理由を確認)
- [ ] Green - 最小限のコードで通過 (dirty OK)
- [ ] Refactor - コード改善 (テストをグリーンに保つ)
- [ ] Commit - 変更をコミット (全チェック通過)
```

## TDDを使わない場合

- プロトタイプ（使い捨てコード）
- 外部API統合（モックを使用）
- シンプルな単発スクリプト
- UI実験（ビジュアルファースト）
