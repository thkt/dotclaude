# /fix ワークフロー - 根本原因分析によるバグ修正

5 Whys分析とTDDリグレッションテストによるバグ修正ワークフロー。

## ワークフロー概要

```text
フェーズ1: 根本原因分析 (5 Whys)
    ↓
フェーズ1.5: リグレッションテスト先行 (Red)
    ↓
フェーズ2: 実装 (Green)
    ↓
フェーズ3: 検証 (品質ゲート)
    ↓
フェーズ3.5: 追加テスト (オプション)
    ↓
完了: 信頼度付きレポート
```

## フェーズ1: 根本原因分析

### 目的

症状ではなく真の原因を見つけるため、5 Whys手法を適用。

### Exploreエージェント使用

```typescript
Task({
  subagent_type: "Explore",
  thoroughness: "quick",
  description: "Bug context exploration",
  prompt: `
    Bug: "${bugDescription}"
    Find: 関連ファイル、依存関係、最近のコミット
    Apply 5 Whys: 症状ではなく根本原因を特定
    Return: [✓/→/?] マーカー付きで発見事項
  `,
});
```

### 5 Whysの質問

1. **なぜこれが起きた?** - 直接の原因
2. **なぜその状態が存在した?** - 何が許容したか
3. **なぜこれが検出されなかった?** - プロセスのギャップ
4. **なぜこのパターンがここにある?** - 孤立か体系的か
5. **なぜこのやり方をしている?** - 設計の欠陥?

### 分析出力

```markdown
**症状**: [ユーザーが見るもの]
**直接原因**: [何が直接失敗したか]
**根本原因**: [なぜ失敗したか]
**スコープ**: [影響ファイル]
**パターン**: [孤立/パターン/体系的]
**信頼度**: 0.XX
```

### スキップする場合

- 些細な修正（タイポ、明らかなチェック漏れ）
- 高信頼度（≥95%）
- 既知原因の本番ホットフィックス

### エスカレーションする場合

- 信頼度 <70% → `/research`
- 体系的問題 → `/think` → `/code`
- 複数コンポーネント → クイック修正の複雑さ超過

## フェーズ1.5: リグレッションテスト先行

### 目的

修正前にバグを再現する失敗テストを書く。

**TDD参照**: [@./shared/tdd-cycle.md](./shared/tdd-cycle.md)

### プロセス

1. バグを正確に再現するテストを書く
2. テスト実行 → 失敗を確認
3. 失敗がバグ説明と一致することを確認

```typescript
it("割引が合計を超えた場合、負ではなく0を返すべき", () => {
  // バグ: 0ではなく-50を返していた
  const result = calculateTotal(100, 150);
  expect(result).toBe(0);
});
```

### 検証

- [x] テストが**失敗する**（バグの存在を証明）
- [x] 失敗理由がバグと一致
- [x] エラーメッセージが明確

テストが即座に通過した場合 → テストがバグを再現していない → 修正が必要。

### スキップする場合

- ドキュメントのみの変更
- 設定更新
- ロジックなしの純粋CSS/スタイリング
- 信頼度 > 0.95 かつ些細な修正

## フェーズ2: 実装

### 信頼度ベースアプローチ

| 信頼度 | 戦略         | アプローチ            |
| ------ | ------------ | --------------------- |
| ≥90%   | 直接修正     | 素直な解決策          |
| 70-89% | 防御的修正   | チェックとガード追加  |
| <70%   | エスカレート | → /research or /think |

### オッカムの剃刀を適用

```typescript
// Good: シンプル
function calculateTotal(price, discount) {
  return Math.max(0, price - discount);
}

// Bad: 過剰設計
class PricingValidator {
  /* 50行 */
}
```

### 周辺コードの再構築はしない

バグのみを修正。リファクタリングは以下に保存:

- RGRCのRefactorフェーズ
- 別のPR

### UI問題にはCSS-First

優先順位: CSS → HTML → JavaScript

## フェーズ3: 検証

### 並列品質チェック

```bash
npm test -- --findRelatedTests &
npm run lint -- --fix &
npm run type-check &
wait
```

### 検証チェックリスト

- [x] リグレッションテスト通過（バグ修正）
- [x] 既存テスト全て通過（リグレッションなし）
- [x] リント通過（エラー0）
- [x] 型チェック通過
- [x] 手動検証完了

### 検証失敗時

| 問題                     | 対応                         |
| ------------------------ | ---------------------------- |
| リグレッションテスト失敗 | 実装を見直し                 |
| 他のテスト失敗           | リグレッション発生、修正する |
| リントエラー             | 自動修正または手動修正       |
| 型エラー                 | 型または実装を修正           |

## フェーズ3.5: 追加テスト (オプション)

### 生成する場合

- バグが微妙/複雑
- 複数のエッジケース存在
- 類似バグの可能性あり
- 重要なビジネスロジック

**パターン参照**: [@./shared/test-generation.md#pattern-2-bug-driven-bug-fixing](./shared/test-generation.md#pattern-2-bug-driven-bug-fixing)

### 生成内容

1. メインリグレッションテスト（フェーズ1.5から）
2. 修正周辺のエッジケース
3. クロスコンポーネントなら統合テスト

### エスカレーション基準

| 信頼度 | フェーズ | アクション                           |
| ------ | -------- | ------------------------------------ |
| <50%   | 根本原因 | /research を即座に実行               |
| 50-69% | 任意     | ユーザーに確認: 続行 or エスカレート |
| 70-79% | 実装     | 防御的アプローチで進行               |
| ≥80%   | 任意     | 通常通り進行                         |
