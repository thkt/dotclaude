# /fixワークフロー

## フロー

```text
根本原因 (5 Whys) → リグレッションテスト (Red) → 実装 (Green) → 検証
```

## フェーズ1: 根本原因分析

### 5 Whysの質問

| #   | 質問                          | 目的             |
| --- | ----------------------------- | ---------------- |
| 1   | なぜこれが起きた?             | 直接の原因       |
| 2   | なぜその状態が存在した?       | 何が許容したか   |
| 3   | なぜ検出されなかった?         | プロセスギャップ |
| 4   | なぜこのパターンがここにある? | 孤立か体系的か   |
| 5   | なぜこのやり方をしている?     | 設計の欠陥?      |

### 出力形式

```markdown
症状: [ユーザー向け]
根本原因: [なぜ失敗したか]
信頼度: 0.XX
パターン: [孤立/パターン/体系的]
```

## フェーズ1.5: リグレッションテスト先行

修正前にバグを再現するテストを書く。テストは**失敗**すべき。

## フェーズ2: 実装

| 信頼度 | 戦略         | アプローチ   |
| ------ | ------------ | ------------ |
| ≥90%   | 直接修正     | 素直な解決策 |
| 70-89% | 防御的       | ガード追加   |
| <70%   | エスカレート | → /research  |

**ルール**: バグのみ修正。リファクタリングは別PRに。

## フェーズ3: 検証

```bash
npm test -- --findRelatedTests &
npm run lint -- --fix &
npm run type-check &
wait
```

## エスカレーション

| 信頼度 | アクション                           |
| ------ | ------------------------------------ |
| <50%   | → /research を即座に実行             |
| 50-69% | ユーザーに確認: 続行 or エスカレート |
| ≥70%   | 通常通り進行                         |
