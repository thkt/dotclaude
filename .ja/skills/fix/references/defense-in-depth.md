# 多層検証 (Defense-in-Depth)

再発しうるバグへの多層検証。失敗が通過しうる各層にチェックを追加し、別のコードパスやリファクタによって同じバグが再発しないようにする。

## 適用タイミング

`use-context-root-cause-analysis` (5 Whys 出力) の Pattern フィールドで決まる。

| Pattern    | 動作                                                                   |
| ---------- | ---------------------------------------------------------------------- |
| Isolated   | スキップ。単一修正箇所、再発経路なし                                   |
| Recurring  | 最低でも layer 1-2 を適用。近傍に類似コード、entry/business リスクあり |
| Systematic | 4 層すべて適用。`/research` エスカレーションと組み合わせる             |

## 4 つの層

各層は独立に検証する。完全なカバレッジによりバグが構造的に不可能になる。

| Layer | 目的                                 | 適用条件                         | 例                                    |
| ----- | ------------------------------------ | -------------------------------- | ------------------------------------- |
| 1     | Entry point: 不正入力を拒否          | 外部入力が関わる場合             | 必須 param が空ならエラーをスロー     |
| 2     | Business logic: データの整合性       | ドメイン不変条件が破られうる場合 | mutation 後にエンティティ状態を検証   |
| 3     | Environment guards: コンテキスト安全 | 操作に環境依存リスクがある場合   | テスト環境で破壊的操作を拒否          |
| 4     | Debug instrumentation: 法医学的記録  | 失敗が再現困難な場合             | リスク操作前に stack trace 付きでログ |

## パターンの適用

| #   | Step                                                             |
| --- | ---------------------------------------------------------------- |
| 1   | データフローを追跡: 不正値の発生元、消費先                       |
| 2   | データが通過する全チェックポイントをマップ                       |
| 3   | Pattern により層を選択 ("適用タイミング" を参照)                 |
| 4   | 選択した各層に検証を追加                                         |
| 5   | 各層を独立にテスト: 1 つを bypass して別の層が捕捉することを確認 |

## 検証

| チェック                                | 必須                  |
| --------------------------------------- | --------------------- |
| Pattern が層選択を駆動した              | Yes                   |
| 各層が独立にテスト可能                  | Yes                   |
| 少なくとも 1 つの層に bypass テスト実施 | Yes (Systematic パス) |
