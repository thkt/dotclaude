# 多層防御バリデーション

バグ修正後、障害が再発しうるすべてのレイヤーにバリデーションを追加する。単一のチェックは、異なるコードパス、リファクタリング、モックによりバイパスされうる。

単一バリデーション:「バグを修正した。」多層バリデーション:「バグを構造的に不可能にした。」

## 4つのレイヤー

| レイヤー | 目的                             | 例                                           |
| -------- | -------------------------------- | -------------------------------------------- |
| 1        | エントリポイント: 不正入力を拒否 | 必須パラメータが空ならスロー                 |
| 2        | ビジネスロジック: データの整合性 | ドメイン不変条件が保持されることを検証       |
| 3        | 環境ガード: コンテキスト安全性   | テスト環境での危険な操作を拒否               |
| 4        | デバッグ計装: フォレンジクス     | リスクのある操作前にスタックトレース付きログ |

## 適用タイミング

根本原因の確認後、「この障害は別のコードパスで再発しうるか？」と問う。Yesなら、障害が通過しうる各レイヤーにバリデーションを追加する。

## パターンの適用

1. データフローをトレース: 不正な値はどこで発生し、どこで使用されるか
2. すべてのチェックポイントをマップ: データが通過するすべてのポイントをリスト
3. 各レイヤーにバリデーションを追加: エントリ、ビジネス、環境、デバッグ
4. 各レイヤーをテスト: レイヤー1をバイパスしてレイヤー2がキャッチすることを検証

各レイヤーは独立してバリデーションしなければならない。異なるコードパスは異なるレイヤーをバイパスする。完全なカバレッジのみがバグを構造的に不可能にする。
