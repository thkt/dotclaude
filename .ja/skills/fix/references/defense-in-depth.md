# 多層防御バリデーション

再発しうるバグに対し、障害が通過しうる各レイヤーにチェックを追加する。異なるコードパスやリファクタリングによって同じバグが再発することを構造的に防ぐ。

## 適用タイミング

`use-context-root-cause-analysis`（5 Whys）出力の Pattern フィールドで駆動する。

| Pattern    | アクション                                                             |
| ---------- | ---------------------------------------------------------------------- |
| Isolated   | スキップ。修正箇所が単一、再発経路なし                                 |
| Recurring  | 最低でも Layer 1-2 を適用。近傍に類似コード、エントリ/ビジネスのリスク |
| Systematic | 全 4 Layer を適用。`/research` エスカレートと併用                      |

## 4 つのレイヤー

各レイヤーは独立にバリデーションする。完全カバレッジでバグを構造的に不可能にする。

| Layer | 目的                               | 適用条件                      | 例                                       |
| ----- | ---------------------------------- | ----------------------------- | ---------------------------------------- |
| 1     | エントリポイント: 不正入力を拒否   | 外部入力が関与                | 必須パラメータが空ならスロー             |
| 2     | ビジネスロジック: データの整合性   | ドメイン不変条件が破られうる  | エンティティ状態を変更後に検証           |
| 3     | 環境ガード: コンテキスト安全性     | 環境依存リスクがある          | テスト環境での破壊的操作を拒否           |
| 4     | デバッグ計装: フォレンジクス       | 障害が再現困難                | リスクのある操作前にスタックトレース付きログ |

## 適用手順

| # | ステップ                                                          |
| - | ----------------------------------------------------------------- |
| 1 | データフローをトレース: 不正値の発生源と消費点を特定              |
| 2 | データが通過する全チェックポイントをマップ                        |
| 3 | Pattern に基づいて Layer を選択（「適用タイミング」参照）         |
| 4 | 選択した各 Layer にバリデーションを追加                           |
| 5 | 各 Layer を独立にテスト: 1 つをバイパスし、別の Layer で捕捉を確認 |

## 検証

| チェック項目                            | 必須                    |
| --------------------------------------- | ----------------------- |
| Pattern が Layer 選択を駆動したか       | Yes                     |
| 各 Layer が独立にテスト可能か           | Yes                     |
| 少なくとも 1 Layer のバイパステスト実施 | Yes (Systematic パス)   |
