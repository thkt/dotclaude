# /fix - クイックバグ修正

根本原因分析とTDD検証による迅速なバグ修正。

## 入力

- `$1`: バグ説明 または `/audit` の提案ID（例: `SUG-001`）
- スコープ: 小さく理解しやすい問題（1-3ファイル）

### ルーティング

| `$1` パターン    | モード                           |
| ---------------- | -------------------------------- |
| `/^SUG-[0-9]+$/` | 提案IDモード                     |
| empty            | 修正プロンプト (AskUserQuestion) |
| otherwise        | 標準フロー                       |

### 修正プロンプト

`$1` が空の場合 AskUserQuestion で確認。

| 質問     | 選択肢                                 |
| -------- | -------------------------------------- |
| 修正種別 | Bug fix / Error message / Test failure |
| 説明     | [Other で自由入力]                     |

### 提案IDモード

| Step | アクション                                                        |
| ---- | ----------------------------------------------------------------- |
| 1    | `$HOME/.claude/workspace/history/` から最新スナップショットを読む |
| 2    | IDで一致する提案を検索                                            |
| 3    | 修正を直接適用（RCA スキップ、監査結果を信頼）                    |
| 4    | テストがパスすることを確認                                        |
| 5    | 対象ファイルのみ再監査を提案: `/audit <変更ファイル>`             |

## 委譲マップ

| タイプ    | 委譲先                           | 目的                                        |
| --------- | -------------------------------- | ------------------------------------------- |
| Skill     | `use-context-root-cause-analysis` | Non-obvious バグの 5 Whys                   |
| Agent     | `test-generator`                 | 症状 + 再現手順からリグレッションテスト生成 |
| Agent     | `build-error-resolver`           | TypeScript / ビルドエラーの切り分け         |
| Reference | `references/defense-in-depth.md` | Recurring / Systematic 時の多層バリデーション |

## 実行

### ビルドチェック

プロジェクトのビルドコマンドを実行（package.json またはプロジェクト設定から検出）。

| 結果             | アクション                                                  |
| ---------------- | ----------------------------------------------------------- |
| ビルドエラーあり | `Task` で `subagent_type: build-error-resolver`、その後終了 |
| エラーなし       | トリアージへ続行                                            |

### トリアージ

| 条件                                                 | パス                        |
| ---------------------------------------------------- | --------------------------- |
| 原因 1 箇所特定 + 修正 1-3 行 + 類似パターンなし     | Obvious: 直接修正           |
| 間欠的, 複数再現条件, ルート不明, いずれか当てはまる | Non-obvious: フルプロトコル |

### Obvious: 直接修正

| Step | アクション                                 |
| ---- | ------------------------------------------ |
| 1    | 最小限の修正を適用                         |
| 2    | 影響範囲のテストを実行（なければスキップ） |

### Non-obvious: フルプロトコル

| Step | アクション                                                                      |
| ---- | ------------------------------------------------------------------------------- |
| 1    | `Skill("use-context-root-cause-analysis")` で 5 Whys。出力: Symptom / Root cause / Pattern |
| 2    | `Task(subagent_type: test-generator)` でリグレテスト（症状 + 再現手順のみ渡す） |
| 3    | リグレテストが Red であることを確認                                             |
| 4    | 修正を適用                                                                      |
| 5    | リグレテストが Green、他テストに回帰がないことを確認                            |
| 6    | Pattern ∈ {Recurring, Systematic} の場合 `references/defense-in-depth.md` を適用 |

## エスカレーション

Objective trigger のみ。信頼度の自己申告なし。

| トリガ                       | アクション                                          |
| ---------------------------- | --------------------------------------------------- |
| RCA で原因特定できず         | エスカレート → `/research`                          |
| 3 回の修正試行失敗           | 停止。フルコンテキストで `/research` へエスカレート |
| 複数ファイル影響（>3 files） | 委譲 → `/code`                                      |
| 新機能スコープ               | 委譲 → `/think`                                     |
| Pattern = Systematic         | エスカレート → `/research`                          |

≥3 ルール: 3 回の異なる修正試行が失敗した場合、問題はローカルなバグではなくアーキテクチャ的な可能性が高い。エスカレーションなしに 4 回目を試みてはならない。

## エラーハンドリング

| エラー                      | アクション                                 |
| --------------------------- | ------------------------------------------ |
| build-error-resolver 失敗   | エラーを提示、ユーザーにガイダンスを求める |
| test-generator タイムアウト | リグレッションテストをスキップ、修正に進む |
| 修正後もテスト失敗          | 根本原因を再分析またはエスカレート         |

## 検証

| チェック項目                             | 必須                           |
| ---------------------------------------- | ------------------------------ |
| 根本原因を特定したか（Non-obvious パス） | Yes                            |
| 全テストがパスするか                     | Yes                            |
| RCA から Pattern フィールドを記録したか  | Yes（Non-obvious パス）        |
| 必要時 defense-in-depth を適用したか     | Yes（Recurring/Systematic のみ） |
