# PR スクリーニングレビュー

AI が事前分析を行い、レビュアーがゼロから理解するのではなく
最終判断に集中できるようにする。

## プロセス

| Step | アクション                | 目的                |
| ---- | ------------------------- | ------------------- |
| 1    | PR コンテキストを収集     | 背景と意図の把握    |
| 2    | ファイルごとに変更を要約  | before/after の理解 |
| 3    | 依存関係の影響を確認      | 回帰リスク          |
| 4    | 指摘事項 + コメントを生成 | レビュアー向け出力  |

## PR コンテキスト収集

```bash
# メタデータ
gh pr view --json title,body,labels,files,url $PR

# 差分
gh pr diff $PR

# 既存コメント
gh pr view --comments $PR

# インラインコメント
gh api repos/{owner}/{repo}/pulls/{number}/comments \
  --jq '.[] | {file: .path, user: .user.login, comment: .body}'
```

gh 出力フィールドに `author` を含めないこと。

## コメントラベル

| ラベル   | 意味                           | 重要度 |
| -------- | ------------------------------ | ------ |
| `[must]` | マージ前に修正必須             | 高     |
| `[want]` | 修正推奨、ブロッキングではない | 中     |
| `[imo]`  | 個人的見解、採否は任意         | 低     |
| `[ask]`  | 確認が必要な質問               | -      |
| `[nits]` | 軽微なスタイル/フォーマット    | 低     |
| `[info]` | 情報共有、アクション不要       | -      |

## コメントトーン

| ルール   | 詳細                                                 |
| -------- | ---------------------------------------------------- |
| 敬意     | 実装者への敬意を忘れず、命令口調を避ける             |
| 簡潔     | コメント1件あたり最大3行                             |
| 提案型   | 「〜のように見受けられます」「〜は間違いです」は不可 |
| 位置指定 | コメントごとに file:line を明記                      |
