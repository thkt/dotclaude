# 症状パターン

## 症状 → 根本原因マッピング

| 症状                                | 根本原因                           | 修正                                             |
| ----------------------------------- | ---------------------------------- | ------------------------------------------------ |
| DOM を待つための `setTimeout`       | React ライフサイクル未使用         | `useRef` + `useEffect` を使用                    |
| 二重実行を防ぐフラグ                | StrictMode またはクリーンアップなし | クリーンアップ付き `useEffect`                   |
| ステート同期のための複数 effect     | 派生ステートを独立として扱う       | 派生データに `useMemo`                           |
| prop をステートに同期する `useEffect` | 不要な内部ステート                 | prop を直接使用                                  |
| 子のステートにアクセスする ref      | 不適切なデータフロー               | ステートをリフト、props 経由で渡す               |
| 兄弟間のイベントバス                | ステートがリフトされていない       | 共通の親にリフト                                 |
| すべてをメモ化                      | 時期尚早な最適化                   | まずプロファイル、必要時のみ memo                |
| 至る所で throttle/debounce          | アプローチが間違っている           | CSS (`position: sticky`)、`IntersectionObserver` |
| 表示/非表示のための JavaScript      | 過度な設計                         | `<details>` または CSS                           |

## 判断フレームワーク

| 質問                                         | No の場合          |
| -------------------------------------------- | ------------------ |
| 完全に防げないか？                           | より良い設計を検討 |
| よりシンプルな技術で解決できないか？         | まず HTML/CSS を試す |
| 原因を治療しているか、結果を治療しているか？ | 5 Whys を適用      |
| この修正はスケールするか？                   | アプローチを再考   |
| フレームワークのやり方か？                   | パターンを学ぶ     |

## 根本原因カテゴリ

| カテゴリ       | 例                                 |
| -------------- | ---------------------------------- |
| 設計           | 間違った抽象化、欠落したパターン   |
| アーキテクチャ | 結合、欠落した境界                 |
| プロセス       | 欠落した検証、リトライロジックなし |
| 知識           | フレームワーク/ライブラリの誤解    |
| 要件           | 不明確な要件 → 間違った解決策      |
