# Code Simplifier

## Purpose

| Goal           | Description                                              |
| -------------- | -------------------------------------------------------- |
| 無駄を除去     | AI slop、冗長テスト、過剰防御を取り除く                  |
| 振る舞いを保持 | ランタイムの振る舞いを変更しない、公開 API を壊さない    |
| 意図を明確化   | 実装が、コードが実際に行うことに近づいて読めるようにする |

## Posture

すべてのコンフリクトでは保持が勝つ。無駄であると証明できるものだけを削除し、保持ルールを確認した後でのみ削除する。

スキップ理由内で禁止する表現: 「looks unused」、「probably dead」、「seems redundant」。これらに手を伸ばしたら、決定する前に検証チェックを実行する。

## Input

簡略化対象のファイル範囲。既定は base に対する git diff。

| Field        | Type     | Example                             |
| ------------ | -------- | ----------------------------------- |
| target_scope | enum     | git_diff (default) / explicit_files |
| target_files | optional | [src/api/client.ts, src/utils.ts]   |
| diff_base    | optional | HEAD~1, main, or feature branch ref |

## Principles

| Priority | Principle              |
| -------- | ---------------------- |
| 1        | 機能の保持 (壊さない)  |
| 2        | 迷ったら残す           |
| 3        | 簡潔さより明確さ       |
| 4        | 構造ではなく無駄を除去 |
| 5        | プロジェクト規約に従う |

## Removal Targets (AI Slop)

| Category           | Examples                                                                                                                            |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------- |
| 冗長コメント       | コードを言い換える HOW コメント、すでに命名済みの識別子を言い換える WHY コメント、削除済みコードのマーカー、`// added for X` スタブ |
| 過剰防御           | 内部限定のバリデーション、到達不能なエラーハンドリング                                                                              |
| 過剰設計           | 単一実装インターフェース、ラッパークラス、一度きりのヘルパー                                                                        |
| 複雑性             | ネストされた三項演算子、深いネスト (>3)、50 行超の関数                                                                              |
| 無意味なテスト     | トートロジー、重複アサーション、空/skip、自身のモック                                                                               |
| 冗長テスト         | 些細な差分のコピペケース、同じ振る舞いを繰り返しテスト                                                                              |
| 冗長な記述のテスト | テスト間で繰り返されるセットアップ、単一の振る舞いに対する過剰なアサーション                                                        |
| 些細なテスト       | getter/setter テスト、フレームワーク既定値テスト、ランタイムでの型ガード                                                            |
| デッドコード       | 未使用 import、未参照変数、コメントアウトされたコード                                                                               |
| 後方互換性         | リネームされた `_vars`、削除コードの再エクスポート、`// removed` スタブ                                                             |

## Preservation Rules

除去対象とのコンフリクトでは保持が勝つ。

### Comments to keep

| Keep                                    | Why                                          |
| --------------------------------------- | -------------------------------------------- |
| WHY 説明 (動機、制約)                   | コードだけからは復元できない                 |
| ドメイン用語の定義                      | 読者がドメイン文脈を持たない可能性           |
| 自明でないエッジケースの根拠            | 将来の開発者による再発防止                   |
| issue/ticket 参照付きの workaround メモ | 将来のクリーンアップのための外部文脈リンク   |
| トラッキング参照付き TODO (issue 番号)  | アクティブな作業項目、デッドコメントではない |

### Tests to keep

| Keep                                        | Why                                      |
| ------------------------------------------- | ---------------------------------------- |
| ある振る舞いに対する唯一のカバレッジ        | 他のテストがこのパスを実行していない     |
| リグレッションテスト (バグ修正と共に追加)   | 過去の失敗を文書化、削除すると再発を招く |
| 境界/エッジケーステスト                     | バグはエッジケースに住む                 |
| プラットフォーム/環境クセを文書化するテスト | 特定環境での破壊を防ぐ                   |
| 異なるシナリオバリエーションを持つテスト    | それぞれが異なる入出力組み合わせをテスト |

### Verification before removal

| Target  | Check before removing                                               |
| ------- | ------------------------------------------------------------------- |
| Test    | 同じ関数をカバーする他のテストを Grep。カバーされている場合のみ削除 |
| Comment | WHY (HOW ではなく) に答えているか? WHY なら残す                     |
| Helper  | 1 箇所からのみ呼ばれる? インライン化。2 箇所以上? 残す              |

## Simplification Rules

| Rule                   | Action                           |
| ---------------------- | -------------------------------- |
| ネストされた三項演算子 | if/else または switch に置き換え |
| 単一使用ヘルパー       | 呼び出し箇所にインライン化       |
| ロジックのないラッパー | ラッパーを削除、内部を直接使う   |
| 推論可能な型 (TS)      | 冗長な型注釈を削除               |
| 再代入されない let     | const に変更                     |
| 未使用 import          | 削除                             |
| 3 レベル超のネスト     | 抽出または early return を使う   |

## Test Audit

| Smell                      | Fix                                                |
| -------------------------- | -------------------------------------------------- |
| 曖昧なテスト名             | 3 部構成: what / scenario / expected               |
| AAA フェーズ混在           | 空行で Arrange、Act、Assert を分離                 |
| コピペされたテストケース   | `test.each` / parameterized で統合                 |
| テスト間の重複セットアップ | `beforeEach` または共有ヘルパーに抽出              |
| 同じパスの複数アサーション | 最小カバーセットに削減                             |
| 冗長なアサーションチェーン | 対象を絞ったマッチャを使用 (`toMatchObject` 等)    |
| 内部の過剰モック           | 公開 API 経由で振る舞いをテスト、impl モックを除去 |

## Process

| Step | Action                                                       | Output             | On dead-end                              |
| ---- | ------------------------------------------------------------ | ------------------ | ---------------------------------------- |
| 1    | 対象スコープを特定 (git diff または明示ファイル)             | ファイル一覧       | 空の diff、「No changes」を出力          |
| 2    | 各変更ファイルを読む                                         | ファイル内容       | ファイル欠落、Skipped セクションに記録   |
| 3    | プロダクションコードに除去対象を適用、保持を先に確認         | 編集をキュー       | 保持ルール発火、残す                     |
| 4    | 簡略化ルールを適用                                           | 編集をキュー       | ルールがプロジェクト規約に違反、スキップ |
| 5    | テストファイルにテスト監査ルールを適用、カバレッジを先に検証 | テスト編集をキュー | 唯一のカバレッジ、テストを残す           |
| 6    | ファイルを直接編集、すべての振る舞いを保持                   | ファイル更新       | -                                        |
| 7    | 出力テンプレートを埋める (全セクション)                      | 最終レポート       | -                                        |

## Constraints

| Constraint      | Detail                                                 |
| --------------- | ------------------------------------------------------ |
| Scope           | 明示的に広く指示されない限り、最近変更されたコードのみ |
| No new features | 機能を追加しない                                       |
| No refactoring  | 構造変更は無駄除去のときのみ                           |
| No formatting   | linter/formatter に任せる                              |
| Conservative    | 削除が安全か不明なら、コードを残す                     |

## Output

3 セクションすべて必須。セクションに発見事項がない場合は「No changes」を使う。

```text
Code: <list changes with file:line>
Tests: <list changes with file:line>
Skipped: <list files not audited, with reason>
```
