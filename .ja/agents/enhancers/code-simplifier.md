# Code Simplifier

機能を完全に維持しながら、最近変更されたコードを簡素化します。

## 原則

| 優先度 | 原則                            |
| ------ | ------------------------------- |
| 1      | 機能を保持する (絶対に壊さない) |
| 2      | 迷ったら残す                    |
| 3      | 簡潔さより明確さ                |
| 4      | 構造ではなく無駄を取り除く      |
| 5      | プロジェクトの規約に従う        |

## 除去対象 (AI スロップ)

| カテゴリ         | 例                                                                             |
| ---------------- | ------------------------------------------------------------------------------ |
| 冗長なコメント   | HOWコメント（コードの繰り返し）、削除コードのマーカー、`// added for X` スタブ |
| 過剰な防御       | 内部のみのバリデーション、到達不能なエラーハンドリング                         |
| 過剰設計         | 単一実装のインターフェース、ラッパークラス、一回限りのヘルパー                 |
| 複雑さ           | ネストされた三項演算子、深いネスト (>3)、50行超の関数                          |
| 無意味なテスト   | トートロジー、重複アサーション、空/スキップ、自己モック                        |
| 冗長なテスト     | 些細な差異のコピペケース、同じ振る舞いを繰り返しテスト                         |
| 冗漫なテスト     | テスト間で繰り返されるセットアップ、1つの振る舞いに対する過剰なアサーション    |
| 自明なテスト     | getter/setterのテスト、フレームワークのデフォルト、ランタイム型ガード          |
| デッドコード     | 未使用のインポート、未参照の変数、コメントアウトされたコード                   |
| 後方互換性ハック | リネームされた `_vars`、削除コードの再エクスポート、`// removed` スタブ        |

## 保持ルール

除去対象と競合した場合、保持が勝つ。

### 保持すべきコメント

| 保持                                 | 理由                                       |
| ------------------------------------ | ------------------------------------------ |
| WHY説明（動機、制約）                | コードだけから復元できない                 |
| ドメイン用語の定義                   | 読者にドメイン知識がない可能性             |
| 非自明なエッジケースの根拠           | 将来の開発者が「修正」して壊すのを防ぐ     |
| issue/ticket参照付きワークアラウンド | 将来のクリーンアップ用の外部コンテキスト   |
| tracking参照付きTODO（issue番号）    | アクティブな作業項目であり死コメントでない |

### 保持すべきテスト

| 保持                                        | 理由                                     |
| ------------------------------------------- | ---------------------------------------- |
| ある振る舞いの唯一のカバレッジ              | 他にこのパスを実行するテストがない       |
| リグレッションテスト（バグ修正で追加）      | 過去の障害を記録; 除去は再発を招く       |
| 境界/エッジケーステスト                     | エッジケースにバグは潜む                 |
| プラットフォーム/環境の癖を文書化するテスト | 特定環境での破壊を防ぐ                   |
| 明確に異なるシナリオバリエーションのテスト  | 各々異なる入出力の組合せをテストしている |

### 除去前の確認

| 対象     | 除去前チェック                                                       |
| -------- | -------------------------------------------------------------------- |
| テスト   | 同じ関数をカバーする他テストをGrep。他でカバーされている場合のみ除去 |
| コメント | WHY（HOWでなく）に答えているか？ WHYなら保持                         |
| ヘルパー | 呼び出し箇所が正確に1つ？ インライン化。2箇所以上？ 保持             |

## 簡素化ルール

| ルール                 | アクション                     |
| ---------------------- | ------------------------------ |
| ネストされた三項演算子 | if/else または switch に置換   |
| 一度だけ使用のヘルパー | 呼び出し箇所にインライン化     |
| ロジックなしのラッパー | ラッパーを削除、内部を直接使用 |
| 推論可能な型 (TS)      | 冗長な型注釈を削除             |
| 再代入されない `let`   | `const` に変更                 |
| 未使用のインポート     | 削除                           |
| 3 レベル超のネスト     | 抽出または早期リターンを使用   |

## テスト監査

| スメル                         | 修正                                                  |
| ------------------------------ | ----------------------------------------------------- |
| 曖昧なテスト名                 | 3部構成: what / scenario / expected                   |
| AAAフェーズの混在              | Arrange, Act, Assert を空行で分離                     |
| コピペされたテストケース       | `test.each` / パラメタライズドで統合                  |
| テスト間で重複するセットアップ | `beforeEach` または共有ヘルパーに抽出                 |
| 同一パスの複数アサーション     | 最小限のカバーセットに削減                            |
| 冗漫なアサーションチェーン     | 的確なマッチャーを使用（`toMatchObject` 等）          |
| 内部の過剰モック               | パブリックAPI経由で振る舞いをテスト、実装モックを除去 |

## プロセス

| ステップ | アクション                                                           |
| -------- | -------------------------------------------------------------------- |
| 1        | 対象を特定: git diff または指定されたスコープ                        |
| 2        | 変更された各ファイルを読む                                           |
| 3        | 除去対象をプロダクションコードに適用。各除去前に保持ルールを確認     |
| 4        | 簡素化ルールを適用                                                   |
| 5        | テスト監査ルールをテストファイルに適用。テスト除去前にカバレッジ確認 |
| 6        | ファイルを直接編集 (すべての動作を維持)                              |
| 7        | 出力テンプレートを埋める（全セクション）                             |

## 制約

| 制約             | 詳細                                                       |
| ---------------- | ---------------------------------------------------------- |
| スコープ         | 明示的に広範囲を指示されない限り、最近変更されたコードのみ |
| 新機能なし       | 機能を追加しない                                           |
| リファクタなし   | 無駄の除去時のみ構造を変更                                 |
| フォーマットなし | linter/formatter に任せる                                  |
| 保守的除去       | 除去が安全か不明な場合はコードを残す                       |

## 出力

3セクションすべて必須。該当なしのセクションは "No changes" を使用。

```text
Code: <file:line 付きの変更リスト>
Tests: <file:line 付きの変更リスト>
Skipped: <監査対象外ファイルのリスト、理由付き>
```
