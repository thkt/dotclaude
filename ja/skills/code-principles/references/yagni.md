# YAGNI - You Aren't Gonna Need It（それは必要ない）

**核心原則**：実際に必要になるまで機能を追加しない

## 哲学

「You Aren't Gonna Need It」 - エクストリームプログラミングの核心プラクティスの1つ

原則はシンプルだが強力：

- **実装しない** - 後で必要になると思う機能を
- **構築しない** - 想像上の将来のユースケースのための抽象化を
- **最適化しない** - 測定していない問題を
- **追加しない** - 「念のため」の柔軟性を

## YAGNIが重要な理由

### 早すぎる機能のコスト

書くすべてのコード行にはコストがある：

- **保守負担**：コードはテスト、文書化、更新が必要
- **認知的負荷**：より多くのコード = より多くの理解すべき複雑さ
- **機会費用**：YAGNI機能に費やす時間は実際のニーズに費やさない時間
- **誤った仮定**：将来の予測は通常間違っている

### 洞察

> 「最良のコードは、まだ存在せず必要ないコードである」

ソフトウェア開発は以下について：

- **今日の知識**で今日の問題を解決する
- **明日の知識**で明日の問題に適応する

## 成果物ファーストの開発

**アーキテクチャではなく、成果物を出荷する**

### 実装フェーズ

この進行に従い、必要な時にのみ各フェーズを追加：

1. **動作させる** - 直近の問題を解決
   - 焦点：機能性
   - 品質基準：基本テストをパス
   - タイムライン：数時間から数日

2. **回復力を持たせる** - エラーが発生した時にエラーハンドリングを追加
   - 焦点：信頼性
   - 品質基準：エッジケースを処理
   - タイムライン：本番使用が失敗モードを明らかにした後

3. **高速化する** - 遅さが測定された時に最適化
   - 焦点：パフォーマンス
   - 品質基準：パフォーマンス要件を満たす
   - タイムライン：プロファイリングがボトルネックを示した後

4. **柔軟にする** - ユーザーが要求した時にオプションを追加
   - 焦点：拡張性
   - 品質基準：実際のユースケースをサポート
   - タイムライン：複数の類似リクエストの後

**重要**：証拠が要求するまでフェーズ2-4を行わない。

## 決定フレームワーク

コードを追加する前に、これらの質問をする：

### 4つの質問

1. **これは今存在する実際の問題を解決しているか？**
   - いいえなら：スキップ
   - はいなら：続行

2. **これは本番で実際に失敗したか？**
   - いいえなら：まだエラーハンドリングを追加しない
   - はいなら：ターゲットを絞った修正を追加

3. **ユーザーがこれについて不満を言ったか？**
   - いいえなら：機能を追加しない
   - はいなら：まず実際のニーズを理解

4. **問題の測定された証拠はあるか？**
   - いいえなら：最適化しない
   - はいなら：プロファイリングしてからボトルネックを修正

**ルール**：質問1に「いいえ」と答えたら、停止。進まない。

## 例

### ❌ YAGNIに違反

```typescript
// 誰も求めていない柔軟性を追加
class UserService {
  constructor(
    private cache: CacheStrategy,      // まだ不要
    private logger: LoggerStrategy,    // まだ不要
    private validator: ValidationStrategy,  // まだ不要
    private transformer: DataTransformer    // まだ不要
  ) {}

  // 将来の拡張性のためのプラグインシステム
  plugins: Plugin[] = []
  registerPlugin(plugin: Plugin) { }  // 誰も使わない
}

// 「念のため」の抽象化
interface DatabaseAdapter {
  query(): Promise<Result>
}

class PostgresAdapter implements DatabaseAdapter { }
class MySQLAdapter implements DatabaseAdapter { }
class MongoAdapter implements DatabaseAdapter { }

// 現実：Postgresしか使っていない
```

**問題**：

- 必要な10倍のコード
- テストされていないコードパス
- 未使用機能の保守負担
- 読み手の認知的オーバーヘッド

### ✅ YAGNIに従う

```typescript
// 実際のニーズを解決
class UserService {
  async getUser(id: string): Promise<User> {
    return db.query('SELECT * FROM users WHERE id = $1', [id])
  }
}

// 2つ目のデータベースが現れたら抽象化を追加
// プロファイリングがデータベースがボトルネックだと示したらキャッシングを追加
// 無効なデータがバグを引き起こしたら検証を追加
```

**利点**：

- シンプルで直接的
- テストしやすい
- 理解しやすい
- ニーズが変わった時にリファクタリングしやすい

## 一般的なYAGNI違反

### 1. 早すぎる抽象化

```typescript
// ❌ YAGNI違反
interface PaymentProcessor {
  process(amount: number): Promise<Result>
}

class StripePaymentProcessor implements PaymentProcessor { }
// 他のプロセッサは存在せず計画もない

// ✅ YAGNI準拠
async function processPayment(amount: number) {
  return stripe.charge(amount)
}
// 2つ目のプロセッサが実際に必要になったらインターフェースを追加
```

### 2. 早すぎる最適化

```typescript
// ❌ YAGNI違反
function getUserPosts(userId: string) {
  // 複雑なキャッシング、ページネーション、プリフェッチ
  // 誰も速度について不満を言っていない
}

// ✅ YAGNI準拠
function getUserPosts(userId: string) {
  return db.query('SELECT * FROM posts WHERE user_id = $1', [userId])
}
// 実際のパフォーマンス問題を測定した後に最適化
```

### 3. 早すぎる設定

```typescript
// ❌ YAGNI違反
const config = {
  cache: {
    ttl: env.CACHE_TTL,
    max: env.CACHE_MAX,
    strategy: env.CACHE_STRATEGY,
    compression: env.CACHE_COMPRESSION
  },
  // さらに50の設定オプション
}

// ✅ YAGNI準拠
const CACHE_TTL = 3600  // バリエーションが必要になるまでハードコード
```

### 4. 汎用的な「再利用可能な」コンポーネント

```typescript
// ❌ YAGNI違反
<DataGrid
  columns={columns}
  data={data}
  sorting={sorting}
  filtering={filtering}
  pagination={pagination}
  grouping={grouping}
  // 使われていない機能のためのさらに30のprops
/>

// ✅ YAGNI準拠
<UserTable users={users} />
// 実際のニーズのためのシンプルなコンポーネント
// 3つ目の類似テーブルが現れたら汎化（3回ルール）
```

## 複雑さを追加すべき時

YAGNIは「決して先を計画しない」という意味ではない。以下の場合に複雑さを追加：

### 事前構築の正当な理由

1. **確認されたタイムライン**を持つ既知の将来要件
   - 例：契約がフェーズ2でマルチテナントを指定
   - 注意：構築前に本当に必要か確認

2. **リファクタリングコスト**が後で禁止的になる
   - 例：100万レコードに影響するデータベーススキーマ移行
   - 注意：本当に確実か？コストを測定

3. **コンプライアンス要件**で最初から構築が必要
   - 例：GDPRデータ保持要件
   - 注意：最小実行可能なコンプライアンスを構築

### テスト

自問：

- 「これが必要になるという証拠は何か？」
- 「後で追加するコストと今のコストは？」
- 「私は推測しているのか、実際の要件に応答しているのか？」

推測しているなら、YAGNIは言う：構築するな。

## YAGNIマインドセット

### 核心原則

- **ハッピーパス**から始める
- **現実への応答**としてのみ複雑さを追加
- すべてのコード行は**負債**
- YAGNIは**デフォルト位置**

### 開発フロー

```markdown
要件 → 最もシンプルな解決策 → 出荷 → 学習 → 反復

ではない：

要件 → すべての将来を予想 → 柔軟なシステム構築 → 出荷 → 柔軟性を決して使わない
```

### チームプラクティス

これらのフレーズを正常化：

- 「想像する問題ではなく、持っている問題を解決しよう」
- 「必要な時にそれを追加できる」
- 「これが必要だという証拠は何か？」
- 「シンプルに出荷し、フィードバックに基づいて反復」

## 他の原則との統合

### YAGNI + オッカムの剃刀

**オッカムの剃刀**：最もシンプルな解決策を選ぶ
**YAGNI**：必要ないものを構築しない

一緒に：**今日の問題のみ**を解決する最もシンプルな解決策を選ぶ

### YAGNI + プログレッシブエンハンスメント

**プログレッシブエンハンスメント**：最小限から始め、徐々に強化
**YAGNI**：想像上の将来のために構築しない

一緒に：最小実行可能機能を構築し、**実際の使用**に基づいて強化

### YAGNI + TDD

**TDD**：テストを書き、パスする最小限のコードを書く
**YAGNI**：テストされていない機能を実装しない

一緒に：テスト駆動開発は**自然にYAGNIを強制**

## レッドフラグ

以下を聞いた時、YAGNIに違反している：

- 「後でこれが必要かもしれない」
- 「念のため柔軟にしよう」
- 「これは将来時間を節約する」
- 「スケールに備えるべき」
- 「Xをサポートする必要がある場合はどうするか？」

**対抗する**：

- 「必要だという証拠は何か？」
- 「必要が生じた時に追加できるか？」
- 「後で追加するコストは？」
- 「今その問題はあるか？」
- 「Xが起きた時に解決しよう」

## 覚えておくこと

> 「将来を予測する最良の方法は、それを待つことである」 - Pragmatic Programmers

**重要なポイント**：

- **今**のために構築し、想像上の将来のためではない
- **証拠への応答**として複雑さを追加し、推測ではない
- すべての機能には**コスト**がある（保守、複雑さ、機会）
- **シンプルさ**は機能であり、制限ではない
- **常により多く追加できる**（しかし削除することは稀）
