# esa 日報作成ツール

## 概要

このスキルは、Google Calendarと統合し、構造化テンプレート形式に従うことで、esa.ioでの日報（日次レポート）の自動作成を可能にします。

## 目的

以下により日報作成プロセスを効率化:

- その日のカレンダーイベントを自動取得
- 事前定義されたテンプレートに従って情報を構造化
- 適切なカテゴリでesaにWIP投稿を作成
- チームコミュニケーション用の一貫したフォーマットを提供

## このスキルを使用するタイミング

- ユーザーが日報作成をリクエスト（例: 「今日の日報を作成」、「日報作って」）
- ユーザーが日報の作成や投稿について言及
- ユーザーが今日の活動をesa用に要約するよう依頼

## 設定

### デフォルト設定

- **esaチーム**: `gaji`
- **screen_name**: 現在のユーザーから自動検出（`esa_get_team_members`を使用）
- **テンプレート投稿番号**: `#126`
- **カテゴリ**: `report/daily-report/`
- **投稿ステータス**: WIP（Work In Progress）

### カスタマイズ

これらの設定はユーザーの好みで上書き可能:

- ユーザーは異なるscreen_nameを指定可能
- ユーザーは異なるカテゴリパスをリクエスト可能
- ユーザーは今日以外の日付を指定可能

## 必要なツール

1. **esa_get_team_members** - 現在のユーザーのscreen_nameを取得
2. **list_gcal_events** - カレンダーイベントを取得
3. **esa_create_post** - esaに投稿を作成
4. **view** - テンプレートファイルを読み取り（必要な場合）

## 実行フロー

### ステップ0: 現在のユーザーのscreen_nameを取得

```markdown
esa_get_team_membersを以下で使用:
- teamName: 'gaji'（またはユーザー指定）
- "myself": trueのメンバーを検索
- そのメンバーからscreen_nameを抽出
```

これにより、日報が正しいユーザーのscreen_nameで自動的に作成されます。

### ステップ1: カレンダーイベントを取得

```markdown
list_gcal_eventsを以下で使用:
- calendar_id: 'primary'
- time_min: 対象日の開始（00:00:00）
- time_max: 対象日の終了（23:59:59）
- time_zone: ユーザーのタイムゾーン（デフォルト: Asia/Tokyo）
```

**イベントフィルタリングルール:**

- 勤務スタイルイベントを除外（例: 「自宅勤務」、「Home office」）
- 「チーム朝会」をMTG振り返りセクションから除外（ただし今日やったことには含める）
- 他のすべてのカレンダーイベントを含める

### ステップ2: ユーザー入力を収集

**まずテンプレートファイルを読み取り**、どのセクションを記入する必要があるかを理解します。

`view /mnt/skills/user/esa-daily-report/assets/default.md`を使用して:

1. ユーザー入力が必要なすべてのセクションを特定（プレースホルダー`-`または空のコンテンツを持つもの）
2. テンプレートに基づいて収集する情報を動的に決定

**対話戦略:**

- **オプションA（推奨）**: 質問を完全にスキップし、カレンダーイベントのみでレポートを作成。ユーザーは他のセクションを記入するためにesaのWIP投稿を編集可能。
- **オプションB**: ユーザーが積極的な場合、簡単に質問: 「他に記載したい情報はありますか？（学んだこと、リードとしての動き、MTG振り返りなど）」提供されたもののみを収集。

**ガイドライン:**

- まずカレンダーイベントを表示し、ユーザーが今日何があったかを知る
- 各セクションを個別に尋ねない - これは時間がかかる
- 空のセクションは完全に許容され、予想される
- ユーザーは後でesaのWIP投稿をいつでも編集可能
- ロボット的ではなく、会話的でサポート的に

**デフォルト動作**: カレンダーイベントと他のコンテンツ用の空のセクションでレポートを作成。これはユーザーの時間を尊重しつつ、後で記入できる構造化された開始点を提供します。

### ステップ3: レポートを構造化

**重要: レポートを構造化する前に、必ず`/mnt/skills/user/esa-daily-report/assets/default.md`からテンプレートを読み取ってください。**

`view`ツールを使用してテンプレートファイルを読み取ります:

```bash
view /mnt/skills/user/esa-daily-report/assets/default.md
```

テンプレートは日報の正確な構造とセクションを定義します。レポートコンテンツを構築する際は、このテンプレート構造に従い、プレースホルダーテキストを以下で置き換えます:

- 「今日やったこと」セクションのカレンダーイベント
- 各セクションのユーザー入力
- 「MTG 振り返り」サブセクションの会議名

**テンプレート処理ルール:**

- **h1見出しを無視**（`# xxx`形式）テンプレートから - 最終レポートに含めない
- h2以下の見出し（`## xxx`、`### xxx`など）を含める - これらがレポート構造を定義
- これによりレポートに重複する最上位タイトルがないことを保証

**テンプレート構造をハードコードしない** - 一貫性を保証し、簡単なテンプレート更新を可能にするため、常にテンプレートファイルを参照。

### ステップ4: esa投稿を作成

```markdown
esa_create_postを以下で使用:
- teamName: 'gaji'（またはユーザー指定）
- name: 'Daily 振り返り YYYY.MM.DD : [screen_name] #reflection'
  例: 'Daily 振り返り 2025.10.22 : screen_name #reflection'
- bodyMd: [ステップ3からの構造化コンテンツ]
- category: 'report/daily-report/'
- wip: true
- tags: []（デフォルトではタグなし、ユーザー指定可能）
```

### ステップ5: 確認と共有

- レポートが作成されたことをユーザーに通知
- esa投稿のURLを共有
- 必要に応じて調整を提案

## 使用例

**ユーザー**: 「今日の日報を作成して」

**アシスタントフロー（デフォルト - 質問なし）**:

1. esa_get_team_membersから現在のユーザーのscreen_nameを取得
2. 今日のカレンダーイベントを取得
3. **`/mnt/skills/user/esa-daily-report/assets/default.md`からテンプレート構造を読み取り**
4. ユーザーにイベントを提示: 「今日はこれらの予定がありましたね: [イベントをリスト]」
5. カレンダーイベントと他のコンテンツ用の空のセクションでesaに投稿を作成
6. 返信: 「日報を作成しました！[URL] カレンダーの予定を記載しています。他の内容は後からesaで編集できます。」

**代替フロー（ユーザーが事前に情報を提供した場合）**:
ユーザーが言った場合: 「今日の日報を作成して。今日はReactのuseEffectについて学んだよ。」

1. screen_nameを取得
2. カレンダーイベントを取得
3. テンプレートを読み取り
4. ユーザーが「今日学んだこと」コンテンツを提供したことを認識
5. カレンダーイベントと提供された情報の両方で投稿を作成
6. 返信: 「日報を作成しました！[URL]」

## 日付処理

**現在の日付検出:**

- 常にシステムから現在の日付/時刻を取得
- ユーザーが他を指定しない限り今日がデフォルト
- 「昨日の日報」、「yesterday's report」などのフレーズをサポート

**タイムゾーン:**

- デフォルト: Asia/Tokyo（JST）
- タイムゾーンオフセット付きのRFC3339として時刻をフォーマット
- 例: 2025-10-22T00:00:00+09:00

## エラーハンドリング

### 一般的な問題

1. **カレンダーイベントが見つからない**
   - ユーザー入力のみでレポートを作成
   - 注記: 「カレンダーには予定がありませんでした」

2. **esa APIエラー**
   - ユーザーに明確に通知
   - 再試行またはパラメータ調整を提案

3. **ユーザー入力の欠落**
   - これは通常で予想されるケース
   - カレンダーイベントと空のセクションで投稿を作成
   - ユーザーは後でesaで詳細を記入

### 検証

- `esa_get_teams`を使用してチーム名が存在するか確認
- 日付形式が有効であることを検証
- カテゴリパスがesa規約に従うことを保証（先頭のスラッシュなし）

## 最良の結果を得るためのヒント

### Claudeの場合

- **常にまずテンプレートファイルを読み取る**: ステップ3の開始時に`view /mnt/skills/user/esa-daily-report/assets/default.md`を使用
- テンプレート構造をハードコードしない - 常に実際のテンプレートファイルを参照
- 会話的に - 機械的にデータを収集するだけではない
- ユーザーのコミュニケーションスタイルに適応
- ユーザーがすべての情報を事前に提供した場合、段階的な質問をスキップ
- WIP投稿は後で編集できることを覚えておく
- テンプレートファイルはレポート構造の単一の真実の情報源

### ユーザーの場合

- 情報は任意の順序で提供するか、完全にスキップ可能
- スキルはカレンダーイベントでWIP投稿を作成 - 他のセクションは後でesaで記入可能
- すぐに特定の情報を含めたい場合は、言及するだけ: 「今日の日報を作成。Reactを学んだよ。」
- 空のセクションは予想され、通常 - 後で都合の良い時に投稿を編集
- 作成後にesaでWIP投稿を編集可能
- 異なる日付やカテゴリが必要な場合は指定
