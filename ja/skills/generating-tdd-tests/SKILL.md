# TDDテスト生成 - 包括的テスト駆動開発ガイド

## 概要

このスキルは以下を組み合わせたテスト駆動開発（TDD）の体系的なガイダンスを提供：

1. **RGRCサイクル** - t_wadaに触発されたRed-Green-Refactor-Commitワークフロー
2. **Baby Steps** - 2分サイクルでの段階的開発
3. **体系的なテスト設計** - 同値分割、境界値分析、デシジョンテーブル
4. **テスト生成** - SOWテスト計画から実行可能なテストケースへ

## コア概念

### 1. Baby Steps - 基盤

**"各ステップで可能な限り最小の変更を行う"** - t_wada

**なぜ重要か:**

- 即座のエラー局所化（バグは最後の2分間の変更にある）
- 継続的な動作状態（常にグリーンまで数秒）
- 迅速なフィードバックループ
- 小さな成功による自信

**2分サイクル:**

1. 最小の失敗するテストを書く（30秒）
2. 最小限でパスさせる（1分）
3. テストを実行（10秒）
4. 小さなリファクタリング（30秒）
5. グリーンならコミット（20秒）

### 2. RGRCサイクル

**目標**: "Clean code that works" (動作するきれいなコード) - Ron Jeffries

#### Redフェーズ

失敗するテストを書く → 正しい失敗理由を検証

#### Greenフェーズ

パスする最小限のコード → "You can sin"（素早く/汚くてOK）

#### Refactorフェーズ

SOLID/DRYを適用 → 構造を改善 → テストをグリーンに保つ

#### Commitフェーズ

すべてのチェックがパス → 安定した状態を保存

### 3. 体系的なテスト設計

**品質 > 量** - より少ない設計されたテストがより多くのバグを捉える

#### 同値分割

同じ動作をする入力をグループ化し、各グループから1つテスト：

```typescript
// 年齢検証: <18（無効）、18-120（有効）、>120（無効）
test('partition 1: rejects 17', () => ...)
test('partition 2: accepts 30', () => ...)
test('partition 3: rejects 121', () => ...)
```

#### 境界値分析

エッジをテスト [min-1, min, max, max+1]：

```typescript
test('boundary: rejects 17 (18-1)', () => ...)
test('boundary: accepts 18 (min)', () => ...)
test('boundary: accepts 120 (max)', () => ...)
test('boundary: rejects 121 (120+1)', () => ...)
```

#### デシジョンテーブルテスティング

複数条件の複雑なロジック：

```typescript
// isLoggedIn × isPremium × isActive → canAccess
// デシジョンテーブルを使用してすべての組み合わせを体系的にカバー
```

## ベストプラクティス

### やるべきこと ✅

- コードの前にテストを書く
- ステップを小さく保つ（Baby Steps）
- テストを頻繁に実行
- グリーン時にコミット
- 体系的なテスト設計を使用
- RefactorフェーズでSOLID/DRYを適用、その前ではない

### やってはいけないこと ❌

- Redフェーズをスキップ
- パスする前に複数のテストを書く
- グリーンテストなしでリファクタリング
- 失敗するテストでコミット
- ランダムなテスト選択
- Greenフェーズでの複雑なアーキテクチャ

## カバレッジ目標

業界推奨目標：

- **C0（ステートメント）**：最低80%
- **C1（ブランチ）**：最低70%

理由：コスト対効果のバランス、クリティカルパスに焦点

## TDDを使用しない場合

以下の場合はスキップ：

- プロトタイプ（使い捨てコード）
- 外部API統合（モックを使用）
- シンプルな一度限りのスクリプト
- UI実験（最初にビジュアル反復）

## テスト優先度マトリクス

すべてをテストする必要はない。影響度で優先順位をつける。

### ✅ 必須テスト（Priority 1）

- **ビジネスロジック**: 計算、バリデーション、状態遷移
- **Service/Repository層**: 単純なCRUD以外のデータ操作
- **クリティカルパス**: 課金、認証、データ永続化
- **エッジケース**: 境界値、null/undefined、空配列

### 🟡 状況次第（Priority 2）

- **ユーティリティ関数**: 複雑なもののみ
- **カスタムフック**: 状態管理ロジック部分
- **変換処理**: 複雑なマッピング/整形

### ❌ テストしない

- 単純なプロパティアクセス/getter
- UIレイアウト/スタイリング
- 外部ライブラリの動作確認
- プロトタイプ/実験コード
- 単純なCRUD（フレームワーク提供機能）
- 設定ファイルの読み込み

**判断基準**: 「このロジックが壊れたら、ユーザーに影響があるか？」

## 命名規則（Jest/Vitest）

一貫した命名でテストの意図を明確にする。

### 推奨パターン

```typescript
describe('[対象クラス/関数名]', () => {
  describe('[メソッド名/シナリオ]', () => {
    it('[条件]のとき、[期待結果]', () => {
      // Arrange - Act - Assert
    })
  })
})
```

### 具体例

```typescript
describe('PriceCalculator', () => {
  describe('calculateTotal', () => {
    it('空配列のとき、0を返す', () => {
      expect(calculator.calculateTotal([])).toBe(0)
    })

    it('割引コード適用時、割引後の合計を返す', () => {
      const items = [{ price: 1000, quantity: 2 }]
      expect(calculator.calculateTotal(items, 'SAVE10')).toBe(1800)
    })

    it('税込み計算時、正しい税額を含む', () => {
      const items = [{ price: 1000, quantity: 1 }]
      expect(calculator.calculateTotal(items, null, { includeTax: true })).toBe(1100)
    })
  })
})
```

### 命名ガイドライン

| 要素 | 良い例 | 悪い例 |
|------|--------|--------|
| 条件 | `空配列のとき` | `test1` |
| 期待結果 | `0を返す` | `正しく動作する` |
| コンテキスト | `割引適用時` | `discount` |

**ヒント**: テスト名をドキュメントとして機能させる記述的な名前を使用
