# サマリー: ワークフロー改善 (レビュー用サマリー)

## 🎯 目的 (1-2文)

Spec駆動開発のベストプラクティスを現行Claude Codeワークフローに適用し、コンテキスト混乱の解消とプロンプト品質の向上を図る。

## 📋 変更概要

| フェーズ | 説明 | 期間 |
|---------|------|------|
| フェーズ1 | Golden Master導入 | 1日 |
| フェーズ2 | コンテキスト最小化 | 1日 |
| フェーズ3 | /think分割（/sow + /spec） | 2日 |
| フェーズ4 | 指示の簡素化 | 2日以上 |

## 📁 影響範囲

**変更対象ファイル:**

- `~/.claude/commands/think.md` → 薄いオーケストレーター化
- `~/.claude/commands/sow.md` → 新規ファイル
- `~/.claude/commands/spec.md` → 新規ファイル

**影響を受けるコンポーネント:**

- `/think`、`/code` ワークフロー
- sow-spec-reviewer エージェント

## ❓ 議論ポイント

1. **コンテキスト削減の閾値**: 1,500行を目標としているが、最適値は検証が必要
2. **テンプレート配置場所**: templates/ vs golden-masters/ ディレクトリ
3. **段階的移行**: 既存ユーザーへの影響を最小化

## ⚠️ リスク

| リスク | 影響度 | 軽減策 |
|--------|--------|--------|
| 既存ワークフローの破壊 | 高 | 段階的移行、互換性レイヤー |
| コンテキスト削減による機能低下 | 中 | 必須原則の慎重な選定 |

## ✅ 主要な受け入れ基準

- [ ] golden-masters/に3つ以上の理想的な例
- [ ] /code参照コンテキストが1,500行以下
- [ ] 各コマンドが200行以下
- [ ] 既存ワークフローとの互換性維持
