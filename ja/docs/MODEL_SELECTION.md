# エージェントのモデル選択ガイドライン

## 概要

このドキュメントは、異なるエージェントタイプに対して`opus`と`sonnet`モデルを選択する基準を定義します。

## モデルの特性

### Opus

- **強み**: 複雑な推論、深い分析、オーケストレーション、多要素の意思決定
- **最適な用途**: ニュアンスのある判断、複数の懸念事項の調整、根本原因分析が必要なタスク
- **リソース使用**: 高い計算コスト、遅いレスポンス時間

### Sonnet

- **強み**: パターン認識、構造化分析、ルールベースの評価、一貫したフォーマット
- **最適な用途**: コードレビュー、パターンマッチング、コンプライアンスチェック、構造化レポート
- **リソース使用**: 低い計算コスト、速いレスポンス時間

## 選択基準

### Opusを使用する場合

1. **オーケストレーションが必要**: 複数のサブエージェントを管理し、その結果を統合
2. **深い分析が必要**: 根本原因の発見、複雑な関係の理解
3. **メタレベルの推論**: 他の分析や設定の品質を分析
4. **複雑なトレードオフ**: 複数の競合する懸念事項のバランス

### Sonnetを使用する場合

1. **パターンベースのレビュー**: 確立されたパターンやルールに対するチェック
2. **構造化分析**: 事前定義されたチェックリストや基準に従う
3. **単一フォーカスのレビュー**: 特定の側面を評価（例：セキュリティ、パフォーマンス）
4. **コード品質チェック**: 可読性、スタイル、ベストプラクティス

## 現在の割り当て

### Opusエージェント（3つ）

- `review-orchestrator`: すべてのレビューエージェントを調整、結果を統合
- `root-cause-reviewer`: 根本的な問題を見つけるための深い分析
- `subagent-reviewer`: エージェント設定のメタ分析

### Sonnetエージェント（10個）

- `accessibility-reviewer`: パターンベースのWCAG準拠
- `design-pattern-reviewer`: Reactパターン認識
- `document-reviewer`: 構造化されたドキュメントレビュー
- `performance-reviewer`: パフォーマンスアンチパターン検出
- `progressive-enhancer`: CSS/HTMLパターンの提案
- `readability-reviewer`: コード可読性パターン
- `security-reviewer`: セキュリティ脆弱性パターン
- `structure-reviewer`: コード構成パターン
- `testability-reviewer`: テストパターン評価
- `type-safety-reviewer`: TypeScriptパターンチェック

## デシジョンツリー

```markdown
タスクは主に何に関するものですか？
├─ 複数の分析を調整する？ → Opus
├─ 複雑な問題の根本原因を見つける？ → Opus
├─ 他の評価を評価する？ → Opus
└─ 既知のパターンに対してチェックする？ → Sonnet
   ├─ コード品質パターン？ → Sonnet
   ├─ セキュリティパターン？ → Sonnet
   ├─ パフォーマンスパターン？ → Sonnet
   └─ アクセシビリティパターン？ → Sonnet
```
