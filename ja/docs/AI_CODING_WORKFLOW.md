# AIコーディングワークフロー - 制御可能なAI開発の実践ガイド

> 「最高のAIコードとは、制御でき、理解でき、信頼できるコードである」

## はじめに

AIコーディングツールはソフトウェア開発を変革しましたが、体系的なワークフローがなければ、解決するよりも多くの問題を生み出す可能性があります。本ガイドは、業界の実践者や確立されたエンジニアリング原則から得た洞察を統合し、実践的でチーム対応可能なワークフローを提供します。

### なぜこのワークフローが重要なのか

**課題**: AIは理解できる速度よりも速くコードを生成できます。規律がなければ、前例のない速度で技術的負債が蓄積します。

**解決策**: AIの強みを最大化しながらリスクを最小化する構造化されたワークフロー：

- **制御可能性**: AIが自律的に決定する範囲を最小化
- **品質**: コード品質に一切妥協しない
- **明示性**: すべての要件と制約を明示的にする

### 対象読者

- AIコーディングツールを導入する開発チーム
- 体系的なAIワークフローを求める個人開発者
- AIコーディング標準を確立するエンジニアリングリーダー

## 3つの柱

### 1. 制御可能性 - LLM責務の最小化

**原則**: AIが決定する範囲が少ないほど、制御できる。

古典的なソフトウェアエンジニアリングに着想を得て、関心を分離します：

```markdown
あなたが制御（決定論的）:
├─ アーキテクチャの決定
├─ タスクの分解
├─ 成功基準
├─ セキュリティポリシー
└─ 品質閾値

AIが支援（非決定論的）:
├─ 制約内での設計探索
├─ パターンに従ったコード生成
├─ 明示的な基準によるトレードオフ分析
└─ 明確な目標を持つリファクタリング提案
```

**実践的な適用**:

- 決定論的操作にはツール（MCP、コマンド）を使用
- AIがコードを生成する前に明示的な制約を定義
- 計画（あなた）と実装（AI）を分離

### 2. 品質 - 100%レビュー基準

**原則**: 「1%でも違和感があれば、完全に破棄する」

このゼロトレランスアプローチは技術的負債の蓄積を防ぎます：

**レビューマインドセット**:

```markdown
✅ 許容可能:
- 完全に理解できるコード
- あなたの基準に合致するパターン
- 要件と一致するソリューション

❌ 許容不可:
- 完全には理解できない「十分に良い」コード
- 明確な正当化のない巧妙なソリューション
- 明示的要件からの逸脱
```

**破棄基準**:

- 同じエラーが5回発生 → アプローチを変更
- 理解度が95%未満のまま → 明確化を求める
- 修正のたびに複雑さが増す → よりシンプルな設計で再開

### 3. 明示性 - 言語化が設計を生む

**原則**: 「言語で明示された要件は、自然に良い設計を生み出す」

求めるものを明示的に述べる行為が、明確さを強制します：

**明示化プロセス**:

1. **分解**: 曖昧な要求を具体的な要件に分解
2. **境界の定義**: スコープ内/外を明示
3. **成功基準の設定**: 測定可能な完了条件を定義

**例**:

```markdown
❌ 曖昧: 「ユーザー認証を追加」

✅ 明示的:
- 要件:
  * メール/パスワード認証
  * 24時間有効期限のJWTトークン
  * パスワード要件: 8文字以上、大文字1つ、数字1つ
  * 15分間ロックアウト前に最大5回のログイン試行

- 境界:
  * スコープ内: 登録、ログイン、ログアウト、パスワードリセット
  * スコープ外: OAuth、2FA、ソーシャルログイン

- 成功基準:
  * すべてのテストが合格（ユニット＋統合）
  * セキュリティ監査で重大な脆弱性なし
  * ログインフローが500ms未満で完了
```

この明示化は、SOLID原則を意識的に適用しなくても、自然にSOLID設計を生み出します。

## フェーズ別ワークフロー

### フェーズ1: 計画 - 明示的要件

**目標**: コードを書く前に95%以上の理解

**ステップ**:

1. **ユーザー向け要件を記述**

   ```markdown
   [ユーザータイプ]として、[目標]したい。なぜなら[利益]だから。
   ```

2. **技術タスクに分解**
   - 単一責任のユニットに分解
   - 各タスクは独立してテスト可能
   - タスクあたり最大1〜4時間と見積もり

3. **受け入れ基準を定義**

   ```markdown
   完了条件:
   - [ ] 仕様通りに機能が動作
   - [ ] すべてのテストが合格（カバレッジ>80%）
   - [ ] リンターエラーなし
   - [ ] パフォーマンスがSLAを満たす
   - [ ] セキュリティレビュー合格
   ```

4. **明示的な制約を設定**

   ```markdown
   必須:
   - 既存の認証フレームワークを使用
   - リポジトリのエラーハンドリングパターンに従う
   - 承認なしに新しい依存関係を追加しない

   禁止:
   - 暗黙のフォールバックを使用
   - 正当化なしにオプショナルパラメータを追加
   - アドホックなソリューションを実装
   ```

**ツール**:

- `/think` コマンドで構造化された計画
- TodoWriteでタスク追跡
- 複雑な機能のためのSOWドキュメント

### フェーズ2: 実装 - 制御された生成

**目標**: 完全に理解できるコードのみを生成

**RGRCサイクル** (Red-Green-Refactor-Commit):

```markdown
1. Red: 失敗するテストを書く
   ├─ 正しい理由で失敗することを確認
   └─ テストは具体的で焦点を絞る

2. Green: 最小限の実装
   ├─ テストを通すのに十分なだけ
   ├─ AIが生成可能、あなたは理解必須
   └─ 不明な点はすべて質問

3. Refactor: 動作を変えずに改善
   ├─ 重複を削除
   ├─ パターンを明示的に適用
   └─ テストをグリーンに保つ

4. Commit: 進捗を保存
   └─ 完了したサイクルごとにアトミックコミット
```

**実装ルール**:

```markdown
厳格な強制:
1. 暗黙のフォールバック禁止 - すべてのエラーハンドリングは明示的
2. 文書化された正当化なしにオプショナルパラメータ禁止
3. アドホックな修正禁止 - 常に根本原因に対処
4. リファクタリング中の動作変更禁止
```

**AIとの対話パターン**:

```typescript
// ❌ 曖昧なリクエスト
"エラーハンドリングを追加"

// ✅ 明示的な指示
"以下のエラーハンドリングを追加:
1. NetworkErrorをキャッチして { success: false, error: 'network' } を返す
2. ValidationErrorをキャッチして { success: false, error: 'validation', details } を返す
3. 他のすべてのエラーは再スロー
4. すべてのエラーをリクエストコンテキストと共にerrorLoggerに記録
暗黙のフォールバック禁止、デフォルトケース禁止"
```

**検証チェックポイント**:

- 各AI応答後: 「これを100%理解しているか？」
- コード受け入れ前: 「これは私の明示的要件に合致しているか？」
- レビュー中: 「自分でこれを書くか？」

**ツール**:

- `/code` コマンドでTDD実装
- `/fix` で小さく理解された変更
- 検証のための信頼度マーカー (✓/→/?)

### フェーズ3: テスト - 包括的検証

**目標**: コードが動作し、適切に失敗することを証明

**テスト設計アプローチ**:

1. **同値分割**

   ```markdown
   年齢検証の場合（18-120）:
   - 無効: <18
   - 有効: 18-120
   - 無効: >120
   ```

2. **境界値分析**

   ```markdown
   テスト: [最小-1, 最小, 最大, 最大+1]
   値: [17, 18, 120, 121]
   ```

3. **デシジョンテーブルテスト** (複雑なロジックの場合)

   ```markdown
   | 条件A | 条件B | 結果 |
   |-------|-------|------|
   | true  | true  | X    |
   | true  | false | Y    |
   | false | true  | Z    |
   | false | false | W    |
   ```

**カバレッジ目標**:

- ステートメントカバレッジ (C0): 最低80%
- ブランチカバレッジ (C1): 最低70%
- メトリクスではなく、クリティカルパスにカバレッジを集中

**テスト実行**:

```bash
# テストの検出と実行
npm test  # またはyarn test、flutter testなど

# すべてのチェックが合格することを確認
✓ テスト合格（終了コード0）
✓ リンター合格（エラー0）
✓ ビルド成功
✓ クリティカルパスの手動スモークテスト
```

**失敗時の対応**:

```markdown
テストが失敗した場合:
1. 即座に修正 - 進めない
2. 同じアプローチで最大5回再試行
3. 5回失敗後: アプローチを完全に変更

リンターが失敗した場合:
1. すべてのエラーを即座に修正
2. 合計<5の場合は警告に対処
3. 文書化された正当化なしに抑制しない
```

**ツール**:

- `/test` コマンドで包括的テスト
- package.json/READMEからのテスト検出
- 指数バックオフによる自動再試行

### フェーズ4: レビュー - 100%基準

**目標**: 最高のエンジニアに誇って見せられるコードのみを出荷

**3レベルレビュー**:

**レベル1: セルフレビュー**

```markdown
問うべき質問:
- [ ] すべての行を理解しているか？
- [ ] 説明が必要な「巧妙な」コードはあるか？
- [ ] すべてのエッジケースが明示的に処理されているか？
- [ ] これを自分で書くか？
- [ ] これは最もシンプルなソリューションか？
```

**レベル2: 原則レビュー**

```markdown
SOLID原則:
- [ ] 単一責任: 各クラス/関数は1つのことを行う
- [ ] 開放閉鎖: 変更ではなく構成を通じて拡張
- [ ] リスコフの置換: サブタイプは基底タイプが使える場所で動作
- [ ] インターフェース分離: 未使用の依存関係なし
- [ ] 依存性逆転: 抽象に依存

可読性:
- [ ] 関数名が「何を」記述（「どのように」ではなく）
- [ ] 関数あたり5パラメータ以下
- [ ] ネストレベル3以下
- [ ] 新しい開発者が1分未満で理解

シンプルさ（オッカムの剃刀）:
- [ ] 必要最小限の依存関係
- [ ] 最低の循環的複雑度
- [ ] 「念のため」の機能なし
```

**レベル3: 自動レビュー**

```markdown
以下のために `/review` コマンドを使用:
- セキュリティ脆弱性
- パフォーマンスボトルネック
- アクセシビリティ問題
- 型安全性の問題
- 可読性の懸念
```

**破棄の判断**:

破棄して再開すべき場合:

```markdown
即座に破棄:
- 明確化の試み後も理解度<95%
- 同じエラーパターンが5回以上
- 複雑さが減少ではなく増加
- 「1%違和感がある」直感

おそらく破棄:
- シンプルなタスクで変更回数>5
- 説明のための広範なコメントが必要
- 自分のコードレビューで不合格
```

## 失敗パターンと解決策

### パターン1: 蓄積する複雑さ

**症状**: 各AI変更でコードがより複雑になる

**なぜ起こるか**: AIはシンプルさではなくテスト合格を最適化

**解決策**:

```markdown
1. 複雑さが増加したら即座に停止
2. 望ましいシンプルさを明示的に述べる
3. よりシンプルな制約で再開
4. タスク分解が間違っていないか検討
```

**例**:

```typescript
// 3回の変更後、以下のようになる:
function processUser(user, options = {}, flags = {}, context = {}) {
  // 50行の分岐ロジック
}

// 破棄して以下で再開:
"processUserを作成:
1. userとprocessOptionsのみを受け取る
2. processOptionsにすべての設定を含める（ドキュメントなしのデフォルト禁止）
3. 単一責任: 検証してから変換
4. 最大15行
5. 必要に応じてサブ関数を抽出"
```

### パターン2: 謎のブラックボックス

**症状**: コードは動作するが、どのようにかを説明できない

**なぜ起こるか**: 完全な理解なしにAI出力を受け入れる

**解決策**:

```markdown
1. 不明確な行すべてをAIに説明させる
2. 説明が不明確な場合、よりシンプルな代替案を要求
3. 2回の説明試行後も不明確なら破棄
4. 思い出して: あなたが午前2時にこのコードを保守する
```

### パターン3: スコープクリープ

**症状**: 実装にリクエストしていない機能が含まれる

**なぜ起こるか**: AIが「役立つ」追加を推論

**解決策**:

```markdown
1. すべてのリクエストで「スコープ内/外」を明示
2. 明示的スコープ外のコードはすべて拒否
3. 要件に追加: 「指定されたもののみを実装。追加機能なし」
```

### パターン4: 暗黙のフォールバック

**症状**: エラーケースに `|| defaultValue` やキャッチオールハンドラー

**なぜ起こるか**: AIが明示性より「堅牢性」を優先

**解決策**:

```markdown
要件に追加:
「エラーハンドリングルール:
- 暗黙のフォールバック禁止（|| defaultValue禁止）
- すべてのエラーケースは明示的であるべき
- 未知のエラーはデフォルトを返さず、スローする必須
- 各エラーハンドリング選択の理論的根拠を文書化」
```

## クイックリファレンス - 日常のワークフロー

### タスク開始時

```markdown
1. 要件を明示的に述べる（95%以上の理解）
2. 受け入れ基準を定義
3. 制約を設定（必須/禁止）
4. <4時間のタスクに分解
5. コマンド選択: /think → /code または /fix（シンプルな変更）
```

### 実装中

```markdown
1. RGRCに従う: Red → Green → Refactor → Commit
2. 各AI応答後: 「100%理解しているか？」
3. 不明確な点は即座に質問
4. アプローチ変更前に最大5回再試行
```

### コード受け入れ前

```markdown
1. セルフレビュー: これを自分で書くか？
2. 原則レビュー: SOLID、可読、シンプル？
3. テスト合格 + リンタークリーン + ビルド成功
4. 1%でも違和感 → 破棄
```

### 破棄トリガー

```markdown
以下の場合破棄:
- 同じエラー5回
- 明確化後も理解度<95%
- 複雑さが増加
- 「これをシニアエンジニアに見せるか？」= いいえ
```

## 成功のメトリクス

### チームレベルの指標

```markdown
良い兆候:
- 破棄率10-20%（選択的になっている）
- 平均タスク完了時間<4時間
- コードレビューコメントが減少
- 本番バグが増加していない

警告兆候:
- 破棄率<5%（多くを受け入れすぎ）
- 平均タスク>8時間（タスクが大きすぎ）
- レビューで「これを理解できない」
- 技術的負債が蓄積
```

### 個人レベルの指標

```markdown
成功している場合:
- AI生成コードのどの行も説明できる
- 変更試行がめったに3を超えない
- テストが初回実行で合格>60%
- 本番へのデプロイに自信を持てる

調整が必要な場合:
- 頻繁に「動くけど理由がわからない」と言う
- 5回再試行制限にしばしば達する
- 実装より修正に時間を費やす
- AI生成コードのデプロイに躊躇
```

## まとめ

AIコーディングツールは強力な加速器ですが、適切に制御された場合にのみです。このワークフローは、以下によって速度と品質のバランスを取ります：

1. **AI自律性の最小化** - あなたが決定、AIが制約内で実行
2. **品質基準の維持** - 100%レビュー、不明確なコードへのゼロトレランス
3. **明示的な言語化** - 明確に述べられた要件が良い設計を生む

### 核心的なマインドセット

```markdown
すべてのAI対話:
├─ 明示的に述べる
├─ 完全に検証する
├─ 冷酷に破棄する
└─ 自信を持って出荷する
```

### 次のステップ

1. **チーム採用**: このワークフローをチームと共有
2. **カスタマイズ**: チームのコンテキストにルールを適応
3. **測定**: 破棄率、完了時間、バグ率を追跡
4. **反復**: チームに有効なものに基づいて改善
