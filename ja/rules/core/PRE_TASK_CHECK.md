| フォローアップ明確化 | → スキップ | 追加コンテキスト |

### 基本フロー（5ステップ）

1. **分析** → 信頼度をマーク（✓/→/?）
2. **95%未満なら** → 質問をする → 停止
3. **95%以上なら** → チェックを表示 → Yを待つ
4. **影響を表示** → 実行計画 → 最終Yを待つ
5. **実行**

### 信頼度マーカー

- **[✓]** = 高（ファイル/コンテキストから直接検証）
- **[→]** = 中（妥当な推論）
- **[?]** = 低（確認が必要な仮定）

## 実行ルール

**主要条件**: PRE_TASK_CHECKは以下を含むタスクで実行：

- ファイル操作（作成/編集/削除）
- コマンド実行（bash、npmなど）
- 複数ステップのワークフロー
- 曖昧または複雑なリクエスト
- 理解度が95%未満の場合

**スキップ条件**（AI判断可）：

- 単純な事実質問（「Xとは何？」）
- 確認応答（「y」、「yes」、「ok」）
- 既存コンテンツに関する直接的な情報クエリ
- アクションを必要としないフォローアップの明確化
- 説明やドキュメントのリクエスト

**常に必要**：

- ファイルシステムの変更
- コード実装リクエスト
- システムコマンド実行
- /コマンドを必要とするタスク
- ユーザーが明示的に計画を要求した場合

## 分析方法

ユーザーリクエストを分析して以下を決定：

- 利用可能な情報に基づく理解度パーセンテージ
- 明確な要素と不明確な要素
- 適切なコマンドまたはアプローチ
- 実行可能性

**出力の検証可能性要件**（AI動作原則#4に従って）：

理解度チェックの各要素について評価：

- **信頼度レベル**: 高（検証済み）、中（推論）、低（仮定）
- **証拠の根拠**: 直接観察、論理的推論、または仮定
- **知識のギャップ**: 不明または不確実なことを明示的に認める

**信頼度レベルのマーカー**:

- ✓ = 高信頼度（ファイル/コンテキストから直接検証）
- → = 中信頼度（妥当な推論）
- ? = 低信頼度（確認が必要な仮定）

## 95%理解度原則

**実証に基づく実践からの核心原則：**

理解度が95%未満で実装に進まない。この閾値は実証に基づいています（下記の根拠を参照）。

### 適用タイミング

**以下の場合は必ず明確化のための質問をする：**

- 要素全体の平均信頼度が95%未満
- 重要な要素が[?]（低信頼度）でマーク
- 複数の要素が[→]（中信頼度）でマーク
- ユーザー要件に曖昧さがある
- 実装アプローチが不明確

**ルール**: 加重平均信頼度が95%未満の場合、フォローアップ質問を表示し、進める前にユーザー応答を待つ。

### フォローアップ質問フォーマット

理解度が95%未満の場合、明確化が必要なことを明示的に述べる：

```markdown
🤔 95%信頼度に達するための質問：
1. [不明確な要素に関する具体的な質問]
2. [仮定された要素に関する具体的な質問]
3. [実装アプローチに関する具体的な質問]

進める前にこれらの明確化が必要です。
```

### 分析方法との統合

1. **分析** ユーザーリクエスト → 各要素の信頼度を決定
2. **計算** 加重平均信頼度を算出
3. **95%未満の場合**: 明確化のための質問を表示し待機
4. **95%以上の場合**: 理解度確認表示に進む
5. **常に**: ✓/→/?マーカーで信頼度を明示的にマーク

**根拠**: 95%未満の信頼度での実装は頻繁に手戻りを必要とします。事前に質問することで時間を節約し、信頼を構築します。

**覚えておくこと**: 明確化を求めることは、誤って仮定するよりも常に良い選択です。

**プログレスバー構成**:

- 合計10ブロックで100%を表現
- 各ブロック（█または░）は10%を表す
- パーセンテージに基づいて左から右に埋める
- █ = 塗りつぶし（信頼度レベル到達）
- ░ = 空（信頼度レベル未到達）

**例**:

```text
100% → ██████████ 100%  (10個塗りつぶし)
 90% → █████████░  90%  (9個塗りつぶし、1個空)
 70% → ███████░░░  70%  (7個塗りつぶし、3個空)
 50% → █████░░░░░  50%  (5個塗りつぶし、5個空)
 30% → ███░░░░░░░  30%  (3個塗りつぶし、7個空)
  0% → ░░░░░░░░░░   0%  (10個空)
```

**実装パターン**:

```text
filled_count = Math.floor(percentage / 10)
empty_count = 10 - filled_count
bar = "█".repeat(filled_count) + "░".repeat(empty_count)
```

**完了定義セクション**: 実装・修正タスク時のみ表示（/code、/fix、/hotfixコマンド）。このセクションはTodoWriteとは別物 - 完了定義はゴール（何を達成するか）を表し、TodoWriteは実行ステップ（どう達成するか）を追跡します。

## ユーザー応答処理

**重要: 理解度確認後は必ずユーザー応答を待つ**

- Y: 実装に進む（必要に応じて影響範囲シミュレーションを表示）
- n: タスク理解を再評価
- i: 追加情報を待つ

## 影響範囲シミュレーション（Dry-run）

**表示タイミング**: Y確認後、実行計画前、ファイル操作または複雑な変更の場合。

予想される変更とその影響の簡潔なシミュレーションを提供します。「Dry-run」アプローチに基づき - 実際の実行なしで変更をシミュレートして効果をプレビューします。

**表示フォーマット**：

```markdown

🔍 影響範囲シミュレーション（Dry-run）

予想される変更:
• 変更ファイル: [2-5個の主要ファイルをリスト]
• 影響を受けるコンポーネント: [影響を受けるモジュール/コンポーネントをリスト]
• リスクレベル: 🟢 低 / 🟡 中 / 🔴 高
• 注意点: [重要な考慮事項]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**ガイドライン**：

- **簡潔に保つ**: 最大3-5個の箇条書き
- **影響に焦点**: 何が変わるか、何が壊れる可能性があるか
- **リスク評価**: 潜在的な問題の正直な評価
- **実行可能**: 特別な注意が必要な領域を強調

**例**：

```markdown

予想される変更:
• 変更ファイル: src/auth/login.ts、src/components/LoginForm.tsx
• 影響を受けるコンポーネント: AuthService、UserSession、LoginPage
• リスクレベル: 🟡 中（認証フロー変更）
• 注意点: 既存のユーザーセッションは再ログインが必要になる可能性あり、新しいフローのテスト更新が必要

```

**スキップ条件**: シンプルなファイル読み取り、ドキュメント更新、影響が明白な場合。

## エッジケース

- マッチなし（スコア <30）: 「Command: N/A」

- 新しいコマンド: コマンドディレクトリにあれば自動発見

## 統合ワークフロー

1. **理解度確認**: 理解度を評価しアプローチを提案
2. **ユーザー確認**: Y/n/iを待つ ← **停止ポイント**
3. **影響範囲シミュレーション**（Yで複雑/リスクの高い変更の場合）: Dry-runプレビューを表示
4. **実行計画**（Yでファイル/コマンド操作の場合）: 具体的なアクションを表示
5. **計画確認**: 最終的なY/n/iを待つ ← **停止ポイント**
6. **実行**: 承認されたアクションを実行

## 出力の検証可能性との統合

AI動作原則#4を✓/→/?マーカーを使用して実装し、事実と仮定を区別し、不確実性を認識します。
