### 🔴 重要: 95%理解ルール

**信頼度<95%で進めない**。必ず最初に明確化の質問をする。

### 決定ツリー

```text
開始
  ↓
ファイル変更する？ ──YES──→ [チェック実行]
  ↓ NO
コマンド実行する？ ──YES──→ [チェック実行]
  ↓ NO
理解度<95%？ ──YES──→ [チェック実行]
  ↓ NO
複数ステップ？ ──YES──→ [チェック実行]
  ↓ NO
[チェックスキップ] ← 単純な質問、確認、または読み取り専用クエリ
```

### 決定マトリックス

| 条件 | アクション | 例 |
|-----------|--------|---------|
| **チェック実行** | | |
| ファイル変更 | → 実行 | CLAUDE.mdの作成/編集 |
| コマンド実行 | → 実行 | `npm test`の実行 |
| 理解度<95% | → 実行 | 曖昧なユーザーリクエスト |
| 複数ステップワークフロー | → 実行 | 機能実装 |
| **チェックスキップ** | | |
| 単純な質問 | → スキップ | "PRE_TASK_CHECKとは？" |
| 確認 | → スキップ | "yes", "ok", "y" |
| 読み取り専用クエリ | → スキップ | ファイル読み取り、Grep検索 |
| フォローアップの明確化 | → スキップ | 追加のコンテキスト |

### 基本フロー（5ステップ）

1. **分析** → 信頼度マーク（✓/→/?）
2. **<95%の場合** → 質問 → 停止
3. **≥95%の場合** → チェック表示 → Yを待つ
4. **影響表示** → 実行計画 → 最終Yを待つ
5. **実行**

### 信頼度マーカー

- **[✓]** = 高（ファイル/コンテキストから直接検証済み）
- **[→]** = 中（合理的推論）
- **[?]** = 低（確認が必要な仮定）

## 実行ルール

**主要**: PRE_TASK_CHECKは以下を含むタスクに対して実行されます：

- ファイル操作（作成/編集/削除）
- コマンド実行（bash、npmなど）
- 複数ステップワークフロー
- 次のいずれかの基準を満たすリクエスト：
  - **曖昧**: 明確な参照先のない代名詞（"それ"、"あれ"）または曖昧な量（"いくつか"、"少し"）を含む
  - **複雑**: 3つ以上の異なるアクションが必要、または5つ以上のファイルに影響、または2つ以上のサブシステムに関係
- 理解度が95%未満の場合

**スキップ条件**（AI判断可能）：

- 単純な事実確認の質問（"Xとは？"）
- 確認応答（"y"、"yes"、"ok"）
- 既存コンテンツに関する直接的な情報クエリ
- アクションを必要としないフォローアップの明確化
- 説明またはドキュメントの要求

**常に必須**：

- すべてのファイルシステム変更
- コード実装リクエスト
- システムコマンド実行
- /commandsを必要とするタスク
- ユーザーが明示的に計画を要求する場合

## 分析方法

ユーザーリクエストを分析して以下を決定：

- 利用可能な情報に基づく理解度のパーセンテージ
- 明確な要素と不明確な要素
- 次の優先基準に基づくコマンド選択：
  1. **タスクスコープ**: 単一ファイル = /fix、複数ファイル = /code、調査 = /research
  2. **変更タイプ**: バグ修正 = /fix、新機能 = /think → /code、緊急 = /hotfix
  3. **信頼度レベル**: <70% = /research優先、70-90% = /think、>90% = /code直接
- 実行可能性

**出力検証可能性要件**（AI操作原則#4による）：

理解度チェックの各要素について評価：

- **信頼度レベル**: 高（検証済み）、中（推論）、低（仮定）
- **証拠の根拠**: 直接観察、論理的推論、または仮定
- **知識のギャップ**: 不明または不確実なことを明示的に認める

**信頼度レベルのマーカー**：

- ✓ = 高信頼度（ファイル/コンテキストから直接検証済み）
- → = 中信頼度（合理的推論）
- ? = 低信頼度（確認が必要な仮定）

## 95%理解ルール

**証拠に基づく実践からのコア原則：**

95%未満の理解度で実装を進めない。この閾値は証拠に基づいています（下記の根拠を参照）。

### 適用タイミング

**以下の場合、常に明確化の質問をする：**

- 要素全体の平均信頼度が95%未満
- 重要な要素が[?]（低信頼度）でマークされている
- 複数の要素が[→]（中信頼度）でマークされている
- ユーザー要件に曖昧さがある
- 実装アプローチが次のいずれかの不明確な条件を満たす：
  - 分析後30秒以内に変更する特定のファイルを識別できない
  - 明確な選択基準のない複数の有効なアプローチが存在する
  - 必要な技術/フレームワークが明示的に述べられていない
  - 成功基準を測定可能な用語で定義できない

**ルール**: 加重平均信頼度が95%未満の場合、フォローアップの質問を表示し、進行前にユーザーの応答を待つ。

### フォローアップ質問フォーマット

理解度が95%未満の場合、明確化が必要なことを明示的に述べる：

```markdown
🤔 95%信頼度に到達するための質問:
1. [不明確な要素に関する具体的な質問]
2. [仮定された要素に関する具体的な質問]
3. [実装アプローチに関する具体的な質問]

進行前にこれらの明確化が必要です。
```

### 分析方法との統合

1. **分析** ユーザーリクエスト → 各要素の信頼度を決定
2. **計算** 加重平均信頼度
3. **<95%の場合**: 明確化の質問を表示して待つ
4. **≥95%の場合**: 理解度チェック表示に進む
5. **常に**: ✓/→/?マーカーで信頼度を明示的にマーク

**根拠**: 95%未満の信頼度での実装は頻繁に作り直しが必要になります。事前に質問することで時間を節約し、信頼を構築します。

**覚えておく**: 誤って仮定するより、明確化のために質問する方が常に良い。

**プログレスバー構築**：

- 100%を表すために合計10ブロックを使用
- 各ブロック（█または░）は10%を表す
- パーセンテージに基づいて左から右に塗りつぶす
- █ = 塗りつぶし（信頼度レベルに到達）
- ░ = 空（信頼度レベル未到達）

**例**：

```text
100% → ██████████ 100%  (10塗りつぶし)
 90% → █████████░  90%  (9塗りつぶし、1空)
 70% → ███████░░░  70%  (7塗りつぶし、3空)
 50% → █████░░░░░  50%  (5塗りつぶし、5空)
 30% → ███░░░░░░░  30%  (3塗りつぶし、7空)
  0% → ░░░░░░░░░░   0%  (10空)
```

**実装パターン**：

```text
filled_count = Math.floor(percentage / 10)
empty_count = 10 - filled_count
bar = "█".repeat(filled_count) + "░".repeat(empty_count)
```

**完了定義セクション**: 実装/修正タスク（/code、/fix、/hotfixコマンド）の場合のみ表示。このセクションはTodoWriteとは別 - 完了定義は目標を表し（何を達成するか）、TodoWriteは実行ステップを追跡します（どのように達成するか）。

**コマンドを提案するタイミング**: 理解度≥70%、技術的タスク

**ソース**: ~/.claude/commands/、.claude/commands/

## ユーザー応答処理

**重要: 理解度チェック表示後は常にユーザー応答を待つ**

- Y: 実装に進む（必要に応じて影響シミュレーション表示）

## Impact Simulation（Dry-run）

**表示タイミング**: Y確認後、実行計画前、ファイル操作または複雑な変更の場合。

予想される変更とその影響の簡潔なシミュレーションを提供。"Dry-run"アプローチに基づく - 実際の実行なしで変更をシミュレートして効果をプレビュー。

**表示フォーマット**：

```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

🔍 Impact Simulation (Dry-run)

予想される変更:
• 変更ファイル: [2-5の主要ファイルをリスト]
• 影響を受けるコンポーネント: [影響を受けるモジュール/コンポーネントをリスト]
• リスクレベル: 🟢 低 / 🟡 中 / 🔴 高
• 注意: [重要な考慮事項]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**ガイドライン**：

- **長さの制限**：
  - 箇条書き: 3-5項目（最大5を厳守）
  - 箇条書きあたりの文字数: 最大100文字
  - Impact Simulation全体: 最大300文字
- **影響に焦点**: 何が変わるか、何が壊れる可能性があるか
- **リスク評価**: 潜在的な問題の正直な評価
- **実行可能**: 特別な注意が必要な領域を強調

**例**：

```markdown
🔍 Impact Simulation (Dry-run)

予想される変更:
• 変更ファイル: src/auth/login.ts、src/components/LoginForm.tsx
• 影響を受けるコンポーネント: AuthService、UserSession、LoginPage
• リスクレベル: 🟡 中（認証フロー変更）
• 注意: 既存のユーザーセッションは再ログインが必要、新フローのテスト更新

```

**スキップ条件**（これらがFALSEの場合のみImpact Simulationを適用）：

- **単純なファイル読み取り**: 変更なしのReadツール
- **ドキュメント更新**: *.mdファイルのみ影響、コード変更なし
- **明らかな低リスク変更**: 単一ファイル、変更<10行、依存関係への影響なし

以下のいずれかが当てはまる場合、常にImpact Simulationを表示：

- コアコンフィグの変更（CLAUDE.md、PRE_TASK_CHECK.md、settings.json）
- 3つ以上のファイルを同時変更
- 認証、セキュリティ、またはデータ整合性ロジックへの影響

## エッジケース

## 統合ワークフロー

1. **理解度チェック**: 理解を評価しアプローチを提案
2. **ユーザー確認**: Y/n/iを待つ ← **停止ポイント**
3. **Impact Simulation**（Yかつ複雑/リスクのある変更の場合）: Dry-runプレビューを表示
4. **実行計画**（Yかつファイル/コマンド操作の場合）: 特定のアクションを表示
