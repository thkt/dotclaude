# 基本原則 - クイック判断質問

## 目的

すべての実装タスクのためのコンパクトで実行可能な判断質問。
詳細はPRINCIPLES_GUIDE.mdまたは個別の原則ファイルで利用可能。

## クイック判断質問

### オッカムの剃刀 / KISS

**「これを達成するよりシンプルな方法はあるか？」**

- 依存関係を最小限に（3+よりも0-2を優先）
- コード行数を少なく（関数あたり<50行を優先）
- 複雑さを低く（分岐<5を優先）

### DRY

**「知識や意図を重複させていないか？」**

- 同じビジネスロジックが複数の場所に？
- 設定値が繰り返されている？
- 三回ルール：3回見たらリファクタリング

### SOLID (SRP)

**「このクラス/モジュールは、変更する明確な単一の理由を持っているか？」**

- 単位あたり1つの責任
- 変更は局所化されるべき

### ミラーの法則

**「新しいチームメンバーは1分以内に理解できるか？」**

- 関数パラメータ ≤ 5
- クラスのpublicメソッド ≤ 7
- 条件分岐 ≤ 5

### YAGNI

**「これは今存在する実際の問題を解決しているか？」**

- 想像上の未来のために構築？ → しない
- "念のため"の柔軟性を追加？ → しない
- 測定なしの最適化？ → しない

## 決定フロー

```text
コードを書く前に:
1. 最もシンプルな解決策か？（オッカムの剃刀）
2. すでに存在するか？（DRY）
3. 単一の責任か？（SRP）
4. 理解可能か？（ミラーの法則）
5. 今必要か？（YAGNI）

いずれかの答えが"いいえ"→アプローチを再考
```

## 警告フラグ

- メソッドチェーン > 3レベル
- 1分で理解できない
- "念のため"の実装
- 複雑な解決策を最初に選択
