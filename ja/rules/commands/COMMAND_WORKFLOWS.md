# コマンドワークフロー

## クイックリファレンス

| パターン | ワークフロー | 使用タイミング |
|---------|----------|------|
| 緊急対応 | `/hotfix` | 本番環境クリティカル |
| クイック修正 | `/fix` | 小規模バグ、開発環境 |
| 調査 | `/research` → `/fix` | 原因不明 |
| 機能追加 | `/research` → `/think` → `/code` → `/test` → `/audit` → `/validate` | 新機能実装 |

## 選択マトリクス

| 基準 | 高優先度 | 中優先度 | 低優先度 |
|----------|---------------|-----------------|--------------|
| **環境** | 本番環境 → `/hotfix` | 開発環境 → `/fix` | 任意 → ワークフロー |
| **理解度** | ≥95% → 直接実行 | 70-94% → `/research` | <70% → 明確化 |
| **複雑度** | 複数ステップ → ワークフロー | 単一ファイル → `/fix` | 不明確 → `/think` |
| **緊急度** | クリティカル → `/hotfix` | 通常 → 標準 | 計画 → `/think` |

## タスク分析

ユーザー意図をコマンドにマッチング:

| ユーザー意図 | 分析 | 結果 |
|-------------|----------|--------|
| "Xが壊れている" | 調査が必要？ | はい → `/research` → `/fix` |
| "Y機能を追加" | 複数ステップ？ | はい → `/think` → `/code` → `/test` |
| "サイトがダウン" | 本番環境クリティカル？ | はい → `/hotfix` |
| "タイポ修正" | シンプルで明確？ | はい → `/fix` |
| "Zはどう動作する？" | 調査のみ | `/research`（実装なし） |

**主要要素**:

- **スコープ**: 単一ファイル vs 複数コンポーネント
- **コンテキスト**: 既知 vs 探索が必要
- **リスク**: 開発環境 vs 本番環境

## 発見プロセス

1. **コマンドスキャン**: `~/.claude/commands/`, `.claude/commands/`
2. **YAML読み取り**: メタデータ抽出
3. **意図マッチング**: 選択マトリクス使用
4. **スコア&ランク**: 理解度レベル、タスク複雑度、環境、緊急度

## エッジケース

| 状況 | アクション |
|-----------|--------|
| 曖昧な意図 | 理解確認で明確化を依頼 |
| コマンドマッチなし | `Command: N/A`を使用、直接実装に進む |
| 複数の有効なアプローチ | ユーザーに選択肢を提示 |
| 不明確な要件 | `/research`から開始 |
| 複雑な複数部分 | サブワークフローに分割 |
