# YAGNI - You Aren't Gonna Need It

**核心原則**: 実際に必要になるまで機能を追加しない

## 哲学

「You Aren't Gonna Need It」 - エクストリームプログラミングの核心プラクティスの一つ

原則はシンプルだが強力：

- 後で必要になると思う機能を**実装しない**
- 想像上の将来のユースケースのための抽象化を**構築しない**
- 測定していない問題を**最適化しない**
- 「念のため」の柔軟性を**追加しない**

## YAGNIが重要な理由

### 早すぎる機能のコスト

書くすべてのコード行にはコストがある：

- **保守負担**: コードはテスト、文書化、更新が必要
- **認知負荷**: コードが多い = 理解すべき複雑さが多い
- **機会費用**: YAGNI機能に費やす時間は実際のニーズに費やせない時間
- **間違った仮定**: 将来の予測は通常間違っている

### 洞察

> 「最高のコードは、まだ存在する必要のないコード」

ソフトウェア開発は：

- **今日の問題**を今日の知識で解決
- **明日の問題**を明日の知識で適応

## 成果優先の開発

**アーキテクチャではなく成果を出荷**

### 実装フェーズ

必要な時のみ各フェーズを追加し、この進行に従う：

1. **動くようにする** - 即座の問題を解決
   - 焦点: 機能性
   - 品質基準: 基本テストをパス
   - タイムライン: 数時間から数日

2. **回復力を持たせる** - エラーが発生したらエラー処理を追加
   - 焦点: 信頼性
   - 品質基準: エッジケースを処理
   - タイムライン: 本番使用が失敗モードを明らかにした後

3. **速くする** - 遅さが測定されたら最適化
   - 焦点: パフォーマンス
   - 品質基準: パフォーマンス要件を満たす
   - タイムライン: プロファイリングがボトルネックを示した後

4. **柔軟にする** - ユーザーが要求したらオプションを追加
   - 焦点: 拡張性
   - 品質基準: 実際のユースケースをサポート
   - タイムライン: 複数の類似リクエストの後

**重要**: エビデンスが要求するまでフェーズ2-4を行わない。

## 判断フレームワーク

コードを追加する前に、これらの質問をする：

### 4つの質問

1. **これは今存在する実際の問題を解決しているか？**
   - いいえなら: スキップ
   - はいなら: 続行

2. **これは実際に本番で失敗したか？**
   - いいえなら: まだエラー処理を追加しない
   - はいなら: ターゲットを絞った修正を追加

3. **ユーザーはこれについて不満を言ったか？**
   - いいえなら: 機能を追加しない
   - はいなら: まず実際のニーズを理解

4. **問題の測定されたエビデンスがあるか？**
   - いいえなら: 最適化しない
   - はいなら: プロファイル後、ボトルネックを修正

**ルール**: 質問1に「いいえ」と答えたら、止まる。進まない。

## 例

### ❌ YAGNIに違反

```typescript
// 誰も求めていない柔軟性を追加
class UserService {
  constructor(
    private cache: CacheStrategy,      // まだ不要
    private logger: LoggerStrategy,    // まだ不要
    private validator: ValidationStrategy,  // まだ不要
    private transformer: DataTransformer    // まだ不要
  ) {}

  // 将来の拡張性のためのプラグインシステム
  plugins: Plugin[] = []
  registerPlugin(plugin: Plugin) { }  // 誰も使っていない
}

// 「念のため」の抽象化
interface DatabaseAdapter {
  query(): Promise<Result>
}

class PostgresAdapter implements DatabaseAdapter { }
class MySQLAdapter implements DatabaseAdapter { }
class MongoAdapter implements DatabaseAdapter { }

// 現実: Postgresしか使っていない
```

**問題**:

- 必要な量の10倍のコード
- テストされていないコードパス
- 使われていない機能の保守負担
- 読者への認知オーバーヘッド

### ✅ YAGNIに従う

```typescript
// 実際のニーズを解決
class UserService {
  async getUser(id: string): Promise<User> {
    return db.query('SELECT * FROM users WHERE id = $1', [id])
  }
}

// 2つ目のデータベースが現れたら抽象化を追加
// プロファイリングでデータベースがボトルネックと示されたらキャッシュを追加
// 無効なデータがバグを引き起こしたらバリデーションを追加
```

**利点**:

- シンプルで直接的
- テストしやすい
- 理解しやすい
- ニーズが変わった時にリファクタリングしやすい

## 一般的なYAGNI違反

### 1. 早すぎる抽象化

```typescript
// ❌ YAGNI違反
interface PaymentProcessor {
  process(amount: number): Promise<Result>
}

class StripePaymentProcessor implements PaymentProcessor { }
// 他のプロセッサは存在しないし予定もない

// ✅ YAGNI準拠
async function processPayment(amount: number) {
  return stripe.charge(amount)
}
// 2つ目のプロセッサが実際に必要になったらインターフェースを追加
```

### 2. 早すぎる最適化

```typescript
// ❌ YAGNI違反
function getUserPosts(userId: string) {
  // 複雑なキャッシュ、ページネーション、プリフェッチ
  // 誰も速度について不満を言っていない
}

// ✅ YAGNI準拠
function getUserPosts(userId: string) {
  return db.query('SELECT * FROM posts WHERE user_id = $1', [userId])
}
// 実際のパフォーマンス問題を測定した後に最適化
```

### 3. 早すぎる設定

```typescript
// ❌ YAGNI違反
const config = {
  cache: {
    ttl: env.CACHE_TTL,
    max: env.CACHE_MAX,
    strategy: env.CACHE_STRATEGY,
    compression: env.CACHE_COMPRESSION
  },
  // さらに50の設定オプション
}

// ✅ YAGNI準拠
const CACHE_TTL = 3600  // バリエーションが必要になるまでハードコード
```

### 4. 汎用的な「再利用可能」コンポーネント

```typescript
// ❌ YAGNI違反
<DataGrid
  columns={columns}
  data={data}
  sorting={sorting}
  filtering={filtering}
  pagination={pagination}
  grouping={grouping}
  // 使われていない機能のためにさらに30のprops
/>

// ✅ YAGNI準拠
<UserTable users={users} />
// 実際のニーズのためのシンプルなコンポーネント
// 3つ目の類似テーブルが現れたら汎用化（3の法則）
```

## 複雑さを追加するタイミング

YAGNIは「決して先を考えるな」という意味ではない。以下の場合に複雑さを追加：

### 先に構築する正当な理由

1. **確認されたタイムラインを持つ既知の将来の要件**
   - 例: 契約書がフェーズ2でマルチテナンシーを指定
   - 注意: 構築前に本当に必要か確認

2. **後でリファクタリングコストが法外になる**
   - 例: 100万レコードに影響するデータベーススキーマ移行
   - 注意: 確かか？コストを測定

3. **前もって構築しなければならないコンプライアンス要件**
   - 例: GDPRデータ保持要件
   - 注意: 最小限の実行可能なコンプライアンスを構築

### テスト

自問：

- 「これが必要になるというエビデンスは何か？」
- 「後で追加するコスト vs 今のコストは？」
- 「推測しているか、実際の要件に対応しているか？」

推測しているなら、YAGNIは言う：構築するな。

## YAGNIマインドセット

### 核心原則

- **ハッピーパス**から始める
- **現実への対応**としてのみ複雑さを追加
- すべてのコード行は**負債**
- YAGNIは**デフォルトの立場**

### 開発フロー

```markdown
要件 → 最もシンプルな解決策 → 出荷 → 学習 → 反復

以下ではない：

要件 → すべての将来を予測 → 柔軟なシステムを構築 → 出荷 → 柔軟性を使わない
```

### チームプラクティス

これらのフレーズを正常化：

- 「想像する問題ではなく、持っている問題を解決しよう」
- 「必要になったら追加できる」
- 「これが必要だというエビデンスは何か？」
- 「シンプルに出荷し、フィードバックに基づいて反復」

## 他の原則との統合

### YAGNI + オッカムの剃刀

**オッカムの剃刀**: 最もシンプルな解決策を選ぶ
**YAGNI**: 必要ないものを構築しない

一緒に: **今日の問題だけを解決する**最もシンプルな解決策を選ぶ

### YAGNI + プログレッシブエンハンスメント

**プログレッシブエンハンスメント**: 最小から始め、徐々に強化
**YAGNI**: 想像上の将来のために構築しない

一緒に: 最小限の実行可能な機能を構築し、**実際の使用**に基づいて強化

### YAGNI + TDD

**TDD**: テストを書き、パスする最小限のコードを書く
**YAGNI**: テストされていない機能を実装しない

一緒に: テスト駆動開発は**自然にYAGNIを強制**

## 危険信号

YAGNIに違反している時に聞くフレーズ：

- 「これは後で必要になるかもしれない」
- 「念のため柔軟にしておこう」
- 「将来的に時間を節約できる」
- 「スケールに備えるべき」
- 「Xをサポートする必要があったら？」

**反論として**:

- 「必要になるというエビデンスは何か？」
- 「必要になったら追加できるのでは？」
- 「後で追加するコストは？」
- 「今その問題があるか？」
- 「Xが起きた時に解決しよう」

## 覚えておくこと

> 「将来を予測する最良の方法は、それを待つこと」 - 達人プログラマー

**重要な結論**:

- **今のため**に構築し、想像上の将来のためではなく
- **エビデンスへの対応**として複雑さを追加し、推測ではなく
- すべての機能には**コスト**がある（保守、複雑さ、機会）
- **シンプルさ**は制限ではなく機能
- **いつでも追加できる**（削除は稀にしかできない）
