# ドキュメント管理ルールとガイドライン

Claude Codeエコシステムにおける一貫性のある、整理されたドキュメントを維持するためのガイドライン。

## 核心原則

1. **完全性より明確性** - ドキュメントにもオッカムの剃刀を適用
2. **言語間の一貫性** - 英語/日本語の同期を維持
3. **論理的階層** - 原則と実践の明確な分離
4. **循環参照の禁止** - クリーンな依存グラフを維持

## ディレクトリ構造

### 配置基準

```markdown
/rules/
├── core/             # AI動作ルール（フック注入）
│   ├── AI_OPERATION_PRINCIPLES.md  # 安全性、権限、ワークフロー
│   ├── PRE_TASK_CHECK_COMPACT.md   # タスク検証（コンパクト版）
│   └── PRE_TASK_CHECK_VERBOSE.md   # タスク検証（詳細版）
│
├── reference/        # 原則・理論
│   └── DOCUMENTATION_RULES.md      # このファイル
│
├── development/      # 実践的適用
│   ├── TDD_RGRC.md                 # TDDサイクル
│   ├── PROGRESSIVE_ENHANCEMENT.md  # CSS-firstアプローチ
│   ├── READABLE_CODE.md            # コードの明瞭性
│   └── TIDYINGS.md                 # マイクロ改善
│
├── commands/         # コマンド固有ルール
│   └── COMMAND_WORKFLOWS.md        # ワークフロー選択
│
├── PRINCIPLES_GUIDE.md             # クイックリファレンス
└── PRINCIPLE_RELATIONSHIPS.md      # 原則間の関係

注記: コア原則（SOLID、DRY、オッカムの剃刀）は以下に配置:
→ /skills/applying-code-principles/SKILL.md
```

### 判断フレームワーク

新しいドキュメントを追加する際：

```markdown
基本原則または理論ですか？
  はい → /rules/guidelines/
  いいえ → 続行

実践的な手法や技法ですか？
  はい → /rules/development/
  いいえ → 続行

コマンドやツールですか？
  はい → /commands/
  いいえ → /docs/
```

## 参照管理

### 参照階層

```markdown
レベル1: CLAUDE.md（最上位設定）
  ├─→ レベル2: 基本原則（/rules/guidelines/）
  ├─→ レベル2: 開発実践（/rules/development/）
  └─→ レベル2: コマンド（/commands/）
      └─→ レベル3: ドキュメント間の相互参照
```

### 参照ルール

1. **最大深度**: 参照は3レベルまで
2. **循環回避**: A → B → A は禁止
3. **重要なものは直接**: 基本原則はCLAUDE.mdから直接参照
4. **関連をグループ化**: セクションヘッダーで参照をグループ化

### 参照フォーマット

```markdown
## 関連する原則

### 基本原則（同レベル）
- [@./SOLID.md] - 説明
- [@./DRY.md] - 説明

### 実践での適用
- [@../development/TDD_RGRC.md] - 説明
```

## 言語同期

### 二言語要件

**すべてのドキュメントファイルには以下が必須:**

- 英語版: `/path/to/FILE.md`
- 日本語版: `/ja/path/to/FILE.md`

### 同期チェックリスト

ドキュメント更新時：

```markdown
☐ 英語版を更新
☐ 日本語版を更新
☐ 構造が一致することを確認
☐ 参照パスを確認（相対パスが異なる！）
☐ セクションヘッダーの整合性を確認
☐ 相互参照が機能することをテスト
```

### パスの違い

英語版の参照：

```markdown
[@./SOLID.md]                    # 同じディレクトリ
[@../development/TDD_RGRC.md]    # 一つ上、developmentへ
```

日本語版の参照：

```markdown
[@./SOLID.md]                    # 同じディレクトリ
[@../development/TDD_RGRC.md]    # 一つ上、developmentへ
```

### 例外: 日本語のみのドキュメント

以下のファイルは日本語版のみ存在します（英語版なし）：

- `ja/docs/ARCHITECTURE.md` - システムアーキテクチャ概要
- `ja/docs/WORKFLOW_GUIDE.md` - クイックスタートガイド
- `ja/docs/DEVELOPMENT_WORKFLOW.md` - 実践的な開発ワークフロー
- `ja/commands/git/worktree-setup.md` - Git worktree設定ガイド

**理由**: これらのドキュメントは日本語話者向けであり、英語版の作成には大きな翻訳労力が必要で、明確な利益がありません。日本語ドキュメントツリー内で適切に参照されています。

### 例外: 英語のみのドキュメント

以下のファイルは英語版のみ存在します（日本語版なし）：

- `skills/automating-browser/` - ブラウザ自動化スキル（技術リファレンス）
- `skills/utilizing-cli-tools/` - CLIツールガイドスキル（技術リファレンス）
- `rules/core/PRE_TASK_CHECK_COMPACT.md` - フック注入されるタスク検証ルール
- `rules/core/PRE_TASK_CHECK_VERBOSE.md` - PRE_TASK_CHECKの詳細仕様

**理由**: これらは主にツールAPIやコマンド構文を文書化した技術リファレンススキルです。内容の大部分はコード例とツール名であり、翻訳を必要としません。PRE_TASK_CHECKファイルはフック経由で注入され、EN/JA両方のCLAUDE.mdから参照されます - 単一の英語ソースを維持することで同期の問題を防ぎます。日本語版の作成は比例した利益なしにメンテナンス負担を増やします。

## 更新手順

### 新しい原則の追加

1. **配置を評価**
   - 理論/原則 → `/rules/guidelines/`
   - 実践/手法 → `/rules/development/`

2. **両バージョンを作成**

   ```bash
   # 英語版
   /rules/guidelines/NEW_PRINCIPLE.md
   # 日本語版
   /ja/rules/guidelines/NEW_PRINCIPLE.md
   ```

3. **参照を追加**
   - 基本原則ならCLAUDE.mdに追加
   - 該当する場合はコマンドに追加
   - 関連ドキュメントと相互参照

4. **インデックスを更新**
   - 関連する場合はCOMMANDS.mdに追加
   - 概要ドキュメントを更新

### 既存ドキュメントの修正

1. **依存関係を確認**

   ```bash
   # すべての参照を検索
   grep -r "FILENAME" ~/.claude/
   ```

2. **両言語を更新**
   - 英語版に変更を加える
   - 日本語版に同じ変更を適用
   - 構造の同等性を維持

3. **参照を検証**
   - すべての受信参照が有効であることをテスト
   - 送信参照が正しいことを確認
   - リンク切れがないこと

### ファイルの移動

1. **移動を計画**

   ```markdown
   現在: /rules/guidelines/FILE.md
   移動先: /rules/development/FILE.md
   ```

2. **すべての参照を検索**

   ```bash
   grep -r "FILE\.md" ~/.claude/
   ```

3. **順序通りに実行**
   - 英語ファイルを移動
   - 日本語ファイルを移動
   - すべての参照を更新
   - リンク切れがないことを確認

## ドキュメント標準

### ファイル構造

```markdown
# タイトル - 明確で説明的

## 核心哲学/目的
なぜこれが存在するかの簡潔な説明

## 主要概念
主なアイデアを明確に説明

## 実践的適用
例と使用例

## 関連する原則
関連ドキュメントへのリンク
```

### 執筆スタイル

1. **ヘッダー**: 文頭大文字、全大文字は避ける
2. **例**: 実践的なコード例を含める
3. **比較**: ❌ 悪い例 vs ✅ 良い例を示す
4. **要約**: 箇条書きで要点をまとめる

### コード例

```typescript
// ❌ 避ける: 複雑な例を最初に
complexImplementation()

// ✅ 推奨: シンプルな例を最初に
simpleImplementation()

// その後、複雑な例への進行を示す
advancedImplementation()
```

```markdown
## 構造

## 参照
☐ すべてのリンクをテストし動作確認
☐ 循環依存がない

## 同期

☐ 日本語版が完成
☐ 構造が完全に一致
☐ パスに合わせて参照を調整

## 統合
☐ 適切なインデックスに追加

☐ 関連ドキュメントと相互参照
```

## 一般的なパターン

### 新しい基本原則の追加

```bash
1. /rules/guidelines/PRINCIPLE.mdを作成
2. /ja/rules/guidelines/PRINCIPLE.mdを作成
3. CLAUDE.mdに追加:
   - **Core principle**: Name → [@path]
4. 関連する実践を見つける
5. 相互参照を追加
```

### 新しい実践の追加

```bash
1. /rules/development/PRACTICE.mdを作成
2. /ja/rules/development/PRACTICE.mdを作成
3. 関連する原則から参照
4. 該当する場合はコマンドドキュメントに追加
```

### コマンドドキュメントの作成

```bash
1. /commands/COMMAND.mdを作成
2. /ja/commands/COMMAND.mdを作成
3. COMMANDS.mdインデックスに追加
4. 関連する原則/実践を参照
```

## 避けるべきアンチパターン

### ❌ これをしない

1. **単一言語の更新** - 常に英語/日本語両方を更新
2. **深いネスト** - 参照は最大3レベル
3. **孤立したドキュメント** - すべてが参照されるべき
4. **循環参照** - A → B → A
5. **誤った配置** - development/フォルダに原則
6. **壊れたリンク** - すべての参照をテスト
7. **不整合な構造** - 英語/日本語は一致すべき

### ✅ 代わりにこうする

1. **同期された更新** - 両言語を一緒に
2. **フラットな階層** - 可能な限り直接参照
3. **接続されたグラフ** - すべてのドキュメントがどこかで参照される
4. **ツリー構造** - 明確な親子関係
5. **正しい配置** - 判断フレームワークに従う
6. **検証済みリンク** - コミット前にテスト
7. **一致する構造** - 同一の組織

## メンテナンスタスク

### 定期レビュー

週次：

- 壊れた参照をチェック
- 英語/日本語の同期を確認
- 最近の変更の一貫性をレビュー

月次：

- ディレクトリ構造を評価
- 孤立したドキュメントをチェック
- 参照深度をレビュー

### リファクタリングのシグナル

以下の場合にリファクタリングを検討：

- 参照深度が3レベルを超える
- 循環依存が検出される
- 配置の混乱（原則 vs 実践）
- 英語/日本語の構造のずれ
- 過度の相互参照（ドキュメントあたり5以上）

## ツールとコマンド

### 便利なコマンド

```bash
# ファイルへのすべての参照を検索
grep -r "FILENAME" ~/.claude/

# 英語/日本語の構造一致を確認
diff <(ls /rules/) <(ls /ja/rules/)

# 孤立したドキュメントを検索
# （受信参照のないファイル）
for file in $(find /rules -name "*.md"); do
  basename=$(basename $file)
  count=$(grep -r "$basename" ~/.claude/ | wc -l)
  if [ $count -eq 0 ]; then echo "孤立: $file"; fi
done

# すべてのリンクを検証
# （シンプルなリンク切れチェッカー）
grep -r "@\[.*\]" ~/.claude/ | while read line; do
  # 各リンクを抽出して検証
done
```

## 進化のガイドライン

### 新しいカテゴリを作成する時

以下の場合に新しいディレクトリを作成：

- 同じカテゴリに5以上の関連ドキュメント
- 明確な概念的境界が現れる
- 配置の混乱が繰り返される

### カテゴリをマージする時

以下の場合にマージを検討：

- カテゴリがそれぞれ3未満のドキュメント
- 概念的境界が不明確
- 頻繁な配置の混乱

### バージョン管理

常に明確なメッセージでコミット：

```bash
# 良いコミットメッセージ
docs: referenceにオッカムの剃刀原則を追加
docs: TDD_RGRCの英語/日本語を同期
docs: TIDYINGSをreferenceからdevelopmentに移動
refactor: 原則ドキュメント構造を再編成

# 何が変わったか、なぜ変わったかを含める
```

## 覚えておくべきこと

> 「最良のドキュメントは最も包括的なものではなく、最も理解しやすいもの」 - ドキュメントに適用されたオッカムの剃刀

以下を保つ：

- **シンプル** - 理解しやすい
- **整理** - 見つけやすい
- **同期** - 言語間で一貫性
- **接続** - 適切に参照される
- **維持** - 定期的に更新される
