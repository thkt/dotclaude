# AI支援開発

**デフォルトマインドセット**: AIは生産性ツール、人間が意思決定者

## 核心哲学

**「AIが生成し、人間が検証する」** - AI支援テストケース生成の研究に基づく

AIツールは強力なアシスタントですが、人間の判断、ドメイン知識、コードレビューを置き換えることはできません。目標は、人間の監督による品質維持しながら、AIを活用して生産性を向上させることです。

## 主要原則

### 1. AIはツールであり、代替品ではない

```markdown
人間の役割:
- ✅ 最終決定を行う
- ✅ AI出力をレビュー・洗練させる
- ✅ コードの「なぜ」を理解する
- ✅ ビジネスロジックの正確性を保証

AIの役割:
- ✅ 定型コードを生成
- ✅ 実装を提案
- ✅ テストの骨組みを作成
- ✅ パターンを識別
```

**覚えておくこと**: AIはあなたのビジネスコンテキストを理解していません - あなたが理解しているのです。

### 2. 品質 = プロンプト品質

研究結果に基づくと、AIの出力品質は指示の品質に大きく依存します：

```typescript
// ❌ 曖昧なプロンプト
「ユーザーサービスを作成して」

// ✅ コンテキスト付きの具体的なプロンプト
「UserRepositoryの既存パターンに従ってユーザーサービスを作成してください。
- データベースには依存性注入を使用
- AAAテストパターンに従う
- スタイルはtests/services/の既存テストを参照」
```

**重要な洞察**: 指示が具体的であるほど、出力が良くなります。

### 3. 過去実績 > 抽象的ルール

**最も効果的なアプローチ**: 既存コードを例として参照

```bash
# AIにコード生成を依頼する前に:
1. 類似の既存実装を見つける
2. それらをプロンプトの例として共有
3. AIに同じパターンに従うよう依頼
```

**なぜ機能するか**:

- 具体例 > 理論的ガイドライン
- コードベースとの一貫性を保証
- 大幅な修正の必要性を削減

**研究的根拠**: 過去実績を参照として使用すると、最も実用的で即座に使えるAI生成コードが得られることが研究で示されています。

### 4. 人間レビューは交渉不可

```markdown
AI生成コードのワークフロー:

1. AIがコードを生成
2. ✅ 人間が以下をレビュー:
   - ビジネスロジックの正確性
   - エッジケースの処理
   - セキュリティへの影響
   - パフォーマンスの考慮事項
3. 人間が洗練・承認
4. コードが本番環境へ

ステップ2-3をスキップしない。
```

**重要ポイント**: AI生成コードは下書きであり、最終製品ではありません。

## 実践的な適用

### テスト生成

```bash
# 効果的なAI支援テスト作成:

1. 同じモジュールの既存テストを見つける
   grep -r "describe" tests/[module]/

2. 例を含めてAIにプロンプト:
   「このパターンに従ってcalculateDiscount()のテストを生成してください:
   [既存コードベースからテスト例を貼り付け]

   計画のテストケース:
   - 有効な割引計算
   - エッジケース: ゼロ購入」

3. 生成されたテストをレビュー:
   - アサーションがビジネスロジックと一致するか検証
   - モックセットアップがプロジェクトスタイルに従うか確認
   - 過剰テストがないか確認
```

### コード実装

```typescript
// ❌ AIの提案を盲目的に受け入れない
// AIが生成:
function processOrder(order) {
  // ... 複雑な実装
}

// ✅ レビュー・洗練
// あなたが検証:
// - これは特定のビジネスルールを処理するか？
// - エッジケースがカバーされているか？
// - エラー処理は適切か？
// - コーディング標準に従っているか？
```

## 警告サイン

### AIへの過度な依存

- ❌ レビューなしでAI出力を受け入れる
- ❌ 生成されたコードを理解していない
- ❌ 手動テストをスキップ
- ❌ AIを絶対視する

### AIの活用不足

- ❌ すべての定型コードを手動で書く
- ❌ 探索にAIを使用しない
- ❌ AI提案を完全に無視
- ❌ AIに十分なコンテキストを提供しない

## ベストプラクティス

### 1. 豊富なコンテキストを提供

```markdown
良いAIプロンプトテンプレート:

「タスク: [具体的タスク]

コンテキスト:
- 既存パターン: [リンクまたは例を貼り付け]
- ビジネスルール: [具体的要件]
- エッジケース: [既知のシナリオ]
- テストフレームワーク: [Jest/Vitest/等]

制約:
- [特定パターン]に従う
- 最大複雑度: [シンプル/中程度]
- 必須処理: [特定ケース]」
```

### 2. 反復的洗練

```bash
1. 初期版を生成
2. レビューして問題を特定
3. 具体的フィードバックでプロンプトを改善
4. 改善版を生成
5. 満足するまで繰り返す
```

### 3. 出力から学ぶ

```typescript
// AIが良いコードを生成した時:
- ✅ なぜ機能したか分析
- ✅ それを生み出したプロンプトを記録
- ✅ 成功パターンを再利用

// AIが悪いコードを生成した時:
- ✅ プロンプトに何が欠けていたか特定
- ✅ 将来の参考のためにギャップを記録
- ✅ プロンプトエンジニアリングを改善
```

## ガードレール

**厳格に禁止**:

- レビューなしでAIコードをコミット
- セキュリティ重要なコードをエキスパートレビューなしでAI使用
- AIの建築的決定を盲目的に受け入れる
- AI生成コードのテストをスキップ

**明示的に要求**:

- すべてのAI出力の人間レビュー
- AI生成コードのテスト
- 使用前に生成コードを理解
- AI支援部分の文書化

## 覚えておくこと

> 「最良のAI支援開発は、AIのスピードと人間の知恵を組み合わせる」

重要なポイント:

- **AIが加速**し、人間が品質を保証
- **過去の例**が抽象的指示に勝る
- **レビューは必須**、オプションではない
- **コードはあなたのもの**、AIのものではない

AIはあなたのアシスタントであり、代替品ではありません。賢く使いましょう。
