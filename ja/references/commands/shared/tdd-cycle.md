# TDDサイクル実装詳細

Red-Green-Refactor-Commitサイクルの詳細な実装ガイダンス。

## 目的

RGRCサイクルの各フェーズの具体的なステップと終了基準を提供。

TDD原則の基本的な理解:

## リアルタイムフィードバック付き拡張RGRCサイクル

### 1. 🔴 Redフェーズ

**信頼度目標**: 0.9

```bash
# 特定のテストを実行
npm test -- --testNamePattern="[現在のテスト]" | grep -E "FAIL|PASS"
```

**ステップ**:

1. 明確な意図を持つ失敗するテストを書く（または生成されたテストを使用）
2. 失敗理由が期待に一致することを確認
3. テストアサーションで理解を文書化
4. テストが実際に何かをテストしていることを確認

**終了基準**:

- ✅ テストが期待通りの理由で失敗
- ✅ 失敗メッセージが明確
- ✅ テストが1つの動作にフォーカス

**よくある間違い**:

- ❌ テストがすぐに合格（何もテストしていない）
- ❌ 失敗理由が不明確
- ❌ テストが広すぎる

### 2. 🟢 Greenフェーズ

**信頼度目標**: 0.7

```bash
# 高速フィードバックのためのウォッチモード
npm test -- --watch --testNamePattern="[現在のテスト]"
```

**ステップ**:

1. 合格するための**最小限**の実装を書く
2. クイックソリューションでOK（まだ最適化しない）
3. 形式より機能に集中
4. テストを通過するならハードコーディングもOK

**終了基準**:

- ✅ テストが一貫して合格
- ✅ 他のテストが壊れていない
- ✅ 実装が最小限

**よくある間違い**:

- ❌ 過剰実装（テストされていない機能の追加）
- ❌ 早すぎる最適化
- ❌ 既存テストの破壊

**例**:

```typescript
// ✅ 良いGreenフェーズ - 最小限
function add(a, b) {
  return a + b; // 合格するのに十分
}

// ❌ 悪いGreenフェーズ - 過剰実装
function add(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new Error('無効な入力'); // まだテストされていない!
  }
  return parseFloat((a + b).toFixed(2)); // 過剰設計!
}
```

### 3. 🔵 Refactorフェーズ

**信頼度目標**: 0.95

```bash
# すべてのテストがまだ合格することを確認
npm test | tail -5 | grep -E "Passing|Failing"
```

**ステップ**:

1. SOLID原則を適用
2. 重複を削除（DRY）
3. 命名と構造を改善
4. パターンが出現したら抽象化を抽出（3回以上）

**終了基準**:

- ✅ すべてのテストがグリーン
- ✅ コードがクリーンで読みやすい
- ✅ 重複なし
- ✅ 名前が明確

**よくある間違い**:

- ❌ 動作の変更（構造の改善のみであるべき）
- ❌ リファクタリング中にテストを壊す
- ❌ 早すぎる抽象化（3回目の使用前）

**リファクタリングチェックリスト**:

- [ ] 変数/関数名が説明的
- [ ] 関数が小さい（20行未満）
- [ ] 重複なし（DRY適用）
- [ ] 単一責任（各関数が1つのことをする）
- [ ] すべてのテストがまだ合格

### 4. ✅ Commitフェーズ

**信頼度目標**: 1.0

**コミット前チェック**:

```bash
# すべてのチェックを並列実行
npm run lint && npm test && npm run type-check
```

**ステップ**:

1. すべての品質チェックが合格することを確認
2. カバレッジが維持/改善されていることを確認
3. 最後にもう一度変更をレビュー
4. 意味のあるコミットメッセージを作成

**終了基準**:

- ✅ すべてのテストが合格
- ✅ リントが合格（0エラー）
- ✅ 型チェックが合格
- ✅ カバレッジ ≥ 以前
- ✅ 変更をレビュー済み

**コミットメッセージフォーマット**:

```bash
# 良いコミットメッセージ
feat: ユーザーバリデーションを追加
fix: 負の割引を防止
refactor: メールバリデーションを抽出

# 悪いコミットメッセージ
wip
stuff
update
```

## サイクルタイミングガイドライン

| フェーズ | 目標時間 | 最大時間 |
| --- | --- | --- |
| Red | 30-60秒 | 2分 |
| Green | 1-2分 | 5分 |
| Refactor | 30-90秒 | 3分 |
| Commit | 20-30秒 | 1分 |
| **合計** | **約2分** | **10分** |

**最大時間を超えた場合**:

- Red: テストが複雑すぎる、分割する
- Green: 実装が野心的すぎる、簡素化する
- Refactor: 後に保存、そのままコミット
- Commit: オプションチェックをスキップ、今すぐコミット

### 信頼度スコアリング

各フェーズには目標信頼度レベルがあります:

- **0.9** (Red): テストが要件を捉えていると高い信頼度
- **0.7** (Green): 実装が動作すると中程度の信頼度
- **0.95** (Refactor): コードがクリーンだと非常に高い信頼度
- **1.0** (Commit): コミット準備ができていると絶対的な信頼度

低い信頼度 → 前のフェーズを再訪

## 統合ポイント

### 機能開発（`/code`）との統合

- Phase 0: spec.mdからスキップされたテストを生成
- Phase 1-4: 各テストのRGRCサイクル
- インタラクティブ: ユーザーが各テストの有効化を確認

### バグ修正（`/fix`）との統合

- Phase 1.5: 回帰テストを書く（Red）
- Phase 2: 修正を実装（Green）
- Phase 3: クリーンアップ（Refactor）
- Phase 3.5: 追加テストを生成

## 成功のヒント

### テストを小さく保つ

```typescript
// ✅ 小さく、フォーカスしたテスト
it('空配列の場合0を返す', () => {
  expect(sum([])).toBe(0)
})

// ❌ 大きく、複数の動作をテスト
it('すべてのエッジケースを処理', () => {
  expect(sum([])).toBe(0)
  expect(sum([1])).toBe(1)
  expect(sum([1, 2, 3])).toBe(6)
  expect(() => sum(null)).toThrow()
})
```

### 誘惑に抵抗

- Greenで「もう1つの機能だけ」追加しない
- Refactorを「今回だけ」スキップしない
- テストを実行せずにコミットしない

### プロセスを信頼

TDDは最初は遅く感じるが:

- デバッグ時間 → ほぼゼロ
- リファクタリングの自信 → 最大
- コード品質 → 一貫して高い
