# 根本原因分析（Phase 1）

症状だけでなく、バグの真の原因を見つける。

## 目的

修正前に5 Whys手法とExploreエージェントを使用して根本原因を特定。

## 動的コンテキスト

### 最近の変更

```bash
!`git diff HEAD~1 --stat | head -10`
```

バグを導入した可能性のある最近のコミットをレビュー。

### テストステータス

```bash
!`npm run || yarn run || pnpm run || echo "No package manager"`
```

現在のテストスイートのステータスを確認。

### Spec参照（オプション）

`/think` からの spec.md が存在する場合:

- `.claude/workspace/planning/**/spec.md` を確認
- FR-xxx とテストシナリオを参照として使用
- バグが既存の要件を破っているか確認

## Phase 1: 根本原因分析

素早いコンテキスト取得のためにExploreエージェントを使用（30秒）:

```typescript
Task({
  subagent_type: "Explore",
  thoroughness: "quick",
  description: "バグコンテキストの探索",
  prompt: `
    バグ: "${bugDescription}"
    検索: 関連ファイル、依存関係、最近のコミット
    5 Whys適用: 症状ではなく根本原因を特定
    返却: [✓/→/?] マーカー付きの発見事項
  `
})
```

### 主要な質問（5 Whys）

1. **なぜこれが起こったのか?**
   - 直接的な原因は何か?

2. **なぜその状態が存在したのか?**
   - 何がこれを発生させたのか?

3. **なぜこれがキャッチされなかったのか?**
   - プロセスの何がこれを見逃したのか?

4. **なぜこのパターンがここにあるのか?**
   - これは孤立した問題か体系的か?

5. **なぜこのようにしているのか?**
   - 設計に本質的な欠陥があるか?

### 分析出力

特定と分類:

- **症状**: ユーザーが見るもの
- **近接原因**: 直接失敗したもの
- **根本原因**: なぜ失敗したか
- **範囲**: 影響を受けるファイルとコンポーネント

### パターン認識

これが何かを判断:

- ✅ **孤立した問題**: 特定条件での一回限りのバグ
- ⚠️ **パターン問題**: 複数の場所で同じ問題
- 🚨 **体系的問題**: リファクタリングが必要な設計上の欠陥

### 信頼度マーカー

分析全体で使用:

- **[✓]** 高 (>0.8) - コード/ファイルから直接確認
- **[→]** 中 (0.5-0.8) - 証拠からの合理的な推論
- **[?]** 低 (<0.5) - 検証が必要な仮定

### 分析例

```markdown
🔍 根本原因分析

**症状**:
- [✓] 割引が価格を超えるとユーザーに負の合計が表示される

**近接原因**:
- [✓] calculateTotalがチェックなしでprice - discountを返す

**根本原因**:
- [✓] 価格を超える割引に対するバリデーションがない
- [→] 要件がこのエッジケースを指定していなかった
- [?] 他の割引計算にも同じ問題があるかもしれない

**範囲**:
- [✓] ファイル: src/utils/pricing.ts:42
- [→] 類似パターン: src/checkout/discount.ts
- [?] 影響する可能性: src/reports/sales.ts

**パターン**: ⚠️ 潜在的なパターン - 他の割引ロジックを確認する必要あり
**信頼度**: 0.85
```

## 分析をスキップする場合

- **些細な修正**: タイポ、明らかに欠けているチェック
- **高信頼度 (>0.95)**: 原因がすぐに明確
- **時間的制約**: 既知の原因を持つ本番ホットフィックス

これらの場合は、Phase 1.5（回帰テスト）に直接進む。

## エスカレーションする場合

根本原因分析で以下が明らかになった場合:

- **信頼度 < 0.7**: より深い調査のため `/research` に切り替え
- **体系的問題**: 適切な設計のため `/think` → `/code` に切り替え
- **複数コンポーネント**: 複雑さがクイックフィックスの範囲を超える

## 統合ポイント

- **次**: Phase 1.5（回帰テスト）
- **エスカレーション先**: `/research`（低信頼度）または `/think`（体系的）
- **参照**: Spec.md（利用可能な場合）

## 出力フォーマット

```markdown
📋 根本原因分析完了

🐛 バグ: [説明]
🎯 根本原因: [特定した原因] [信頼度: 0.XX]
📁 影響: [ファイル]
⚠️ パターン: [孤立/パターン/体系的]

次: 回帰テストを書く
```
