# /commit - Gitコミットメッセージジェネレーター

## 目的

ステージング済みの変更を分析し、Conventional Commits仕様に従った適切なコミットメッセージを生成します。

## 動的コンテキスト分析

### ステージング済み変更

```bash
!`git diff --staged --stat`
```

### コミット予定のファイル

```bash
!`git status --short`
```

### 詳細な差分サマリー

```bash
`git diff --staged --numstat`
```

### 変更されたファイルタイプ

```bash
`git diff --staged --name-only`
```

### 最近のコミットスタイル（一貫性のため）

```bash
!`git log --oneline -10`
```

## コミットメッセージ生成プロセス

### 1. 変更タイプの検出

変更を分析してコミットタイプを判定：

| タイプ | 説明 | トリガーパターン |
|-------|------|-----------------|
| `feat` | 新機能 | 新しいファイル、新しい関数、新しいコンポーネント |
| `fix` | バグ修正 | エラー処理、検証修正、訂正 |
| `docs` | ドキュメント | .mdファイル、コメント、README更新 |
| `style` | フォーマット | 空白、フォーマット、セミコロンの欠落 |
| `refactor` | コード再構築 | 名前変更、移動、関数の抽出 |
| `perf` | パフォーマンス | 最適化、キャッシング、アルゴリズム改善 |
| `test` | テスト | テストファイル、テストの追加/変更 |
| `chore` | メンテナンス | 依存関係、設定、ビルドスクリプト |
| `ci` | CI/CD | GitHub Actions、CI設定ファイル |
| `build` | ビルドシステム | Webpack、npmスクリプト、ビルドツール |
| `revert` | コミット取り消し | 以前の変更を元に戻す |

### 2. スコープ検出

```bash
# 主要コンポーネント/モジュールを検出
`git diff --staged --name-only`
```

```bash
# 破壊的変更の存在確認（大規模なリファクタリング）
`git diff --staged --stat`
```

### 3. メッセージコンポーネント

#### 件名行の分析

以下に基づいて簡潔な件名を生成：

- 実行された主要アクション
- 影響を受ける主要ファイル/コンポーネント
- ユーザー向けの影響

#### 本文の生成（複雑な変更用）

5ファイル以上または100行以上の変更の場合：

- 動機を説明
- 主要な変更をリスト
- 破壊的変更を記載

#### フッター要素

- 破壊的変更：`BREAKING CHANGE: 説明`
- イシュー参照：`Closes #123`、`Fixes #456`
- 共同作成者：`Co-authored-by: name <email>`

### 複雑な変更

```markdown
## 📝 複雑な変更のコミットメッセージ

### 構造化フォーマット

```text

[動機/コンテキスト]
[何が変更されたか]
[なぜこのアプローチか]

[破壊的変更（もしあれば）]
[イシュー参照]

### 例

```text
refactor(api): サービスレイヤーアーキテクチャを再構築

動機：
以前のサービスレイヤーには循環依存関係とモジュール間の
密結合があり、テストとメンテナンスが困難でした。

変更内容：
- すべてのサービスのインターフェースを抽出
- 依存性注入を実装
- ビジネスロジックをデータアクセスから分離
- サービスファクトリパターンを追加

このアプローチにより、テスタビリティが向上し、後方互換性を
維持しながら単体テストでのモッキングが容易になります。

BREAKING CHANGE: サービスコンストラクタは明示的な依存関係が必要になりました
Closes #456, #457
```

## 規約ガイドライン

### 件名行のルール

1. **72文字以内に制限**
2. **命令形を使用**（"added"や"adds"ではなく"add"）
3. **タイプの後の最初の文字を大文字にしない**
4. **末尾にピリオドを付けない**
5. **具体的だが簡潔に**

### 良い例

- ✅ `feat(auth): パスワードリセット機能を追加`
- ✅ `fix(api): ユーザーエンドポイントのタイムアウトを解決`
- ✅ `docs(readme): インストール手順を更新`
- ✅ `perf(search): データベースクエリを最適化`

### 悪い例

- ❌ `バグを修正`（タイプなし、曖昧すぎる）
- ❌ `feat: 新機能を追加。`（大文字、ピリオド）
- ❌ `コードを更新`（タイプなし、具体的でない）
- ❌ `FEAT(AUTH): ログインを追加`（すべて大文字）

## ワークフローとの統合

以下とシームレスに動作：

- `/branch` - 最初にブランチを作成
- `/pr` - コミット後にPR説明を生成
- `/test` - コミット前にテストがパスすることを確認

## コンテキスト認識の提案

### バグ修正用

強調事項：

- 何が壊れていたか
- どのように修正されたか
- ユーザーへの影響

### 機能用

ハイライト：

- 新しい機能
- ユーザーの利点
- 実装アプローチ

### リファクタリング用

説明内容：

- なぜリファクタリングが必要だったか
- 何が改善されたか
- 後方互換性
