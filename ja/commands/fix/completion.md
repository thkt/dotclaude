# 完了とレポート

`/fix` コマンドの完了定義と出力フォーマット。

## 完了定義

**信頼度目標**: 全体 ≥0.9

バグ修正が完了するのは、以下のすべての基準を満たした場合：

### 必須基準（必ず合格）

- ✅ **根本原因を特定**（症状だけでなく）
  - 5 Whys 手法を適用
  - 信頼度マーカー: [✓]

- ✅ **最小複雑性の解決策**
  - オッカムの剃刀を適用
  - 不要な抽象化なし
  - コードが読みやすい

- ✅ **すべての関連テストが合格**
  - 回帰テストが合格
  - 既存テストが壊れていない
  - 終了コード: 0

- ✅ **新しいリントエラーなし**
  - `npm run lint` が合格
  - 0エラー（警告5未満は許容）

- ✅ **回帰が検出されない**
  - 手動スポットチェック実施
  - 関連機能が動作
  - 新しいバグが導入されていない

### オプション基準（推奨）

- ⭐ **追加テストを生成**（Phase 3.5）
  - エッジケースをカバー
  - 統合テストを追加
  - カバレッジが向上

- ⭐ **ドキュメントを更新**
  - コメントで非自明な修正を説明
  - 動作変更時にREADMEを更新
  - 変更履歴エントリを追加

### エスカレーション基準

| 信頼度 | フェーズ | アクション |
| --- | --- | --- |
| <0.5 | 根本原因 | → `/research` を即座に実行 |
| 0.5-0.7 | 任意 | → ユーザーに確認: 続行 or エスカレーション? |
| 0.7-0.8 | 実装 | 防御的アプローチで続行 |
| >0.8 | 任意 | 通常通り続行 |

### エスカレーションコマンド

```bash
# 調査用
/research "なぜ [バグ] が発生するのか?"

# 計画用
/think "[コンポーネント] を適切な [機能] で再設計"

# 完全な実装用
/code "TDDで [機能] を実装"
```

## 適用した原則

どの原則が修正を導いたかを文書化:

### よく使用される原則

- **オッカムの剃刀**: 最もシンプルな解決策を選択
  - 例: "複雑なバリデーションの代わりに Math.max を使用"

- **TIDYINGS**: 関連コードのみに触れる
  - 例: "calculateTotal のみを修正、関連コードはそのまま"

- **段階的強化**: UIはCSS優先
  - 例: "JavaScript配置の代わりにGridレイアウトを使用"

- **DRY**: パターンの重複を特定
  - 例: "3つの類似した割引計算を発見、共通関数にリファクタリング"

### 原則テンプレート

```markdown
## 適用した原則

- **オッカムの剃刀** ✅
  バリデーションフレームワークの代わりに `Math.max(0, result)` を選択

- **TIDYINGS** ✅
  影響を受けた関数のみを変更、周囲のコードはそのまま

- **段階的強化** ✅
  CSS Grid でレイアウトを修正、JavaScript 不要
```

## 次のステップ

修正完了後:

### 成功パス

1. **文書化**: 修正を説明するコメントを追加
2. **コミット**: 意味のあるコミットメッセージを作成
3. **レビュー**: 変更をセルフレビュー
4. **PR**: プルリクエストを作成（必要な場合）

### フォローアップパス

1. **調査**: 類似パターンが見つかった場合
2. **リファクタリング**: 技術的負債が特定された場合
3. **文書化**: 動作が変更された場合

### 追加アクション不要

バグ修正完了、テスト合格、完了! 🎉
