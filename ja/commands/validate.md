# /validate - SOW基準チェッカー

## 目的

完成した作業に対してSOWの受け入れ基準を手動検証するために表示します。

**簡略化**: 手動チェックリストレビューツール。

## 機能

### 受け入れ基準の表示

```bash
# 最新のSOWから基準を表示
!`ls -t ~/.claude/workspace/planning/*/sow.md | head -1 | xargs grep -A 20 "Acceptance Criteria"`
```

### 手動レビュープロセス

1. SOW基準を表示
2. 各項目を手動でレビュー
3. 実装と照合
4. 結果を文書化

## 出力フォーマット

```markdown
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📋 SOW検証チェックリスト

機能: ユーザー認証
作成日: 2025-01-14

## 受け入れ基準:

□ AC-01: メールによるユーザー登録
  → 確認: 登録フォームが存在するか？
  → 確認: メール検証が実装されているか？

□ AC-02: パスワード要件の実施
  → 確認: 最小8文字？
  → 確認: 特殊文字が必要？

□ AC-03: OAuth統合
  → 確認: Google OAuthが動作するか？
  → 確認: GitHub OAuthが動作するか？

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

手動レビューが必要:
- 各機能をテスト
- 基準に対して検証
- ギャップを文書化
```

## 使用例

### 最新のSOWを検証

```bash
/validate
```

最新のSOWから受け入れ基準を表示します。

### 特定のSOWを検証

```bash
/validate "feature-name"
```

特定の機能の基準を表示します。

## 手動検証プロセス

### ステップ1: 基準をレビュー

```text
- 各受け入れ基準を読む
- 要件を理解する
- 曖昧な点を記録
```

### ステップ2: 実装をテスト

```text
- アプリケーションを実行
- 各機能をテスト
- 動作を文書化
```

### ステップ3: 結果を比較

```text
- 動作を基準と照合
- ギャップを特定
- 必要な改善を記録
```

## ワークフローとの統合

```text
1. 実装を完了
2. /validateを実行して基準を確認
3. 各項目を手動でテスト
4. 結果でドキュメントを更新
```

## 簡略化されたアプローチ

- **自動化なし**: 人間の判断が必要
- **明確なチェックリスト**: フォローしやすい
- **手動プロセス**: 徹底的な検証

## 関連コマンド

- `/think` - 基準付きSOWを作成
- `/sow` - 完全なSOW文書を表示
- `/test` - 自動テストを実行

## 適用された原則

### オッカムの剃刀

- シンプルなチェックリスト表示
- 複雑な検証ロジックなし
- 人間の判断を重視

### 単一責任

- 基準の表示のみ
- 検証は手動で実行
