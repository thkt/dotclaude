# /think - 高度な計画とソリューション設計

## 目的

動的なプロジェクト理解、リスク評価、信頼度スコア付きソリューション計画（SOW）による詳細な問題分析を実行します。

**推奨**: 最初に `/research` を実行してコードベースのコンテキストを理解してから計画を立てることを推奨します。実際のコードベース知識に基づいた計画により、より正確で実装可能なSOWが得られます。

## コンテキストエンジニアリング統合

**重要**: 分析を開始する前に、既存の調査コンテキストを確認してください。

### 自動コンテキスト発見

```bash
# 最新の調査コンテキストファイルを検索
!`find .claude/workspace/research ~/.claude/workspace/research -name "*-context.md" 2>/dev/null | sort -r | head -1`
```

### コンテキスト読み込み戦略

1. **最近の調査を確認**: 過去24時間以内のコンテキストファイルを探す
2. **プロジェクトローカルを優先**: グローバル`~/.claude/workspace/research/`より`.claude/workspace/research/`を優先
3. **重要情報を抽出**: 目的、前提条件、利用可能なデータ、制約条件
4. **計画に統合**: 調査結果を使ってSOW作成に反映

### コンテキスト情報に基づいた計画

調査コンテキストが見つかった場合:

- **🎯 目的**: SOWの目標を調査の目的と整合させる
- **📋 前提条件**: 検証済みの事実に基づき、仮定を検証
- **📊 利用可能なデータ**: 発見されたファイルとスタックを参照
- **🔒 制約条件**: 特定された制限を尊重

**メリット**:

- **高い信頼度**: 実際のコードベース知識に基づいた計画
- **仮定の削減**: 不明点を検証済み事実に置き換え
- **より良い見積もり**: 発見された複雑さに基づいた現実的な見積もり
- **目標の整合**: 調査から実装まで目的主導

## 階層的分析プロセス

### フェーズ1: 問題理解

信頼度スコアリング付き深い分析：

1. **問題定義**: 証拠を伴う明確なステートメント
2. **影響分析**: 影響を受けるシステムとユーザー
3. **根本原因**: なぜ問題が存在するか（症状だけでなく）
4. **制約**: 技術的、時間的、リソースの制限

### フェーズ2: ソリューション発見

複数のアプローチを生成して評価：

1. **ソリューション生成**: 3つ以上の代替アプローチ
2. **原則評価**: SOLID、DRY、プログレッシブエンハンスメント
3. **リスク評価**: 技術的、スケジュール、メンテナンスリスク
4. **信頼度スコアリング**: 各ソリューションを評価（0.0-1.0）

### 技術的リスクマトリックス

```text
| リスク要因 | レベル | 影響 | 軽減策 |
```

### 複雑度スコアリング

- **シンプル** (1-3): 単一ファイル、明確なロジック、50行未満
- **中程度** (4-6): 複数ファイル、いくつかの依存関係、200行未満
- **複雑** (7-9): 多数のファイル、複雑なロジック、200行以上
- **非常に複雑** (10): アーキテクチャ変更、複数システム

### 動的表示制御

タスクの複雑さに基づいて詳細レベルを表示：

#### シンプルなタスク（複雑度1-3）

- サマリーのみ表示
- 完全なSOWを静かに保存
- 信頼度 > 0.9の場合は自動続行

#### 中程度のタスク（複雑度4-6）

- サマリー + 主要リスクを表示
- 確認を求める
- リクエストに応じて完全なSOWを利用可能

#### 複雑なタスク（複雑度7-10）

- 拡張サマリーを表示
- リスクマトリックスを含む
- 明示的な確認が必要
- 完全なSOWのレビューを推奨

### SOW保存先（自動判定）

SOWファイルの保存場所は、git方式のディレクトリ検索により自動的に決定されます：

1. **上位検索**: カレントディレクトリから上位へ`.claude/`ディレクトリを検索
2. **見つかった場合**: `.claude/workspace/planning/[timestamp]-[feature]/sow.md`に保存（プロジェクトローカル）
3. **見つからない場合**: `~/.claude/workspace/planning/[timestamp]-[feature]/sow.md`に保存（グローバル）

**フィードバック**: 保存場所はコンテキスト表示と共に表示されます：

- `✅ SOW保存先: .claude/workspace/planning/... (プロジェクトローカル: .claude/を検出)`
- `✅ SOW保存先: ~/.claude/workspace/planning/... (グローバル: .claude/が見つかりません)`

**メリット**:

- **ゼロコンフィグ**: プロジェクト構造に自動適応
- **チーム共有**: プロジェクトローカルでgitベースの共有が可能
- **個人メモ**: グローバルストレージで探索的作業に対応
- **柔軟性**: `.claude/`を作成してプロジェクトローカルモードに切り替え可能

### 表示詳細度設定

SOW表示のユーザー設定をサポート：

```yaml
# ユーザーは.claude/settings.jsonで設定可能
{
  "sow_display": "summary",     # summary | normal | detailed | auto
  "auto_save_sow": true,        # 常に完全版を保存
  "show_confidence": true,      # 信頼度スコアを表示
  "show_progress": true,        # リアルタイム進捗インジケーター
  "verbosity_by_complexity": {
    "simple": "summary",
    "medium": "normal",
    "complex": "detailed"
  }
}
```

3. **完全な詳細**: 完全なSOW（保存されたファイルまたはインライン）

- ✅ 特定セクションへのクイックアクセスを提供
- ✅ クイックスキャンのためビジュアルインジケーター（絵文字/色）を使用
- ✅ 可読性のためサマリーを15行以下に保つ

#### DON'T

- ❌ サマリーモードで重要なリスクを隠す
- ❌ 低信頼度タスクで自動続行
- ❌ シンプルなタスクで詳細で圧倒
- ❌ 「クイック」タスクのSOW生成をスキップ
- ❌ サマリーと詳細コンテンツを混在

### 自動ドキュメントレビュー

sow.md と spec.md を生成した後、自動的に sow-spec-reviewer を呼び出します:

```typescript
Task({
  subagent_type: "sow-spec-reviewer",
  description: "SOW/Spec自動レビュー",
  prompt: `
    生成されたドキュメントをレビュー:
    - SOW: ${sowPath}
    - Spec: ${specPath}

    100点満点スコアリング（90点合格基準）を適用。
    SOW ↔ Spec の整合性チェックを実施。

    採点基準（各25点）:
    - 正確性: 信頼度マーカー（✓/→/?）の適切な使用
    - 完成度: 必要セクションの網羅性
    - 関連性: 目的との整合性、適切なスコープ
    - 実行可能性: 実装可能な具体性

    結果を日本語で報告。
  `
})
```

## 計画後の次のステップ

- **高信頼度（>0.8）** → `/code`に進む
- **中信頼度（0.6-0.8）** → 必要に応じてさらに`/research`を検討
- **低信頼度（<0.6）** → 進む前に明確化または`/research`が必要

## ベストプラクティス

1. **常に複数のソリューションを検討**: 最初のアイデアで満足しない
2. **すべてをスコア**: 信頼度、リスク、複雑度
3. **仮定を文書化**: 暗黙を明示的に
4. **失敗に備える**: ロールバック戦略を含める
5. **成功を測定**: 明確な基準を定義
6. **履歴から学ぶ**: 過去の類似作業を確認

## SOWストレージ

将来の参照のためすべてのSOWを保存：

```bash
.claude/workspace/planning/
├── YYYY-MM-DD-[タイトル].md      # アクティブなSOW
├── completed/                      # 完了した作業
│   └── YYYY-MM-DD-[タイトル].md
└── archived/                       # キャンセル/古いもの
    └── YYYY-MM-DD-[タイトル].md
```

実装完了時に`completed/`に移動。
古い計画は履歴参照のためアーカイブ。
