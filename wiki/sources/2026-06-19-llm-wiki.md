## Summary
Andrej Karpathy 提唱の「LLM Wiki」を解説した記事。RAG が質問のたびにゼロから探し直し知識が蓄積しないのに対し、LLM Wiki は LLM が個人のナレッジベースを持続的に構築・管理する persistent, compounding artifact。Claude Code 等の手元エージェントと一緒に育てる。

## Key Facts
- RAG の限界: 質問のたびにゼロから情報を探し直すため知識が蓄積しない。
- LLM Wiki の最大の価値は複数資料を横断した「概念ページ」が自動生成される「繋げる力」。10本投入すると共通評価軸や思想の対立軸を AI が抽出する。
- 維持管理(Bookkeeping)は LLM に委譲。1回の資料投入で関連 10〜15 ページを一気に書き換える。人間は良質な資料選定と問いを立てることに集中。
- 三層構造: Raw sources(人間投入、LLM読取専用) / Wiki(LLM管理の Markdown 群) / Schema(人間定義の運用ルール)。
- 3 オペレーション: Ingest(新資料追加) / Query(質問し結果を資産化) / Lint(矛盾・リンク切れ点検)。
- 導入 3 ステップ: Karpathy gist を読ませ雛形生成 → 関心領域の記事5本を Ingest し挙動を目視 → ドメインに合わせ Schema を LLM 経由で育てる。

## Concepts
### LLM Wiki
単発要約を超えた持続的に蓄積する成果物。完成アプリではなく手元 AI と育てる運用パターン。

### 繋げる力 (concept-page synthesis)
別々に投入した複数ソースから AI が共通軸・対立軸を抽出し概念ページを自動生成する。点と点が線になる体験を自動化する。

### LLM-delegated bookkeeping
相互リンク張り直し・矛盾修正・加筆という人間が挫折しがちな維持管理を LLM が飽きずに担う。
